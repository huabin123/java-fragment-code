## 问题发现与应急响应

### 监控告警
通过以下监控系统发现JVM内存异常：
- **Prometheus + Grafana**：指标监控和可视化
- **SkyWalking/Pinpoint**：APM应用性能监控
- **云监控平台**：阿里云ARMS、腾讯云APM

### 告警指标
```yaml
# 关键监控指标
- jvm_memory_used_bytes{area="heap"} > 阈值
- jvm_gc_pause_seconds > 1s
- jvm_memory_pool_bytes_used{pool="old"} / jvm_memory_pool_bytes_max{pool="old"} > 0.9
- Full GC频率 > 5次/分钟
```

### 快速止血措施

**1. 流量控制**
```bash
# 摘除故障节点（从负载均衡器移除）
# Nginx
upstream backend {
    server 10.0.1.10:8080 down;  # 标记为down
    server 10.0.1.11:8080;
}

# K8s
kubectl cordon node-name  # 标记节点不可调度
kubectl drain node-name   # 驱逐Pod
```

**2. 保留现场**
```bash
#!/bin/bash
# 应急脚本：保留现场

PID=$(pgrep -f "your-app")
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
DUMP_DIR="/data/dumps/${TIMESTAMP}"

mkdir -p ${DUMP_DIR}

# 1. 堆dump（最重要）
echo "Dumping heap..."
jmap -dump:live,format=b,file=${DUMP_DIR}/heap.hprof ${PID}

# 2. 线程栈
echo "Dumping threads..."
jstack ${PID} > ${DUMP_DIR}/threads.txt

# 3. GC日志（如果没有启动参数）
jstat -gcutil ${PID} 1000 10 > ${DUMP_DIR}/gc.txt

# 4. JVM参数
jinfo ${PID} > ${DUMP_DIR}/jvm_info.txt

# 5. 系统信息
top -b -n 1 > ${DUMP_DIR}/top.txt
free -h > ${DUMP_DIR}/memory.txt
df -h > ${DUMP_DIR}/disk.txt

echo "Dump completed: ${DUMP_DIR}"
```

**3. 重启服务**
```bash
# 滚动重启（保证服务可用）
# K8s
kubectl rollout restart deployment/your-app

# 传统部署
systemctl restart your-app
```

```bash
# 启动Arthas

# 常用命令

memory             # 查看内存使用

# 监控方法调用
watch com.example.Service method '{params, returnObj, throwExp}' -x 2

# 查看对象实例
vmtool --action getInstances --className java.lang.String --limit 10
```

#### JDK自带工具

**jmap - 内存映射**
```bash
# 生成堆dump
jmap -dump:live,format=b,file=heap.hprof <pid>

# 查看堆内存统计
jmap -heap <pid>

# 查看对象统计（按占用空间排序）
jmap -histo:live <pid> | head -n 20
```

**jstat - GC统计**
```bash
# 每秒输出GC统计，共10次
jstat -gcutil <pid> 1000 10

# 输出示例
S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
0.00  99.99  50.00  75.23  95.45  92.12   156    1.234    5    2.345   3.579

# 查看GC原因
jstat -gccause <pid> 1000
```

**jstack - 线程堆栈**
```bash
# 导出线程栈
jstack <pid> > threads.txt

# 查找死锁
jstack <pid> | grep -A 10 "Found one Java-level deadlock"

# 统计线程状态
jstack <pid> | grep "java.lang.Thread.State" | sort | uniq -c
```

**jinfo - JVM配置**
```bash
# 查看所有JVM参数
jinfo <pid>

# 查看特定参数
jinfo -flag MaxHeapSize <pid>

# 动态修改参数（部分支持）
jinfo -flag +PrintGCDetails <pid>
```

### 2. 离线分析工具

#### Eclipse MAT (Memory Analyzer Tool)
最强大的堆dump分析工具。

**核心功能**：
```
1. Leak Suspects Report - 自动识别内存泄漏嫌疑
2. Dominator Tree - 支配树（找出占用内存最多的对象）
3. Histogram - 对象直方图
4. OQL - 对象查询语言
5. Thread Overview - 线程视图
```

**使用步骤**：
```
1. File → Open Heap Dump → 选择.hprof文件
2. 自动生成Leak Suspects报告
3. 查看Dominator Tree，找出大对象
4. 右键对象 → Path to GC Roots → exclude weak references
5. 分析引用链，定位泄漏点
```

**OQL查询示例**：
```sql
-- 查找所有String对象
SELECT * FROM java.lang.String

-- 查找大于1MB的对象
SELECT * FROM java.lang.Object WHERE @retainedHeapSize > 1048576

-- 查找特定类的实例
SELECT * FROM com.example.User WHERE this.age > 30

-- 统计对象数量
SELECT className, count(*) FROM java.lang.Object GROUP BY className
```

#### JProfiler / YourKit
商业性能分析工具，功能更强大。

**优势**：
- 实时监控（可attach到运行中的JVM）
- CPU性能分析
- 内存分配跟踪
- 数据库查询分析
- 线程分析
- 更友好的UI

## 定位问题步骤

### 第一阶段：现场保留

**自动化脚本**（推荐部署到所有服务器）：
```bash
#!/bin/bash
# /usr/local/bin/jvm-emergency-dump.sh

set -e

if [ $# -lt 1 ]; then
    echo "Usage: $0 <pid> [dump_dir]"
    exit 1
fi

PID=$1
DUMP_DIR=${2:-"/data/jvm-dumps/$(date +%Y%m%d_%H%M%S)"}
APP_NAME=$(ps -p ${PID} -o comm=)

mkdir -p ${DUMP_DIR}

echo "=== JVM Emergency Dump ==="
echo "PID: ${PID}"
echo "App: ${APP_NAME}"
echo "Time: $(date)"
echo "Dir: ${DUMP_DIR}"

# 1. 堆dump（live对象，触发Full GC）
echo "[1/6] Heap dump (this may take a while)..."
jmap -dump:live,format=b,file=${DUMP_DIR}/heap.hprof ${PID} 2>&1 | tee ${DUMP_DIR}/jmap.log

# 2. 堆统计
echo "[2/6] Heap histogram..."
jmap -histo:live ${PID} > ${DUMP_DIR}/histogram.txt 2>&1

# 3. 线程栈（连续3次，间隔5秒）
echo "[3/6] Thread dumps..."
for i in 1 2 3; do
    jstack -l ${PID} > ${DUMP_DIR}/threads_${i}.txt 2>&1
    sleep 5
done

# 4. GC统计
echo "[4/6] GC stats..."
jstat -gcutil ${PID} 1000 30 > ${DUMP_DIR}/gc_stats.txt 2>&1 &
JSTAT_PID=$!

# 5. JVM信息
echo "[5/6] JVM info..."
jinfo ${PID} > ${DUMP_DIR}/jvm_info.txt 2>&1

# 6. 系统信息
echo "[6/6] System info..."
{
    echo "=== CPU ==="
    top -b -n 1 | head -20
    echo -e "\n=== Memory ==="
    free -h
    echo -e "\n=== Disk ==="
    df -h
    echo -e "\n=== Network ==="
    netstat -an | grep ESTABLISHED | wc -l
    echo -e "\n=== Process ==="
    ps aux | grep ${PID}
} > ${DUMP_DIR}/system_info.txt

# 等待jstat完成
wait ${JSTAT_PID}

# 压缩（heap dump通常很大）
echo "Compressing..."
tar -czf ${DUMP_DIR}.tar.gz -C $(dirname ${DUMP_DIR}) $(basename ${DUMP_DIR})

echo "=== Dump Complete ==="
echo "Archive: ${DUMP_DIR}.tar.gz"
echo "Size: $(du -h ${DUMP_DIR}.tar.gz | cut -f1)"
```

### 第二阶段：离线分析

#### MAT分析流程

**1. 打开dump文件**
```
File → Open Heap Dump → heap.hprof
选择 "Leak Suspects Report"
```

**2. 查看Leak Suspects**
```
自动生成的报告会列出：
- Problem Suspect 1: 占用XX MB (XX%)
- 可疑对象类型
- 引用链路
```

**3. Dominator Tree分析**
```
点击 "Dominator Tree"
按 Retained Heap 排序
展开占用最大的对象
查看：
- Shallow Heap: 对象自身大小
- Retained Heap: 对象及其引用的总大小
```

**4. 查看引用链（GC Root路径）**
```
右键对象 → Path to GC Roots → exclude weak/soft references
分析为什么对象无法被回收：
- 被静态变量引用
- 被ThreadLocal持有
- 被缓存持有
- 被监听器持有
```

**5. Histogram分析**
```
查看对象实例数量和占用空间
找出异常增长的类：
- char[] / byte[] 数组过多 → 可能是字符串/缓存问题
- HashMap$Node 过多 → 可能是Map泄漏
- Thread 过多 → 可能是线程泄漏
```

#### 常见内存泄漏模式

**模式1：静态集合累积**
```java
// 问题代码
public class UserCache {
    private static Map<String, User> cache = new HashMap<>();
    
    public void addUser(User user) {
        cache.put(user.getId(), user);  // 永远不清理
    }
}

// MAT中表现：
// - HashMap$Node 数量持续增长
// - GC Root: static field UserCache.cache
```

**模式2：ThreadLocal未清理**
```java
// 问题代码
public class RequestContext {
    private static ThreadLocal<User> currentUser = new ThreadLocal<>();
    
    public void setUser(User user) {
        currentUser.set(user);  // 线程池复用时未清理
    }
}

// MAT中表现：
// - Thread → ThreadLocalMap → Entry[] → User
// - 线程池中的线程持有大量对象
```

**模式3：监听器未注销**
```java
// 问题代码
public class EventManager {
    private List<EventListener> listeners = new ArrayList<>();
    
    public void register(EventListener listener) {
        listeners.add(listener);  // 从不移除
    }
}

// MAT中表现：
// - EventManager.listeners 持有大量对象
// - 即使业务对象不再使用，仍被监听器引用
```

**模式4：资源未关闭**
```java
// 问题代码
public void processFile(String path) {
    FileInputStream fis = new FileInputStream(path);
    // 处理文件...
    // 忘记关闭，导致文件句柄泄漏
}

// MAT中表现：
// - FileInputStream / FileDescriptor 数量异常
// - 系统层面：lsof 显示大量打开文件
```

### 第三阶段：代码定位

#### 根据MAT结果定位代码

**示例：发现HashMap泄漏**
```
MAT分析结果：
- com.example.cache.LocalCache$CacheMap 占用 2GB
- 包含 500万个 Entry
- GC Root: static field LocalCache.INSTANCE

定位步骤：
1. 搜索代码中的 LocalCache 类
2. 检查缓存清理策略
3. 查看缓存大小限制
4. 分析业务逻辑是否合理
```

**代码审查重点**：
```java
// 检查点1：缓存是否有上限
Map<String, Object> cache = new HashMap<>();  // ❌ 无限增长
Map<String, Object> cache = new LRUCache<>(1000);  // ✅ 有上限

// 检查点2：是否有过期机制
cache.put(key, value);  // ❌ 永不过期
cache.put(key, value, 10, TimeUnit.MINUTES);  // ✅ 定时过期

// 检查点3：资源是否正确关闭
InputStream is = new FileInputStream(file);  // ❌ 可能泄漏
try (InputStream is = new FileInputStream(file)) {  // ✅ 自动关闭
    // ...
}

// 检查点4：ThreadLocal是否清理
threadLocal.set(value);  // ❌ 未清理
try {
    threadLocal.set(value);
    // ...
} finally {
    threadLocal.remove();  // ✅ 及时清理
}
```

#### 使用Arthas动态验证

```bash
# 1. 查看某个类的实例数量
vmtool --action getInstances \
       --className com.example.cache.LocalCache \
       --limit 10

# 2. 查看对象的字段值
ognl '@com.example.cache.LocalCache@INSTANCE.cache.size()'

# 3. 监控方法调用
watch com.example.cache.LocalCache put '{params, returnObj}' -x 2

# 4. 追踪方法调用路径
trace com.example.cache.LocalCache put -n 5

# 5. 查看对象内存占用
memory -c com.example.cache.LocalCache
```

#### 3. 方案评审（2-3小时）

**技术方案模板**
```markdown
## 根因分析
**问题定位**:
- LocalCache.INSTANCE.cache 占用2.3GB
- 包含500万个缓存条目
- 缓存key为用户ID，永不过期

**根本原因**:
- 3月20日上线的功能引入了用户级缓存
- 代码中未设置缓存上限和过期时间
- 随着用户增长，缓存无限累积

## 修复方案
**方案1（临时）**: 重启服务 + 降级缓存
- 优点：快速恢复
- 缺点：治标不治本

**方案2（根本）**: 引入Guava Cache，设置上限和过期
```java
// 修改前
private static Map<String, User> cache = new HashMap<>();

// 修改后
private static LoadingCache<String, User> cache = CacheBuilder.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(new CacheLoader<String, User>() {
        public User load(String key) {
            return loadUserFromDB(key);
        }
    });
```
- 优点：根本解决问题
- 缺点：需要测试验证

**推荐方案**: 方案2
**风险评估**: 低（仅修改缓存实现，不改变业务逻辑）
**测试计划**: 单元测试 + 压测验证
```

### 案例1：ThreadLocal泄漏

**背景**：
- 服务：用户中心
- 现象：内存缓慢增长，7天后OOM
- 线程池：固定200个线程

**问题代码**：
```java
public class UserContext {
    private static ThreadLocal<UserSession> session = new ThreadLocal<>();
    
    public static void setSession(UserSession s) {
        session.set(s);  // 设置用户会话
    }
    
    public static UserSession getSession() {
        return session.get();
    }
    
    // ❌ 缺少清理方法
}

// Controller中使用
@PostMapping("/login")
public Result login(LoginRequest req) {
    UserSession session = authService.login(req);
    UserContext.setSession(session);  // 设置到ThreadLocal
    return Result.success();
    // ❌ 请求结束后未清理
}
```

**MAT分析**：
```
Thread "http-nio-8080-exec-1" 
  → ThreadLocalMap 
    → Entry[] 
      → UserSession (50MB)
        → User
        → Permissions (大量权限数据)

200个线程 × 50MB = 10GB
```

**修复方案**：
```java
// 方案1：使用Filter统一清理
@Component
public class ThreadLocalCleanFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
        try {
            chain.doFilter(req, res);
        } finally {
            UserContext.clear();  // 请求结束清理
        }
    }
}

public class UserContext {
    private static ThreadLocal<UserSession> session = new ThreadLocal<>();
    
    public static void clear() {
        session.remove();  // ✅ 清理ThreadLocal
    }
}

// 方案2：使用TransmittableThreadLocal（阿里）
private static TransmittableThreadLocal<UserSession> session = 
    new TransmittableThreadLocal<>();
```

### 案例2：Mybatis ResultMap泄漏

**背景**：
- 服务：报表服务
- 现象：查询大量数据后OOM
- 数据量：单次查询100万条记录

**问题代码**：
```java
// Mapper
@Select("SELECT * FROM orders WHERE create_time > #{startTime}")
List<Order> queryOrders(@Param("startTime") Date startTime);

// Service
public void exportOrders() {
    List<Order> orders = orderMapper.queryOrders(yesterday);
    // ❌ 一次性加载100万条数据到内存
    for (Order order : orders) {
        // 处理...
    }
}
```

**MAT分析**：
```
ArrayList (2GB)
  → Order[1000000]
    → 每个Order包含：
      - 基本字段
      - 关联的User对象
      - 关联的Product列表
      
平均每个Order 2KB × 100万 = 2GB
```

**修复方案**：
```java
// 方案1：分页查询
public void exportOrders() {
    int pageSize = 1000;
    int pageNum = 0;
    List<Order> orders;
    
    do {
        orders = orderMapper.queryOrders(yesterday, pageNum * pageSize, pageSize);
        for (Order order : orders) {
            // 处理...
        }
        pageNum++;
    } while (orders.size() == pageSize);
}

// 方案2：流式查询（Mybatis 3.4+）
@Options(resultSetType = ResultSetType.FORWARD_ONLY, fetchSize = 1000)
@Select("SELECT * FROM orders WHERE create_time > #{startTime}")
void queryOrders(@Param("startTime") Date startTime, ResultHandler<Order> handler);

// 使用
orderMapper.queryOrders(yesterday, context -> {
    Order order = context.getResultObject();
    // 处理单条记录
});

// 方案3：使用游标（Mybatis 3.4+）
@Select("SELECT * FROM orders WHERE create_time > #{startTime}")
Cursor<Order> queryOrdersCursor(@Param("startTime") Date startTime);

// 使用
try (Cursor<Order> cursor = orderMapper.queryOrdersCursor(yesterday)) {
    for (Order order : cursor) {
        // 逐条处理，不会一次性加载到内存
    }
}
```

### 案例3：Netty堆外内存泄漏

**背景**：
- 服务：网关服务（基于Netty）
- 现象：堆内存正常，但进程内存持续增长
- 监控：JVM堆1GB，进程RSS 8GB

**问题代码**：
```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    ByteBuf buf = (ByteBuf) msg;
    // 处理消息...
    // ❌ 忘记释放ByteBuf
}
```

**诊断方法**：
```bash
# 1. 查看进程内存
ps aux | grep java
# RSS: 8GB（远超堆内存）

# 2. 查看堆外内存（JDK 8+）
jcmd <pid> VM.native_memory summary

# 输出示例
Native Memory Tracking:
Total: reserved=8192MB, committed=6144MB
-       Java Heap (reserved=1024MB, committed=1024MB)
-          Thread (reserved=200MB, committed=200MB)
-            Code (reserved=250MB, committed=150MB)
-             GC  (reserved=100MB, committed=100MB)
-        Internal (reserved=6418MB, committed=4770MB)  # ← 异常！

# 3. 使用Netty内存泄漏检测
-Dio.netty.leakDetection.level=PARANOID
```

**修复方案**：
```java
// 方案1：手动释放
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    ByteBuf buf = (ByteBuf) msg;
    try {
        // 处理消息...
    } finally {
        buf.release();  // ✅ 释放ByteBuf
    }
}

// 方案2：使用SimpleChannelInboundHandler（自动释放）
public class MyHandler extends SimpleChannelInboundHandler<ByteBuf> {
    @Override
    protected void channelRead0(ChannelHandlerContext ctx, ByteBuf msg) {
        // 处理消息...
        // ✅ 方法结束后自动释放
    }
}

// 方案3：使用ReferenceCountUtil
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    try {
        // 处理消息...
    } finally {
        ReferenceCountUtil.release(msg);  // ✅ 安全释放
    }
}
```

### 案例4：Spring循环依赖导致的内存问题

**背景**：
- 服务：商品服务
- 现象：启动时内存飙升，创建大量重复对象

**问题代码**：
```java
@Service
public class ProductService {
    @Autowired
    private CategoryService categoryService;  // 依赖CategoryService
    
    public Product getProduct(Long id) {
        Product product = productDao.get(id);
        product.setCategory(categoryService.getCategory(product.getCategoryId()));
        return product;
    }
}

@Service
public class CategoryService {
    @Autowired
    private ProductService productService;  // 依赖ProductService（循环依赖）
    
    public Category getCategory(Long id) {
        Category category = categoryDao.get(id);
        category.setProducts(productService.getProductsByCategory(id));
        return category;
    }
}
```

**MAT分析**：
```
ProductService$$EnhancerBySpringCGLIB (大量代理对象)
  → CategoryService$$EnhancerBySpringCGLIB
    → ProductService$$EnhancerBySpringCGLIB
      → ...（循环引用）
```

**修复方案**：
```java
// 方案1：重构依赖关系（推荐）
@Service
public class ProductService {
    @Autowired
    private CategoryService categoryService;
    
    public Product getProduct(Long id) {
        Product product = productDao.get(id);
        product.setCategory(categoryService.getCategory(product.getCategoryId()));
        return product;
    }
}

@Service
public class CategoryService {
    // ✅ 移除对ProductService的依赖
    @Autowired
    private ProductDao productDao;
    
    public Category getCategory(Long id) {
        Category category = categoryDao.get(id);
        category.setProducts(productDao.findByCategoryId(id));
        return category;
    }
}

// 方案2：使用@Lazy延迟加载
@Service
public class CategoryService {
    @Autowired
    @Lazy  // ✅ 延迟注入
    private ProductService productService;
}

// 方案3：使用Setter注入
@Service
public class CategoryService {
    private ProductService productService;
    
    @Autowired
    public void setProductService(ProductService productService) {
        this.productService = productService;
    }
}
