# 分代垃圾回收

## 💡 大白话精华总结

**分代回收是什么？**
- 把堆内存分成"年轻人区"和"老年人区"，分别用不同方法清理
- 就像垃圾分类：可回收垃圾和不可回收垃圾分开处理

**为什么要分代？基于3个假说：**
1. **弱分代假说**：98%的对象朝生夕死（大部分对象很快就没用了）
2. **强分代假说**：熬过多次GC的对象很难死（老对象很顽强）
3. **跨代引用假说**：老对象很少引用新对象

**堆内存的划分：**
```
堆内存
├── 新生代（Young Generation）- 1/3
│   ├── Eden区（伊甸园）- 80%
│   ├── Survivor0（幸存者0）- 10%
│   └── Survivor1（幸存者1）- 10%
└── 老年代（Old Generation）- 2/3
```

**新生代GC（Minor GC）流程：**
```
1. 新对象在Eden区出生
2. Eden满了 → Minor GC
3. 活着的对象 → 复制到Survivor区
4. Survivor区的对象年龄+1
5. 年龄达到15 → 晋升到老年代
6. 清空Eden和用过的Survivor
```

**老年代GC（Major GC/Full GC）：**
```
老年代满了 → Full GC
- 速度慢（对象多）
- 暂停时间长（STW）
- 尽量避免！
```

**对象晋升老年代的条件：**
1. **年龄够了**：经历15次Minor GC（默认）
2. **太大了**：大对象直接进老年代
3. **Survivor放不下**：Survivor区满了，提前晋升
4. **动态年龄判定**：Survivor中相同年龄对象超过50%，该年龄及以上的对象晋升

**GC类型对比：**
| GC类型 | 触发条件 | 速度 | 暂停时间 | 频率 |
|--------|---------|------|---------|------|
| Minor GC | Eden满 | 快 | 短（几ms） | 高 |
| Major GC | 老年代满 | 慢 | 长（几百ms） | 低 |
| Full GC | 整个堆满 | 很慢 | 很长（几秒） | 很低 |

**一句话记住：**
> 分代回收就是"年轻人死得快用复制，老年人活得久用整理"！

---

## 📚 概述

分代垃圾回收是基于"大部分对象朝生夕死"的假设，将堆内存划分为不同的代，针对不同代采用不同的回收策略，从而提高GC效率。

## 一、分代假说

### 1.1 弱分代假说

**大部分对象朝生夕死**

```java
public void method() {
    // 局部变量，方法结束后立即死亡
    Object temp = new Object();
    String str = "hello";
    int[] arr = new int[10];
}

// 统计数据：
// - 98%的对象在创建后很快死亡
// - 只有2%的对象能存活较长时间
```

### 1.2 强分代假说

**熬过多次GC的对象难以消亡**

```java
public class Service {
    // 静态变量，生命周期很长
    private static Cache cache = new Cache();
    
    // 单例对象，生命周期很长
    private static Service instance = new Service();
}

// 统计数据：
// - 经历15次GC仍存活的对象
// - 99%会继续存活很长时间
```

### 1.3 跨代引用假说

**跨代引用相对于同代引用来说占极少数**

```
老年代对象 → 新生代对象（少）
新生代对象 → 新生代对象（多）
老年代对象 → 老年代对象（多）

原因：
- 新生代对象很快死亡
- 如果老年代对象引用新生代对象
- 新生代对象很快死亡，引用断开
- 老年代对象很少长期持有新生代对象的引用
```

## 二、分代模型

### 2.1 堆内存划分

```
Java堆内存结构：

+------------------------------------------+
|                  Java Heap               |
+------------------------------------------+
|  Young Generation  |   Old Generation   |
+--------------------+--------------------+
| Eden | S0  |  S1   |    Old/Tenured    |
+------+-----+-------+--------------------+
  8:1    1:1    1:1          更大
```

### 2.2 各区域说明

| 区域 | 大小比例 | 用途 | GC频率 |
|------|---------|------|--------|
| **Eden区** | 80% | 新对象分配 | 高 |
| **Survivor0** | 10% | 存活对象 | 高 |
| **Survivor1** | 10% | 存活对象 | 高 |
| **老年代** | 2倍新生代 | 长期存活对象 | 低 |

### 2.3 为什么这样划分

```
1. Eden区大（80%）：
   - 大部分对象在Eden区分配
   - 大部分对象很快死亡
   - 减少GC频率

2. Survivor区小（各10%）：
   - 只有少量对象存活
   - 10%的空间足够
   - 节省内存

3. 老年代大：
   - 存放长期存活对象
   - GC频率低
   - 需要更大空间
```

## 三、新生代GC（Minor GC）

### 3.1 工作流程

```
初始状态：
Eden: [对象A][对象B][对象C][对象D][对象E]
S0:   [空]
S1:   [空]

第1次Minor GC：
1. Eden区满
2. 标记存活对象（A、C、E）
3. 复制到S0
4. 清空Eden

结果：
Eden: [空]
S0:   [对象A(age=1)][对象C(age=1)][对象E(age=1)]
S1:   [空]

继续分配：
Eden: [对象F][对象G][对象H]
S0:   [对象A(age=1)][对象C(age=1)][对象E(age=1)]
S1:   [空]

第2次Minor GC：
1. Eden区满
2. 标记Eden和S0的存活对象（A、C、F、H）
3. 复制到S1，age+1
4. 清空Eden和S0

结果：
Eden: [空]
S0:   [空]
S1:   [对象A(age=2)][对象C(age=2)][对象F(age=1)][对象H(age=1)]
```

### 3.2 详细步骤

#### 步骤1：对象分配

```java
public class AllocationDemo {
    public static void main(String[] args) {
        // 对象在Eden区分配
        byte[] obj1 = new byte[1024 * 1024];  // 1MB
        byte[] obj2 = new byte[1024 * 1024];  // 1MB
        byte[] obj3 = new byte[1024 * 1024];  // 1MB
        
        // Eden区满，触发Minor GC
        byte[] obj4 = new byte[1024 * 1024];  // 1MB
    }
}
```

#### 步骤2：触发Minor GC

```
触发条件：
1. Eden区空间不足
2. 显式调用System.gc()（不推荐）

GC过程：
1. Stop The World
2. 标记Eden和From Survivor的存活对象
3. 复制到To Survivor
4. 清空Eden和From Survivor
5. 交换From和To
6. 恢复应用线程
```

#### 步骤3：对象年龄增长

```java
// 对象头中的年龄字段（4位，最大15）
class ObjectHeader {
    int age : 4;  // 年龄，0-15
    // ...
}

// 每次Minor GC，存活对象age+1
age = 0  // 新创建
age = 1  // 经历1次Minor GC
age = 2  // 经历2次Minor GC
...
age = 15 // 经历15次Minor GC（最大值）
```

### 3.3 Minor GC示例

```java
/**
 * VM参数：
 * -Xms20M -Xmx20M -Xmn10M -XX:+PrintGCDetails -XX:SurvivorRatio=8
 * 
 * 堆大小：20M
 * 新生代：10M（Eden=8M, S0=1M, S1=1M）
 * 老年代：10M
 */
public class MinorGCDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        byte[] allocation1, allocation2, allocation3, allocation4;
        
        // 分配3个1MB对象，在Eden区
        allocation1 = new byte[_1MB];
        allocation2 = new byte[_1MB];
        allocation3 = new byte[_1MB];
        
        System.out.println("分配3个1MB对象后");
        printMemory();
        
        // 分配4MB对象，Eden区不足，触发Minor GC
        allocation4 = new byte[4 * _1MB];
        
        System.out.println("分配4MB对象后（触发Minor GC）");
        printMemory();
    }
    
    private static void printMemory() {
        Runtime runtime = Runtime.getRuntime();
        long total = runtime.totalMemory() / _1MB;
        long free = runtime.freeMemory() / _1MB;
        long used = total - free;
        
        System.out.println("总内存: " + total + "MB");
        System.out.println("空闲内存: " + free + "MB");
        System.out.println("已用内存: " + used + "MB");
        System.out.println("----------------------------");
    }
}

// 输出示例：
// [GC (Allocation Failure) [PSYoungGen: 7291K->1016K(9216K)] 7291K->5128K(19456K), 0.0034567 secs]
// 
// 解读：
// PSYoungGen: 新生代GC
// 7291K->1016K(9216K): 新生代从7291K降到1016K，总容量9216K
// 7291K->5128K(19456K): 整个堆从7291K降到5128K，总容量19456K
// 0.0034567 secs: GC耗时
```

### 3.4 Minor GC特点

| 特点 | 说明 |
|------|------|
| **频率** | 高（Eden区小，很快填满） |
| **速度** | 快（存活对象少，复制快） |
| **暂停时间** | 短（几毫秒到几十毫秒） |
| **算法** | 标记-复制 |
| **影响** | 小（暂停时间短） |

## 四、老年代GC（Major GC / Full GC）

### 4.1 触发条件

```
1. 老年代空间不足
   - 大对象直接进入老年代
   - 长期存活对象晋升到老年代

2. 空间分配担保失败
   - Survivor区放不下存活对象
   - 需要老年代担保

3. Metaspace空间不足
   - 类元数据过多

4. 显式调用System.gc()
   - 不推荐使用

5. CMS GC的并发失败
   - 并发模式失败
   - 晋升失败
```

### 4.2 工作流程

```
老年代GC流程：

1. 标记阶段（STW）
   ├─ 从GC Roots开始标记
   ├─ 标记所有可达对象
   └─ 耗时较长

2. 清除/整理阶段
   ├─ 标记-清除：直接清除未标记对象
   └─ 标记-整理：移动存活对象，整理内存

3. 恢复应用
```

### 4.3 Full GC示例

```java
/**
 * VM参数：
 * -Xms20M -Xmx20M -Xmn10M -XX:+PrintGCDetails
 */
public class FullGCDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        byte[] allocation1, allocation2, allocation3, allocation4;
        
        // 分配对象，触发Minor GC
        allocation1 = new byte[2 * _1MB];
        allocation2 = new byte[2 * _1MB];
        allocation3 = new byte[2 * _1MB];
        
        // 创建大对象，直接进入老年代
        allocation4 = new byte[8 * _1MB];
        
        System.out.println("创建大对象后");
        printGCInfo();
        
        // 老年代空间不足，触发Full GC
        byte[] allocation5 = new byte[8 * _1MB];
        
        System.out.println("再次创建大对象后（触发Full GC）");
        printGCInfo();
    }
    
    private static void printGCInfo() {
        System.out.println("GC信息已打印");
        System.out.println("----------------------------");
    }
}

// 输出示例：
// [Full GC (Ergonomics) [PSYoungGen: 2048K->0K(9216K)] [ParOldGen: 8192K->10240K(10240K)] 10240K->10240K(19456K), 0.0123456 secs]
//
// 解读：
// Full GC: 全堆GC
// PSYoungGen: 新生代从2048K降到0K
// ParOldGen: 老年代从8192K升到10240K（对象晋升）
// 整个堆从10240K到10240K
```

### 4.4 Full GC特点

| 特点 | 说明 |
|------|------|
| **频率** | 低（老年代大，不容易填满） |
| **速度** | 慢（对象多，扫描慢） |
| **暂停时间** | 长（几百毫秒到几秒） |
| **算法** | 标记-清除或标记-整理 |
| **影响** | 大（暂停时间长，影响用户体验） |

## 五、对象晋升

### 5.1 晋升条件

```
对象晋升到老年代的条件：

1. 年龄达到阈值（默认15）
   - 经历15次Minor GC仍存活
   - 可通过-XX:MaxTenuringThreshold设置

2. 大对象直接晋升
   - 大小超过-XX:PretenureSizeThreshold
   - 直接在老年代分配

3. Survivor区空间不足
   - 存活对象超过Survivor容量
   - 提前晋升到老年代

4. 动态年龄判定
   - Survivor区中相同年龄对象大小总和 > Survivor空间一半
   - 年龄 >= 该年龄的对象直接晋升
```

### 5.2 年龄阈值晋升

```java
/**
 * VM参数：
 * -XX:MaxTenuringThreshold=15  // 设置最大年龄
 * -XX:+PrintTenuringDistribution  // 打印年龄分布
 */
public class AgeTenuringDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        // 创建对象
        byte[] allocation1 = new byte[_1MB / 4];
        
        // 触发多次Minor GC
        for (int i = 0; i < 20; i++) {
            byte[] temp = new byte[4 * _1MB];
            temp = null;
            System.gc();
        }
        
        // allocation1经历多次GC后晋升到老年代
    }
}

// 输出示例：
// Desired survivor size 524288 bytes, new threshold 15 (max 15)
// - age   1:     262144 bytes,     262144 total
// - age   2:     262144 bytes,     524288 total
// ...
// - age  15:     262144 bytes,    3932160 total
```

### 5.3 大对象直接晋升

```java
/**
 * VM参数：
 * -XX:PretenureSizeThreshold=3145728  // 3MB
 * -XX:+UseSerialGC  // 只有Serial和ParNew支持此参数
 */
public class PretenureSizeDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        // 小对象，在Eden区分配
        byte[] allocation1 = new byte[2 * _1MB];
        
        // 大对象（>3MB），直接在老年代分配
        byte[] allocation2 = new byte[4 * _1MB];
        
        System.out.println("大对象直接进入老年代");
    }
}

// 为什么大对象直接进入老年代？
// 1. 避免在Eden和Survivor之间复制
// 2. 减少Minor GC的开销
// 3. 大对象通常生命周期较长
```

### 5.4 动态年龄判定

```java
/**
 * 动态年龄判定规则：
 * 如果Survivor区中相同年龄所有对象大小的总和 > Survivor空间的一半
 * 年龄 >= 该年龄的对象可以直接进入老年代
 */
public class DynamicAgeTenuringDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        // Survivor区大小：1MB
        
        // 创建多个相同年龄的对象
        byte[] obj1 = new byte[_1MB / 4];  // age=1
        byte[] obj2 = new byte[_1MB / 4];  // age=1
        byte[] obj3 = new byte[_1MB / 4];  // age=1
        
        // 总大小 = 3/4 MB > 1/2 MB
        // 触发Minor GC后，这些对象会直接晋升到老年代
        // 即使age < MaxTenuringThreshold
        
        System.gc();
    }
}
```

### 5.5 晋升流程图

```
对象创建
   ↓
Eden区分配
   ↓
Eden区满？ ─No→ 继续分配
   ↓Yes
Minor GC
   ↓
对象存活？ ─No→ 回收
   ↓Yes
   ├─ 大对象？ ─Yes→ 老年代
   ├─ age >= 阈值？ ─Yes→ 老年代
   ├─ Survivor满？ ─Yes→ 老年代
   ├─ 动态判定？ ─Yes→ 老年代
   └─ No → Survivor区（age+1）
```

## 六、空间分配担保

### 6.1 什么是空间分配担保

在Minor GC前，JVM会检查老年代最大可用连续空间是否大于新生代所有对象总空间。

```
Minor GC前的检查：

老年代最大连续空间 >= 新生代所有对象？
   ↓Yes
安全，执行Minor GC
   ↓No
检查HandlePromotionFailure设置
   ↓
老年代最大连续空间 >= 历次晋升平均大小？
   ↓Yes
冒险执行Minor GC
   ↓No
执行Full GC
```

### 6.2 为什么需要担保

```
问题场景：
1. Minor GC时，Eden和Survivor的存活对象需要复制到另一个Survivor
2. 如果存活对象太多，Survivor放不下
3. 需要老年代提供担保，将对象直接晋升到老年代

担保机制：
- 确保Minor GC不会失败
- 避免频繁Full GC
```

### 6.3 担保失败

```java
/**
 * 空间分配担保失败示例
 */
public class HandlePromotionFailureDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        // 老年代剩余空间：5MB
        
        // 创建大量对象
        List<byte[]> list = new ArrayList<>();
        for (int i = 0; i < 10; i++) {
            list.add(new byte[_1MB]);
        }
        
        // Minor GC时，存活对象需要晋升
        // 但老年代空间不足
        // 触发Full GC
        
        System.gc();
    }
}

// 输出示例：
// [GC (Allocation Failure) [PSYoungGen: 8192K->1024K(9216K)] 8192K->5120K(19456K), 0.0012345 secs]
// [Full GC (Ergonomics) [PSYoungGen: 1024K->0K(9216K)] [ParOldGen: 4096K->5120K(10240K)] 5120K->5120K(19456K), 0.0123456 secs]
```

### 6.4 JDK版本差异

```
JDK 6 Update 24之前：
- 需要设置-XX:+HandlePromotionFailure
- 控制是否允许担保失败

JDK 6 Update 24之后：
- 不再需要此参数
- 只要老年代连续空间 > 新生代对象总大小 或 历次晋升平均大小
- 就会进行Minor GC
- 否则进行Full GC
```

## 七、分代GC的优化

### 7.1 对象分配优化

#### 1. 栈上分配

```java
// 逃逸分析：对象不会逃逸出方法
public void method() {
    Point p = new Point(1, 2);
    int x = p.getX();
    int y = p.getY();
    // p不会逃逸出方法
}

// JVM优化：在栈上分配对象
// 好处：
// - 方法结束，栈帧销毁，对象自动回收
// - 不需要GC
// - 性能更好
```

#### 2. 标量替换

```java
// 原始代码
public void method() {
    Point p = new Point(1, 2);
    int sum = p.getX() + p.getY();
}

// 标量替换后
public void method() {
    int x = 1;
    int y = 2;
    int sum = x + y;
    // 不创建Point对象
}
```

#### 3. TLAB（Thread Local Allocation Buffer）

```
问题：
- 多线程并发分配对象
- 需要同步，性能差

解决：
- 每个线程在Eden区预分配一块缓冲区（TLAB）
- 线程在自己的TLAB中分配对象
- 无需同步，性能好

TLAB结构：
Eden区：
[Thread1 TLAB][Thread2 TLAB][Thread3 TLAB][共享区域]
```

```java
/**
 * TLAB参数：
 * -XX:+UseTLAB  // 启用TLAB（默认启用）
 * -XX:TLABSize  // TLAB大小
 * -XX:+PrintTLAB  // 打印TLAB信息
 */
public class TLABDemo {
    public static void main(String[] args) {
        // 对象在线程的TLAB中分配
        for (int i = 0; i < 1000; i++) {
            Object obj = new Object();
        }
    }
}
```

### 7.2 GC频率优化

```
减少Minor GC频率：
1. 增大Eden区
   -XX:NewRatio=2  // 新生代:老年代 = 1:2
   -XX:SurvivorRatio=8  // Eden:Survivor = 8:1

2. 减少对象创建
   - 对象复用
   - 使用对象池

3. 使用栈上分配
   -XX:+DoEscapeAnalysis  // 启用逃逸分析

减少Full GC频率：
1. 增大老年代
   -Xmx4g  // 增大堆大小

2. 避免大对象
   - 拆分大对象
   - 使用流式处理

3. 及时释放引用
   - 不再使用的对象置为null
```

### 7.3 晋升优化

```
延迟对象晋升：
1. 增大Survivor区
   -XX:SurvivorRatio=6  // Eden:Survivor = 6:1

2. 提高晋升年龄阈值
   -XX:MaxTenuringThreshold=15

3. 避免大对象直接晋升
   -XX:PretenureSizeThreshold=10485760  // 10MB

加速对象晋升：
1. 降低晋升年龄阈值
   -XX:MaxTenuringThreshold=3

2. 减小Survivor区
   -XX:SurvivorRatio=10
```

## 八、分代GC的问题

### 8.1 跨代引用

```
问题：
老年代对象 → 新生代对象

Minor GC时：
- 需要扫描老年代找到引用
- 老年代很大，扫描慢

解决：
- 使用记忆集（Remembered Set）
- 记录老年代→新生代的引用
- Minor GC时只扫描记忆集
```

### 8.2 卡表实现

```
卡表（Card Table）：
- 将堆划分为512字节的卡片
- 用字节数组记录每个卡片的状态

卡表数组：
[0][1][0][0][1][0]...
 ↑  ↑        ↑
 |  |        └─ 卡片4有跨代引用
 |  └────────── 卡片1有跨代引用
 └──────────── 卡片0无跨代引用

写屏障（Write Barrier）：
- 对象引用更新时
- 检查是否跨代引用
- 更新卡表
```

```java
// 写屏障伪代码
void oop_store(oop* addr, oop value) {
    *addr = value;  // 更新引用
    
    // 检查是否跨代引用
    if (is_in_young(value) && is_in_old(addr)) {
        // 标记卡表
        mark_card_dirty(addr);
    }
}
```

### 8.3 并发问题

```
问题：
- Minor GC时需要STW
- 影响应用性能

解决：
- 使用并发GC（如G1、ZGC）
- 减少STW时间
```

## 九、G1的分代模型

### 9.1 传统分代 vs G1分代

```
传统分代（连续内存）：
+------------------------------------------+
|  Young Generation  |   Old Generation   |
+--------------------+--------------------+
| Eden | S0  |  S1   |    Old/Tenured    |
+------+-----+-------+--------------------+

特点：
- 内存连续
- 固定大小
- 严格分代

G1分代（Region分区）：
+---+---+---+---+---+---+---+---+
| E | E | S | O | O | E | S | O |
+---+---+---+---+---+---+---+---+
| E | O | O | E | S | O | E | O |
+---+---+---+---+---+---+---+---+

E: Eden Region
S: Survivor Region
O: Old Region
H: Humongous Region（大对象）

特点：
- 内存不连续
- 动态大小
- 逻辑分代
```

### 9.2 G1的分代特点

#### 1. 逻辑分代

```
传统分代：
- 物理上连续
- Eden、Survivor、Old是固定的内存区域

G1分代：
- 逻辑上分代
- 每个Region可以动态变化角色
- 今天是Eden，明天可能是Old

示例：
时刻1：Region 1 = Eden
时刻2：Minor GC后，Region 1被清空
时刻3：Region 1 = Old（存放晋升对象）
```

#### 2. 动态调整

```
G1会根据以下因素动态调整：
1. 停顿时间目标（MaxGCPauseMillis）
2. 堆使用情况
3. GC频率

调整内容：
- Eden Region数量
- Survivor Region数量
- 每次GC回收的Region数量
```

#### 3. 分代GC过程

```
Young GC（年轻代GC）：
1. 选择所有Eden Region
2. 选择所有Survivor Region
3. 将存活对象复制到新的Survivor或Old Region
4. 清空原Region

特点：
- 与传统分代类似
- 但是Region是动态的
- 停顿时间可控

Mixed GC（混合GC）：
1. 选择所有Eden Region
2. 选择所有Survivor Region
3. 选择部分Old Region（垃圾最多的）
4. 一起回收

特点：
- G1特有
- 同时回收年轻代和老年代
- 分多次进行
```

### 9.3 G1的对象晋升

```
G1的晋升条件：
1. 年龄达到阈值（默认15）
2. Survivor Region放不下
3. 动态年龄判定

与传统分代的区别：
传统分代：
- 对象从Survivor晋升到Old区
- Old区是固定的内存区域

G1：
- 对象从Survivor Region晋升到Old Region
- Old Region是动态分配的
- 可能分配到堆的任何位置
```

### 9.4 G1的优势

```
1. 内存利用率高
   - 不需要连续内存
   - 减少碎片

2. 停顿时间可控
   - 可以设置停顿时间目标
   - G1会尽力达到目标

3. 适合大堆
   - 传统分代：堆越大，Full GC越慢
   - G1：堆大小对停顿时间影响小

4. 并发执行
   - 大部分工作与应用并发
   - 减少STW时间
```

### 9.5 G1 vs 传统分代对比

| 特性 | 传统分代 | G1分代 |
|------|---------|--------|
| **内存布局** | 连续 | 分散（Region） |
| **分代方式** | 物理分代 | 逻辑分代 |
| **大小调整** | 固定 | 动态 |
| **停顿时间** | 不可控 | 可控 |
| **内存碎片** | 有（老年代） | 无 |
| **适用堆大小** | < 8GB | > 4GB |
| **Full GC** | 频繁 | 罕见 |
| **调优难度** | 复杂 | 简单 |

### 9.6 何时使用G1

```
推荐使用G1的场景：
✅ 堆内存 > 4GB
✅ 需要可预测的停顿时间
✅ 应用响应时间敏感
✅ 老年代占用率变化大
✅ 对象分配速率高

继续使用传统分代的场景：
✅ 堆内存 < 4GB
✅ 追求最高吞吐量
✅ 停顿时间不敏感
✅ 应用稳定，GC参数已调优
```

### 9.7 G1配置示例

```bash
# 基本配置
java -XX:+UseG1GC \
     -Xms8g -Xmx8g \
     -XX:MaxGCPauseMillis=200 \
     -jar app.jar

# 详细配置
java -XX:+UseG1GC \
     -Xms8g -Xmx8g \
     -XX:MaxGCPauseMillis=200 \
     -XX:G1HeapRegionSize=4m \
     -XX:InitiatingHeapOccupancyPercent=45 \
     -XX:G1NewSizePercent=5 \
     -XX:G1MaxNewSizePercent=60 \
     -XX:ConcGCThreads=2 \
     -XX:ParallelGCThreads=8 \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -jar app.jar
```

## 十、总结

### 9.1 分代GC核心思想

```
1. 根据对象生命周期划分代
2. 不同代使用不同GC策略
3. 大部分GC只回收新生代
4. 减少Full GC频率
```

### 9.2 关键参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `-Xmn` | 新生代大小 | 堆的1/3 |
| `-XX:NewRatio` | 老年代/新生代比例 | 2 |
| `-XX:SurvivorRatio` | Eden/Survivor比例 | 8 |
| `-XX:MaxTenuringThreshold` | 晋升年龄阈值 | 15 |
| `-XX:PretenureSizeThreshold` | 大对象阈值 | 0（不启用） |

### 9.3 最佳实践

1. ✅ 合理设置堆大小和代比例
2. ✅ 避免创建大对象
3. ✅ 及时释放不再使用的引用
4. ✅ 使用对象池减少对象创建
5. ✅ 监控GC日志，及时调优
