# 垃圾收集器详解

## 💡 大白话精华总结

**垃圾收集器是什么？**
- 就是具体干活的"清洁工"，算法是"工作方法"，收集器是"具体的人"
- 不同的清洁工有不同的特长

**7大垃圾收集器（从老到新）：**

**1. Serial（串行）**
```
特点：单线程，简单
适用：单核CPU，客户端应用
比喻：一个人扫地，其他人都停下来等
```

**2. ParNew（并行）**
```
特点：Serial的多线程版本
适用：多核CPU，配合CMS使用
比喻：多个人一起扫地，其他人还是要等
```

**3. Parallel Scavenge（吞吐量优先）**
```
特点：关注吞吐量（干活时间/总时间）
适用：后台计算任务
比喻：追求效率，不在乎停顿时间长短
```

**4. Serial Old（串行老年代）**
```
特点：Serial的老年代版本
适用：客户端应用
```

**5. Parallel Old（并行老年代）**
```
特点：Parallel Scavenge的老年代版本
适用：配合Parallel Scavenge
```

**6. CMS（并发标记清除）**
```
特点：低延迟，与应用并发执行
缺点：内存碎片，CPU占用高
适用：响应时间敏感的应用（Web服务器）
比喻：边扫地边让人走，但扫得不彻底
```

**7. G1（Garbage First）**
```
特点：
- 分区域回收（化整为零）
- 可预测停顿时间
- 兼顾吞吐量和延迟
适用：大堆内存（>4GB），JDK 9+默认
比喻：把房间分成小块，哪块垃圾多先扫哪块
```

**8. ZGC（超低延迟）**
```
特点：
- 停顿时间<10ms
- 支持TB级堆
- JDK 11+
适用：超大堆，极低延迟要求
比喻：边走边扫，几乎不停顿
```

**如何选择收集器？**
```
堆内存 < 100MB  → Serial
堆内存 < 4GB    → ParNew + CMS 或 Parallel
堆内存 > 4GB    → G1（推荐）
超低延迟要求    → ZGC
高吞吐量要求    → Parallel
```

**重要参数：**
```bash
-XX:+UseSerialGC          # 使用Serial
-XX:+UseParNewGC          # 使用ParNew
-XX:+UseParallelGC        # 使用Parallel
-XX:+UseConcMarkSweepGC   # 使用CMS
-XX:+UseG1GC              # 使用G1（JDK 9+默认）
-XX:+UseZGC               # 使用ZGC
```
**一句话记住：**
> Serial单线程，Parallel多线程，CMS低延迟有碎片，G1分区可预测，ZGC超低延迟！

## 📚 开篇：一个真实的线上故障

> **场景**：某电商平台在双11大促时，系统突然出现卡顿，用户下单失败率飙升至30%。
> 
> **排查过程**：
> - 查看监控：发现每隔几秒就有一次长达2秒的停顿
> - 分析GC日志：发现使用的是CMS收集器，频繁Full GC
> - 问题根源：堆内存8GB，CMS产生大量内存碎片，无法分配大对象
> 
> **解决方案**：切换到G1收集器，停顿时间降至100ms以内，系统恢复正常。

**这个故事告诉我们**：选择合适的垃圾收集器，比任何代码优化都重要！

### 1.1 收集器分类（三个维度理解）

## 二、Serial收集器（单线程老兵）

### 💡 一句话理解
> Serial就像一个勤劳的清洁工，虽然只有一个人干活，但胜在简单可靠，适合小房子（小堆内存）。

### 2.1 工作原理图解

```
📱 场景：客户端应用（堆内存100MB）

时间线：
0ms          10ms         30ms         40ms
 |            |            |            |
 v            v            v            v
应用运行 ──→ [全部暂停] ──→ 应用恢复运行
              ↓
           GC线程单独工作
           (清理垃圾)

特点：
✅ 简单：只有一个线程，逻辑清晰
✅ 高效：没有线程切换开销
✅ 省内存：GC线程占用资源少
❌ 停顿：应用必须完全暂停
```

### 2.2 特点对比表

| 维度 | Serial | 现代收集器(G1) | 说明 |
|------|--------|----------------|------|
| **线程数** | 1个 | 多个 | 单打独斗 vs 团队作战 |
| **STW时间** | 较长(100ms+) | 较短(<100ms) | 堆越大差距越明显 |
| **内存占用** | 极低 | 较高 | 适合资源受限环境 |
| **适用堆大小** | <100MB | >1GB | 各有千秋 |
| **复杂度** | 简单 | 复杂 | 易于理解和调试 |

### 🎯 适用场景

```
✅ 推荐使用：
1. 桌面应用程序（小内存）
2. 嵌入式设备（资源受限）
3. 单核CPU环境
4. 开发测试环境（简单可控）

❌ 不推荐使用：
1. 服务端应用（停顿时间敏感）
2. 大堆内存应用（>1GB）
3. 多核服务器
4. 高并发场景
```

### 2.3 使用方式

```bash
# 启用Serial收集器
java -XX:+UseSerialGC -jar app.jar

# 参数说明：
# -XX:+UseSerialGC  新生代和老年代都使用Serial
```

### 2.4 示例

```java
/**
 * Serial GC演示
 * VM参数：-XX:+UseSerialGC -Xms20M -Xmx20M -Xmn10M -XX:+PrintGCDetails
 */
public class SerialGCDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        byte[] allocation1, allocation2, allocation3, allocation4;
        
        allocation1 = new byte[2 * _1MB];
        allocation2 = new byte[2 * _1MB];
        allocation3 = new byte[2 * _1MB];
        
        // 触发Minor GC
        allocation4 = new byte[4 * _1MB];
    }
}

// 输出示例：
// [GC (Allocation Failure) [DefNew: 6487K->671K(9216K), 0.0034567 secs] 6487K->4767K(19456K), 0.0035123 secs]
// DefNew: Serial收集器的新生代名称
```

## 三、ParNew收集器（多线程战队）

### 💡 一句话理解
> ParNew = Serial的多线程版本，就像从一个清洁工变成一个清洁团队，多核CPU下效率更高。

### 3.1 工作原理对比

```
🔄 Serial vs ParNew 对比

Serial（单线程）：
应用线程：████████ [停顿20ms] ████████
GC线程：          ▓▓▓▓▓▓▓▓▓▓

ParNew（4线程）：
应用线程：████████ [停顿6ms] ████████
GC线程1：         ▓▓▓
GC线程2：         ▓▓▓
GC线程3：         ▓▓▓
GC线程4：         ▓▓▓

结论：多核环境下，ParNew停顿时间约为Serial的 1/线程数
```

### 3.2 性能对比实测

| CPU核数 | Serial耗时 | ParNew耗时 | 加速比 | 说明 |
|---------|-----------|-----------|--------|------|
| 1核 | 100ms | 110ms | 0.9x | 单核下反而慢（线程切换开销） |
| 2核 | 100ms | 60ms | 1.7x | 接近理论值2x |
| 4核 | 100ms | 30ms | 3.3x | 接近理论值4x |
| 8核 | 100ms | 20ms | 5.0x | 边际效益递减 |

💡 **结论**：ParNew在多核CPU下才有优势，单核反而不如Serial！

### 🎯 适用场景

```
✅ 最佳搭档：ParNew + CMS
   - ParNew负责新生代（快速回收）
   - CMS负责老年代（低延迟）
   - 这是JDK 8时代的经典组合

✅ 推荐使用：
1. 多核服务器（2核+）
2. 配合CMS使用
3. 对延迟敏感的Web应用
4. 堆内存2-8GB

❌ 不推荐使用：
1. 单核CPU（用Serial更好）
2. JDK 9+（直接用G1）
3. 独立使用（必须配合老年代收集器）
```

### 3.3 使用方式

```bash
# 启用ParNew收集器
java -XX:+UseParNewGC -jar app.jar

# 设置GC线程数
java -XX:+UseParNewGC -XX:ParallelGCThreads=4 -jar app.jar

# 参数说明：
# -XX:+UseParNewGC  新生代使用ParNew
# -XX:ParallelGCThreads  GC线程数（默认等于CPU核数）
```

### 3.4 示例

```java
/**
 * ParNew GC演示
 * VM参数：-XX:+UseParNewGC -XX:+UseConcMarkSweepGC -Xms20M -Xmx20M -Xmn10M -XX:+PrintGCDetails
 */
public class ParNewGCDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        byte[] allocation1, allocation2, allocation3, allocation4;
        
        allocation1 = new byte[2 * _1MB];
        allocation2 = new byte[2 * _1MB];
        allocation3 = new byte[2 * _1MB];
        
        // 触发Minor GC（多线程并行）
        allocation4 = new byte[4 * _1MB];
    }
}

// 输出示例：
// [GC (Allocation Failure) [ParNew: 6487K->671K(9216K), 0.0023456 secs] 6487K->4767K(19456K), 0.0024012 secs]
// ParNew: ParNew收集器的名称
```

## 四、Parallel Scavenge收集器

### 4.1 概述

Parallel Scavenge是吞吐量优先的收集器，目标是达到可控的吞吐量。

```
吞吐量 = 运行用户代码时间 / (运行用户代码时间 + GC时间)

例如：
- 程序运行100分钟
- GC耗时1分钟
- 吞吐量 = 99 / 100 = 99%
```

### 4.2 特点

| 特点 | 说明 |
|------|------|
| **线程** | 多线程并行 |
| **算法** | 标记-复制 |
| **目标** | 高吞吐量 |
| **适用场景** | 后台计算、批处理 |
| **优点** | 吞吐量高 |
| **缺点** | 停顿时间可能较长 |

### 4.3 使用方式

```bash
# 启用Parallel Scavenge收集器
java -XX:+UseParallelGC -jar app.jar

# 设置最大GC停顿时间
java -XX:+UseParallelGC -XX:MaxGCPauseMillis=100 -jar app.jar

# 设置吞吐量目标
java -XX:+UseParallelGC -XX:GCTimeRatio=99 -jar app.jar

# 自适应调节策略
java -XX:+UseParallelGC -XX:+UseAdaptiveSizePolicy -jar app.jar

# 参数说明：
# -XX:+UseParallelGC  新生代使用Parallel Scavenge
# -XX:MaxGCPauseMillis  最大GC停顿时间（毫秒）
# -XX:GCTimeRatio  吞吐量目标（默认99，即1%的GC时间）
# -XX:+UseAdaptiveSizePolicy  自动调节新生代大小、Eden/Survivor比例等
```

### 4.4 示例

```java
/**
 * Parallel Scavenge GC演示
 * VM参数：-XX:+UseParallelGC -Xms20M -Xmx20M -Xmn10M -XX:+PrintGCDetails
 */
public class ParallelScavengeDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        byte[] allocation1, allocation2, allocation3, allocation4;
        
        allocation1 = new byte[2 * _1MB];
        allocation2 = new byte[2 * _1MB];
        allocation3 = new byte[2 * _1MB];
        
        // 触发Minor GC
        allocation4 = new byte[4 * _1MB];
    }
}

// 输出示例：
// [GC (Allocation Failure) [PSYoungGen: 6487K->671K(9216K)] 6487K->4767K(19456K), 0.0023456 secs]
// PSYoungGen: Parallel Scavenge的新生代名称
```

### 4.5 自适应调节

```java
/**
 * 自适应调节演示
 * VM参数：-XX:+UseParallelGC -XX:+UseAdaptiveSizePolicy -XX:+PrintAdaptiveSizePolicy
 */
public class AdaptiveSizePolicyDemo {
    
    private static final int _1MB = 1024 * 1024;
    
    public static void main(String[] args) {
        List<byte[]> list = new ArrayList<>();
        
        // 持续分配对象
        for (int i = 0; i < 100; i++) {
            list.add(new byte[_1MB]);
            
            if (i % 10 == 0) {
                System.out.println("已分配: " + (i + 1) + "MB");
            }
        }
        
        // JVM会自动调节：
        // - 新生代大小
        // - Eden/Survivor比例
        // - 晋升年龄阈值
        // 以达到最佳吞吐量
    }
}
```

## 五、Serial Old收集器

### 5.1 概述

Serial Old是Serial的老年代版本，使用标记-整理算法。

### 5.2 特点

| 特点 | 说明 |
|------|------|
| **线程** | 单线程 |
| **算法** | 标记-整理 |
| **STW** | 是 |
| **适用场景** | 客户端应用、CMS的后备方案 |

### 5.3 使用场景


## 七、CMS收集器（低延迟先驱）

### 💡 一句话理解
> CMS是第一个"边工作边打扫"的收集器，大部分时间与应用并发执行，停顿时间短，但会产生内存碎片。

### 🎯 设计理念

```
传统收集器（Serial/Parallel）：
工作模式：停下来 → 打扫干净 → 继续工作
问题：停顿时间长，用户体验差

CMS收集器：
工作模式：边工作边打扫（大部分时间）
优势：停顿时间短，响应速度快
代价：打扫不彻底（有碎片）、占用CPU资源

💡 适用场景：互联网应用、Web服务器、在线交易系统
```

### 7.1 真实案例：为什么需要CMS？

```
📊 某电商网站的痛点（使用Parallel GC）：

问题现象：
- 每次GC停顿200-500ms
- 用户点击按钮无响应
- 订单提交超时
- 用户体验极差

切换到CMS后：
- GC停顿降至50-100ms
- 用户几乎无感知
- 订单成功率提升15%
- 业务指标显著改善

结论：对于响应时间敏感的应用，CMS是救星！
```

### 7.2 工作流程详解（4个阶段）

```
📋 CMS的4个阶段（记住：2次短暂停顿 + 2次并发执行）

阶段1：初始标记（Initial Mark）⚡ STW
┌─────────────────────────────────────┐
│ 耗时：很短（几十ms）                  │
│ 工作：标记GC Roots直接关联的对象      │
│ 特点：快速扫描，只标记一层            │
└─────────────────────────────────────┘

阶段2：并发标记（Concurrent Mark）🔄 并发
┌─────────────────────────────────────┐
│ 耗时：最长（几秒）                    │
│ 工作：遍历整个对象图，标记所有存活对象 │
│ 特点：与应用线程同时运行，不停顿      │
└─────────────────────────────────────┘

阶段3：重新标记（Remark）⚡ STW
┌─────────────────────────────────────┐
│ 耗时：较短（几十到几百ms）            │
│ 工作：修正并发标记期间的变化          │
│ 特点：处理并发阶段产生的新垃圾        │
└─────────────────────────────────────┘

阶段4：并发清除（Concurrent Sweep）🔄 并发
┌─────────────────────────────────────┐
│ 耗时：较长（几秒）                    │
│ 工作：清除未标记的对象                │
│ 特点：与应用线程同时运行，不停顿      │
└─────────────────────────────────────┘

⏱️ 时间线可视化：

0s    0.05s  0.1s        3s    3.1s  3.3s        6s
|      |      |          |      |     |          |
应用 → [停] → 应用运行... → [停] → 应用运行...
       ↓                         ↓
     初始标记                  重新标记
              ↓                        ↓
           并发标记                 并发清除
           (3秒)                   (3秒)

总停顿时间：0.05s + 0.2s = 0.25s（仅占总时间的4%！）
```

### 7.3 特点

| 特点 | 说明 |
|------|------|
| **线程** | 多线程并发 |
| **算法** | 标记-清除 |
| **STW** | 初始标记和重新标记 |
| **优点** | 停顿时间短、并发执行 |
| **缺点** | 产生碎片、占用CPU资源、并发失败 |

### 7.4 使用方式

```bash

java -XX:+UseConcMarkSweepGC -jar app.jar

# 设置CMS参数
java -XX:+UseConcMarkSweepGC \
     -XX:CMSInitiatingOccupancyFraction=75 \
     -XX:+UseCMSInitiatingOccupancyOnly \
     -XX:+CMSScavengeBeforeRemark \

# 参数说明：
```


### 7.6 CMS的三大致命缺陷（必须了解！）

#### ⚠️ 缺陷1：内存碎片（最常见问题）

```
🔴 问题现象：
某电商系统运行3天后，突然频繁Full GC

GC日志：
[Full GC (Allocation Failure) 6.5G->6.5G(8G), 5.234 secs]
[Full GC (Allocation Failure) 6.5G->6.5G(8G), 5.456 secs]
...

原因分析：
┌─────────────────────────────────────┐
│ 老年代内存布局（碎片化）：            │
│ [对象][空][对象][空][对象][空]...     │
│                                     │
│ 虽然总空闲空间有1.5GB                │
│ 但最大连续空间只有100MB               │
│ 无法分配200MB的大对象 ❌              │
└─────────────────────────────────────┘

✅ 解决方案：
// 方案1：定期整理内存
-XX:+UseCMSCompactAtFullCollection      // Full GC时整理
-XX:CMSFullGCsBeforeCompaction=5        // 5次Full GC后整理一次

// 方案2：升级到G1（推荐）
-XX:+UseG1GC  // G1使用复制算法，无碎片问题
```

#### 🔥 缺陷2：并发失败（Concurrent Mode Failure）

```
🔴 问题现象：
[CMS-concurrent-mark: 2.345 secs]
[CMS-concurrent-sweep: 1.234 secs]
[CMS Concurrent Mode Failure]  ⚠️ 致命错误！
[Full GC (Allocation Failure) 7.8G->5.2G(8G), 8.567 secs]

问题链路：
1. CMS并发清除时，应用继续运行
2. 应用不断产生新对象（浮动垃圾）
3. 老年代空间不足，无法容纳新对象
4. CMS宣告失败，降级使用Serial Old
5. 触发漫长的Full GC（停顿8秒！）

💡 为什么这么严重？
Serial Old是单线程的，8GB堆可能停顿10秒以上！

✅ 解决方案：
// 方案1：提前触发CMS
-XX:CMSInitiatingOccupancyFraction=70   // 老年代70%时触发（默认92%）
-XX:+UseCMSInitiatingOccupancyOnly      // 只使用设定阈值

// 方案2：增大堆内存
-Xms10g -Xmx10g  // 给CMS更多缓冲空间

// 方案3：优化代码，减少对象晋升
```

#### 💻 缺陷3：CPU资源占用

```
🔴 问题现象：
应用响应时间变慢，但GC日志显示停顿时间正常

原因：
CMS并发阶段占用CPU资源，影响应用性能

默认GC线程数 = (CPU核数 + 3) / 4

实际影响：
┌──────────┬────────────┬──────────┬────────────┐
│ CPU核数  │ GC线程数   │ 占用率   │ 影响       │
├──────────┼────────────┼──────────┼────────────┤
│ 2核      │ 1个        │ 50%      │ 严重 ❌    │
│ 4核      │ 1个        │ 25%      │ 明显 ⚠️    │
│ 8核      │ 2个        │ 25%      │ 可接受 ✅  │
│ 16核     │ 4个        │ 25%      │ 可接受 ✅  │
└──────────┴────────────┴──────────┴────────────┘

结论：CMS不适合CPU核数少的服务器！

✅ 解决方案：
// 方案1：增加CPU核数（治标不治本）
// 方案2：升级到G1（推荐）
-XX:+UseG1GC
```

#### 📊 CMS三大缺陷总结

| 缺陷 | 严重程度 | 发生频率 | 解决难度 | 推荐方案 |
|------|---------|---------|---------|---------|
| 内存碎片 | ⭐⭐⭐⭐ | 高 | 中 | 升级G1 |
| 并发失败 | ⭐⭐⭐⭐⭐ | 中 | 高 | 调参/升级G1 |
| CPU占用 | ⭐⭐⭐ | 低 | 低 | 增加CPU |

💡 **终极建议**：如果你在JDK 8+，直接用G1，避开CMS的所有坑！

## 八、G1收集器（现代首选，王者归来）

### 💡 一句话理解
> G1是"化整为零"的智能收集器，把堆分成小块，哪块垃圾多就先收拾哪块，既能控制停顿时间，又能保证吞吐量。

### 🎯 为什么G1成为默认收集器？

```
📊 G1 vs 传统收集器的对比

传统收集器的困境：
┌─────────────────────────────────────┐
│ Serial/Parallel：停顿时间长          │
│ CMS：内存碎片、并发失败               │
│ 所有收集器：堆越大，问题越严重        │
└─────────────────────────────────────┘

G1的突破：
✅ 可预测停顿：设置目标停顿时间（如100ms）
✅ 无内存碎片：使用复制算法
✅ 适合大堆：支持几十GB堆，停顿仍可控
✅ 智能回收：优先回收垃圾最多的区域
✅ 简化配置：大多数情况只需设置堆大小和停顿目标

🏆 结论：G1是CMS的完美替代者！
```

### 8.1 核心创新：Region分区模型

```
传统分代模型（Serial/Parallel/CMS）：
┌────────────────────────────────────────────────┐
│ [新生代: Eden | S0 | S1] [老年代............] │
└────────────────────────────────────────────────┘
问题：必须连续内存，堆大了就慢

G1的Region模型：
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ E │ E │ S │ O │ O │ E │ S │ O │  E=Eden
├───┼───┼───┼───┼───┼───┼───┼───┤  S=Survivor
│ E │ O │ O │ E │ S │ O │ H │ H │  O=Old
├───┼───┼───┼───┼───┼───┼───┼───┤  H=Humongous(大对象)
│ O │ E │ S │ O │ E │ O │ O │ E │
└───┴───┴───┴───┴───┴───┴───┴───┘

优势：
1. 化整为零：堆被分成2048个小Region（每个1-32MB）
2. 灵活分配：每个Region可以是任意代
3. 按需回收：每次只回收部分Region
4. 无需连续：不需要连续内存空间
```

### 8.2 内存布局

```
传统分代：
[新生代：Eden | S0 | S1] [老年代]

G1分区：
+---+---+---+---+---+---+---+---+
| E | E | S | O | O | E | S | O |
+---+---+---+---+---+---+---+---+
| E | O | O | E | S | O | E | O |
+---+---+---+---+---+---+---+---+

E: Eden区
S: Survivor区
O: Old区
H: Humongous区（大对象）

特点：
- 将堆划分为多个大小相等的Region（1-32MB）
- 每个Region可以是Eden、Survivor或Old
- 不需要连续的内存空间
```

### 8.3 工作流程

```
G1 GC的阶段：

1. Young GC
   - 回收Eden区和Survivor区
   - 存活对象复制到Survivor或Old区
   - STW，但时间可控

2. Mixed GC
   - 回收整个Young区 + 部分Old区
   - 选择回收价值最大的Region
   - STW，但时间可控

3. Full GC
   - 回收整个堆
   - 使用Serial Old算法
   - STW时间长（应该避免）
```

### 8.4 特点

| 特点 | 说明 |
|------|------|
| **内存布局** | Region分区 |
| **算法** | 标记-复制 + 标记-整理 |
| **停顿时间** | 可预测 |
| **适用场景** | 大堆内存、低延迟要求 |
| **优点** | 停顿时间可控、无碎片 |
| **缺点** | 内存占用高、CPU占用高 |

### 8.5 使用方式

```bash
# 启用G1收集器
java -XX:+UseG1GC -jar app.jar

# 设置停顿时间目标
java -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -jar app.jar

# 设置Region大小
java -XX:+UseG1GC -XX:G1HeapRegionSize=16m -jar app.jar

# 设置并发GC线程数
java -XX:+UseG1GC -XX:ConcGCThreads=4 -jar app.jar

# 参数说明：
# -XX:+UseG1GC  启用G1
# -XX:MaxGCPauseMillis  最大停顿时间目标（默认200ms）
# -XX:G1HeapRegionSize  Region大小（1-32MB，必须是2的幂）
# -XX:ConcGCThreads  并发GC线程数
# -XX:InitiatingHeapOccupancyPercent  触发并发GC的堆占用率（默认45%）
```

### 8.6 示例

### 8.7 G1的优势

```
1. 可预测的停顿时间
   - 设置停顿时间目标
   - G1会尽力达到目标

2. 不产生内存碎片
   - 使用复制算法
   - 整理内存

3. 适合大堆内存
   - 支持几十GB的堆
   - 停顿时间仍然可控

4. 并发执行
   - 大部分工作与应用并发
   - 减少停顿时间
```

### 8.8 G1的核心概念

#### 1. Region（区域）

```
Region特点：
- 堆被划分为2048个左右的Region
- 每个Region大小相同（1-32MB，必须是2的幂）
- Region大小 = 堆大小 / 2048

计算示例：
- 4GB堆：4096MB / 2048 = 2MB/Region
- 8GB堆：8192MB / 2048 = 4MB/Region
- 16GB堆：16384MB / 2048 = 8MB/Region

Region类型：
- Eden：新对象分配区
- Survivor：存活对象区
- Old：老年代区
- Humongous：大对象区（对象大小 > Region的50%）
```

#### 2. SATB（Snapshot-At-The-Beginning）

```
SATB算法：
- 在并发标记开始时，对堆做一个逻辑快照
- 标记快照中的所有存活对象
- 并发标记期间新分配的对象都认为是存活的

作用：
- 保证并发标记的正确性
- 避免漏标（对象被误回收）
```

#### 3. RSet（Remembered Set）

```
RSet作用：
- 记录其他Region中的对象引用本Region对象的关系
- 避免扫描整个堆来找引用
- 加速GC

示例：
Region A的对象 → 引用 → Region B的对象
那么Region B的RSet会记录这个引用

好处：
- 回收Region B时，只需扫描RSet
- 不需要扫描整个堆
```

#### 4. CSet（Collection Set）

```
CSet定义：
- 本次GC要回收的Region集合
- G1会选择回收价值最大的Region

选择策略：
1. 所有Eden Region（必选）
2. 部分Survivor Region
3. 部分Old Region（Mixed GC时）

回收价值 = 垃圾对象数量 / 回收耗时
```

### 8.9 G1的GC过程详解

#### 1. Young GC（年轻代GC）

```
触发条件：
- Eden Region满了

回收过程：
1. 选择所有Eden Region和Survivor Region
2. 将存活对象复制到新的Survivor或Old Region
3. 清空原Region

特点：
- STW（Stop The World）
- 并行执行
- 时间可控（通过MaxGCPauseMillis）

示例：
[GC pause (G1 Evacuation Pause) (young) 256M->45M(1024M), 0.0123456 secs]
```

#### 2. Concurrent Marking（并发标记）

```
触发条件：
- 堆占用率达到InitiatingHeapOccupancyPercent（默认45%）

标记阶段：
1. 初始标记（Initial Mark）- STW
   - 标记GC Roots直接可达的对象
   - 通常与Young GC一起进行
   
2. 根区域扫描（Root Region Scan）- 并发
   - 扫描Survivor Region
   - 标记被引用的对象
   
3. 并发标记（Concurrent Mark）- 并发
   - 遍历整个堆
   - 标记存活对象
   - 耗时最长
   
4. 重新标记（Remark）- STW
   - 处理并发标记期间的变化
   - 使用SATB算法
   
5. 清理（Cleanup）- 部分STW
   - 统计每个Region的存活对象
   - 回收完全空闲的Region
   - 重置RSet

时间线：
应用线程：运行 → 暂停 → 运行 ──────→ 暂停 → 运行 ──→ 运行
                  ↓           ↓              ↓
GC线程：         初始  →  并发标记  →  重新  →  清理
                标记                  标记
```

#### 3. Mixed GC（混合GC）

```
触发条件：
- 并发标记完成后
- 老年代占用率达到阈值

回收过程：
1. 选择回收价值最大的Region
   - 所有Eden Region
   - 部分Survivor Region
   - 部分Old Region（垃圾最多的）
   
2. 将存活对象复制到新Region
3. 清空原Region

特点：
- STW
- 并行执行
- 分多次进行（不是一次回收所有Old Region）

参数控制：
-XX:G1MixedGCCountTarget=8  # Mixed GC次数（默认8次）
-XX:G1MixedGCLiveThresholdPercent=85  # Region存活率阈值

示例：
[GC pause (G1 Evacuation Pause) (mixed) 512M->256M(1024M), 0.0234567 secs]
```

#### 4. Full GC（完全GC）

```
触发条件（应该避免）：
1. Mixed GC无法跟上对象分配速度
2. 大对象分配失败
3. 显式调用System.gc()

回收过程：
- 使用Serial Old算法
- 单线程
- STW时间很长

示例：
[Full GC (Allocation Failure) 1024M->512M(1024M), 2.3456789 secs]
```

### 8.10 G1调优参数详解

#### 核心参数

```bash
# 1. 堆大小设置
-Xms4g -Xmx4g  # 初始堆和最大堆（建议相等）

# 2. 停顿时间目标
-XX:MaxGCPauseMillis=200  # 最大停顿时间（默认200ms）
# 注意：这是目标，不是保证！设置过小会导致频繁GC

# 3. Region大小
-XX:G1HeapRegionSize=16m  # Region大小（1-32MB，2的幂）
# 自动计算：堆大小 / 2048
# 建议：让大对象不超过Region的50%

# 4. 并发标记触发阈值
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用率达到45%时触发并发标记
# 调大：减少GC频率，但可能导致Full GC
# 调小：增加GC频率，但避免Full GC

# 5. 并发GC线程数
-XX:ConcGCThreads=4  # 并发GC线程数
# 默认：ParallelGCThreads / 4
# 建议：CPU核数的1/4

# 6. 并行GC线程数
-XX:ParallelGCThreads=8  # 并行GC线程数
# 默认：CPU核数（核数 <= 8时）
# 默认：8 + (核数 - 8) * 5/8（核数 > 8时）

# 7. Mixed GC参数
-XX:G1MixedGCCountTarget=8  # Mixed GC次数
-XX:G1MixedGCLiveThresholdPercent=85  # Region存活率阈值
-XX:G1HeapWastePercent=5  # 允许的堆浪费百分比

# 8. 大对象阈值
-XX:G1HeapRegionSize=16m  # 间接控制大对象阈值（Region的50%）
```

#### 推荐配置

```bash
# 小型应用（4GB堆）
java -XX:+UseG1GC \
     -Xms4g -Xmx4g \
     -XX:MaxGCPauseMillis=200 \
     -XX:G1HeapRegionSize=2m \
     -XX:InitiatingHeapOccupancyPercent=45 \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -jar app.jar

# 中型应用（8GB堆）
java -XX:+UseG1GC \
     -Xms8g -Xmx8g \
     -XX:MaxGCPauseMillis=100 \
     -XX:G1HeapRegionSize=4m \
     -XX:InitiatingHeapOccupancyPercent=40 \
     -XX:ConcGCThreads=2 \
     -XX:ParallelGCThreads=8 \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -jar app.jar

# 大型应用（16GB堆）
java -XX:+UseG1GC \
     -Xms16g -Xmx16g \
     -XX:MaxGCPauseMillis=100 \
     -XX:G1HeapRegionSize=8m \
     -XX:InitiatingHeapOccupancyPercent=35 \
     -XX:ConcGCThreads=4 \
     -XX:ParallelGCThreads=16 \
     -XX:G1MixedGCCountTarget=8 \
     -XX:G1HeapWastePercent=5 \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -jar app.jar
```

### 8.11 G1常见问题和解决方案（实战踩坑指南）

#### 🔥 问题1：频繁Full GC（最严重！）

```
🔴 真实案例：某社交平台API服务

问题现象：
[Full GC (Allocation Failure) 16G->12G(16G), 5.123 secs]
[Full GC (Allocation Failure) 16G->12G(16G), 5.234 secs]
每小时触发3-5次Full GC，每次停顿5秒+

影响：
- 用户请求超时
- 服务降级
- 告警狂响

排查步骤：
1️⃣ 查看GC日志，确认是Full GC
2️⃣ 分析堆使用情况：16G堆，回收后仍有12G
3️⃣ 检查对象分配速率：每秒500MB
4️⃣ 发现问题：Mixed GC跟不上对象分配速度

根因分析：
┌─────────────────────────────────────┐
│ 对象分配速度 > Mixed GC回收速度      │
│ ↓                                   │
│ 老年代持续增长                       │
│ ↓                                   │
│ 无法分配新对象                       │
│ ↓                                   │
│ 触发Full GC（最后的杀手锏）          │
└─────────────────────────────────────┘

✅ 解决方案（按优先级）：

方案1：增大堆内存（立竿见影）
-Xms20g -Xmx20g  // 从16G增到20G

方案2：提前触发并发标记（治本）
-XX:InitiatingHeapOccupancyPercent=35  // 从45%降到35%

方案3：增加并发GC线程（加速回收）
-XX:ConcGCThreads=4  // 从2增到4

方案4：增大Region（如果有大对象）
-XX:G1HeapRegionSize=16m  // 从8M增到16M

实施效果：
✅ Full GC频率：3-5次/小时 → 0次/天
✅ 停顿时间：5秒 → 100ms以内
✅ 服务稳定性：大幅提升
```

#### ⏱️ 问题2：停顿时间过长

```
🔴 真实案例：某电商订单服务

问题现象：
[GC pause (G1 Evacuation Pause) (young) 4G->2G(8G), 0.5123 secs]
目标停顿100ms，实际停顿500ms+

影响：
- 订单提交慢
- 用户体验差

排查步骤：
1️⃣ 检查GC日志：Young GC停顿500ms
2️⃣ 分析停顿原因：存活对象太多，复制耗时
3️⃣ 检查配置：MaxGCPauseMillis=100（设置过于激进）

根因：
停顿目标设置过小，G1为了达标：
- 减少每次回收的Region数量
- 导致GC频率增加
- 反而影响吞吐量

✅ 解决方案：

方案1：放宽停顿目标（最有效）
-XX:MaxGCPauseMillis=200  // 从100ms调到200ms
💡 不要设置过小的目标，G1会尽力达成但不保证

方案2：调整新生代比例
-XX:G1NewSizePercent=5      // 最小5%
-XX:G1MaxNewSizePercent=50  // 最大50%（从60%降低）

方案3：增加并行GC线程
-XX:ParallelGCThreads=16  // 利用更多CPU

实施效果：
✅ 停顿时间：500ms → 150ms
✅ GC频率：降低30%
✅ 吞吐量：提升20%
```

#### 📉 问题3：吞吐量下降

```
🔴 真实案例：某数据处理服务

问题现象：
- 应用响应变慢
- CPU使用率高（80%+）
- GC日志显示频繁GC

排查步骤：
1️⃣ 监控CPU：发现GC线程占用25%
2️⃣ 分析GC频率：每秒触发1次Young GC
3️⃣ 检查堆大小：8G堆，对象分配速率高

根因：
堆内存不足 + 停顿目标过小 → GC频繁 → 吞吐量下降

✅ 解决方案：

方案1：增大堆内存（首选）
-Xms16g -Xmx16g  // 翻倍

方案2：放宽停顿目标
-XX:MaxGCPauseMillis=200  // 从100调到200

方案3：调整并发线程数
-XX:ConcGCThreads=2  // 从4降到2，减少CPU占用

方案4：提高触发阈值
-XX:InitiatingHeapOccupancyPercent=50  // 从45%提高到50%

实施效果：
✅ GC频率：1次/秒 → 1次/10秒
✅ CPU占用：80% → 50%
✅ 吞吐量：提升40%
```

#### 🐘 问题4：大对象分配失败

```
🔴 真实案例：某图片处理服务

问题现象：
[GC pause (G1 Humongous Allocation) (young) (to-space exhausted)]
java.lang.OutOfMemoryError: Java heap space

排查步骤：
1️⃣ 分析对象大小：发现大量20MB+的图片对象
2️⃣ 检查Region大小：8MB（太小！）
3️⃣ 计算：20MB对象需要3个连续Region

根因：
大对象 > Region的50% → 需要连续Region → 分配困难

✅ 解决方案：

方案1：增大Region大小（治本）
-XX:G1HeapRegionSize=32m  // 从8M增到32M
💡 让大对象 < Region的50%

方案2：增大堆内存
-Xms20g -Xmx20g

方案3：优化代码（最佳）
// 分块处理图片，避免一次性加载大对象
BufferedImage image = ImageIO.read(new File("large.jpg"));
// 改为流式处理

实施效果：
✅ OOM错误：消失
✅ GC停顿：减少50%
✅ 内存利用率：提升30%
```

#### 📊 问题总结表

| 问题 | 症状 | 根因 | 优先方案 | 难度 |
|------|------|------|---------|------|
| 频繁Full GC | 长时间停顿 | 堆不足/回收慢 | 增大堆内存 | ⭐⭐ |
| 停顿过长 | 超过目标 | 目标过小 | 放宽目标 | ⭐ |
| 吞吐量低 | CPU高/慢 | GC频繁 | 增大堆/放宽目标 | ⭐⭐ |
| 大对象失败 | OOM | Region太小 | 增大Region | ⭐⭐⭐ |

💡 **调优心法**：
1. 先看日志，找到问题类型
2. 分析根因，不要盲目调参
3. 一次只改一个参数，观察效果
4. 记录调优前后的对比数据

## 九、ZGC收集器

### 9.1 概述

ZGC是JDK 11引入的超低延迟垃圾收集器，目标是停顿时间不超过10ms。

```
设计目标：
- 停顿时间 < 10ms
- 支持TB级堆内存
- 吞吐量下降 < 15%
```

### 9.2 特点

| 特点 | 说明 |
|------|------|
| **停顿时间** | < 10ms |
| **堆大小** | 支持TB级 |
| **算法** | 标记-复制 |
| **并发** | 几乎全程并发 |
| **适用场景** | 超大堆、超低延迟 |

### 9.3 使用方式

```bash
# 启用ZGC（JDK 11+）
java -XX:+UseZGC -jar app.jar

# 设置堆大小
java -XX:+UseZGC -Xms16g -Xmx16g -jar app.jar

# 参数说明：
# -XX:+UseZGC  启用ZGC
# -XX:ConcGCThreads  并发GC线程数
# -XX:ZCollectionInterval  GC间隔（秒）
```

### 9.4 关键技术

```
1. 染色指针（Colored Pointers）
   - 在指针中存储元数据
   - 标记、重定位信息

2. 读屏障（Load Barrier）
   - 对象访问时检查指针
   - 自动完成重定位

3. 并发重定位
   - 移动对象与应用并发
   - 不需要STW
```

### 🎯 核心要点总结

#### 1. 收集器选择三原则

```
原则1：优先考虑业务需求
├─ 延迟敏感（Web/API）→ G1/ZGC
├─ 吞吐量优先（批处理）→ Parallel
└─ 资源受限（嵌入式）→ Serial

原则2：根据堆大小选择
├─ <2GB    → Serial
├─ 2-8GB   → G1
├─ 8-64GB  → G1
└─ >64GB   → ZGC

原则3：跟随JDK版本
├─ JDK 8   → G1（手动开启）
├─ JDK 9+  → G1（默认）
└─ JDK 11+ → G1/ZGC
```

#### 2. 常见误区避坑指南

```
❌ 误区1：盲目追求低延迟
   - 错误：MaxGCPauseMillis=10（设置过小）
   - 后果：GC频繁，吞吐量暴跌
   - 正确：根据业务需求设置合理目标（100-200ms）

❌ 误区2：堆内存越大越好
   - 错误：给8GB应用分配64GB堆
   - 后果：浪费资源，GC停顿更长
   - 正确：根据实际需求合理分配（留20-30%余量）

❌ 误区3：过度调优
   - 错误：修改几十个GC参数
   - 后果：系统复杂，难以维护
   - 正确：先用默认配置，有问题再调优

❌ 误区4：忽略监控
   - 错误：上线后不看GC日志
   - 后果：问题发现晚，影响大
   - 正确：持续监控GC指标，及时发现问题

❌ 误区5：频繁切换收集器
   - 错误：今天CMS，明天G1，后天ZGC
   - 后果：不稳定，难以积累经验
   - 正确：选定后长期使用，充分测试
```

## 十二、GC触发时机全解析

### 📋 概述

不同的垃圾收集器有不同的GC触发机制。理解GC触发时机对于调优和问题排查至关重要。

### 12.1 GC类型说明

```
GC类型分类：

Young GC（Minor GC）
├─ 只回收新生代
├─ 频率高，速度快
└─ STW时间短（几十ms）

Old GC（Major GC）
├─ 只回收老年代
├─ 仅CMS有此概念
└─ 其他收集器没有单独的Old GC

Mixed GC（混合GC）
├─ 回收新生代 + 部分老年代
├─ 仅G1有此概念
└─ 选择回收价值最大的Region

Full GC（完全GC）
├─ 回收整个堆（新生代 + 老年代）
├─ 频率低，速度慢
└─ STW时间长（秒级）
```

---

### 12.2 Serial / Serial Old 触发时机

#### Young GC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. Eden区满                              │
│    ├─ Eden空间不足以分配新对象           │
│    └─ 触发Minor GC                       │
│                                         │
│ 2. 显式调用System.gc()                   │
│    ├─ 建议JVM进行GC                      │
│    └─ 可能触发Full GC                    │
└─────────────────────────────────────────┘

GC日志示例：
[GC (Allocation Failure) [DefNew: 6487K->671K(9216K), 0.0034567 secs]
                          ↑
                    Eden区分配失败
```

#### Full GC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. 老年代空间不足                        │
│    ├─ 对象晋升时老年代空间不够           │
│    └─ 触发Full GC                        │
│                                         │
│ 2. 永久代/元空间不足（JDK 7-）           │
│    ├─ 类加载过多                         │
│    └─ 触发Full GC                        │
│                                         │
│ 3. System.gc()                          │
│    ├─ 显式调用                           │
│    └─ 触发Full GC                        │
│                                         │
│ 4. 空间分配担保失败                      │
│    ├─ Minor GC前检查失败                 │
│    └─ 触发Full GC                        │
└─────────────────────────────────────────┘
```

---

### 12.3 ParNew + CMS 触发时机

#### Young GC触发条件（ParNew）

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. Eden区满                              │
│    └─ 触发ParNew GC                      │
│                                         │
│ 2. 大对象直接进入老年代失败              │
│    └─ 可能触发Young GC                   │
└─────────────────────────────────────────┘

GC日志示例：
[GC (Allocation Failure) [ParNew: 6487K->671K(9216K), 0.0023456 secs]
```

#### CMS GC触发条件（老年代）

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. 老年代使用率达到阈值                  │
│    ├─ 默认：92%                          │
│    ├─ 参数：-XX:CMSInitiatingOccupancyFraction=70 │
│    └─ 触发CMS并发GC                      │
│                                         │
│ 2. 永久代/元空间使用率达到阈值           │
│    ├─ 默认：92%                          │
│    └─ 触发CMS GC                         │
│                                         │
│ 3. 新生代晋升担保失败                    │
│    ├─ 预测老年代空间不足                 │
│    └─ 提前触发CMS GC                     │
└─────────────────────────────────────────┘

CMS GC阶段：
1. Initial Mark（初始标记）- STW
2. Concurrent Mark（并发标记）- 并发
3. Concurrent Preclean（并发预清理）- 并发
4. Final Remark（重新标记）- STW
5. Concurrent Sweep（并发清除）- 并发
6. Concurrent Reset（并发重置）- 并发

GC日志示例：
[GC (CMS Initial Mark) [1 CMS-initial-mark: 10812K(20480K)] 14012K(29696K), 0.0001234 secs]
[CMS-concurrent-mark-start]
[CMS-concurrent-mark: 0.0123456 secs]
```

#### Full GC触发条件（降级）

```
触发时机（CMS失败场景）：
┌─────────────────────────────────────────┐
│ 1. Concurrent Mode Failure               │
│    ├─ CMS并发清除时，老年代空间不足      │
│    ├─ 无法容纳新晋升的对象               │
│    └─ 降级使用Serial Old进行Full GC      │
│                                         │
│ 2. Promotion Failed                     │
│    ├─ 内存碎片导致大对象分配失败         │
│    └─ 触发Full GC                        │
│                                         │
│ 3. 永久代/元空间不足                     │
│    └─ 触发Full GC                        │
└─────────────────────────────────────────┘

GC日志示例：
[CMS-concurrent-mark: 2.345 secs]
[CMS Concurrent Mode Failure]  ⚠️ CMS失败！
[Full GC (Allocation Failure) 7.8G->5.2G(8G), 8.567 secs]
                                ↑
                        降级为Serial Old Full GC
```

### 12.4 Parallel Scavenge + Parallel Old 触发时机

#### Young GC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. Eden区满                              │
│    └─ 触发Parallel Scavenge GC          │
│                                         │
│ 2. 自适应调节触发                        │
│    ├─ -XX:+UseAdaptiveSizePolicy        │
│    └─ JVM自动调整触发时机                │
└─────────────────────────────────────────┘

GC日志示例：
[GC (Allocation Failure) [PSYoungGen: 6487K->671K(9216K)] 6487K->4767K(19456K), 0.0023456 secs]
```

#### Full GC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. 老年代空间不足                        │
│    └─ 触发Parallel Old Full GC          │
│                                         │
│ 2. 永久代/元空间不足                     │
│    └─ 触发Full GC                        │
│                                         │
│ 3. System.gc()                          │
│    └─ 触发Full GC                        │
│                                         │
│ 4. 空间分配担保失败                      │
│    └─ 触发Full GC                        │
│                                         │
│ 5. 吞吐量目标未达成                      │
│    ├─ -XX:GCTimeRatio=99                │
│    └─ 自适应调整，可能触发Full GC        │
└─────────────────────────────────────────┘

GC日志示例：
[Full GC (Ergonomics) [PSYoungGen: 1024K->0K(2048K)] [ParOldGen: 6144K->5120K(8192K)] 7168K->5120K(10240K), 0.123 secs]
```

### 12.5 G1 触发时机（重点）

#### Young GC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. Eden Region满                         │
│    ├─ 所有Eden Region都被占用            │
│    └─ 触发Young GC（Evacuation Pause）   │
│                                         │
│ 2. 无法分配新对象                        │
│    ├─ Eden区没有空闲Region               │
│    └─ 触发Young GC                       │
│                                         │
│ 3. 达到停顿时间目标                      │
│    ├─ -XX:MaxGCPauseMillis=200          │
│    └─ G1自动调整GC频率                   │
└─────────────────────────────────────────┘

特点：
- 回收所有Eden Region
- 回收所有Survivor Region
- 存活对象复制到新的Survivor或Old Region
- STW，但时间可控

GC日志示例：
[GC pause (G1 Evacuation Pause) (young) 256M->45M(1024M), 0.0123456 secs]
   [Parallel Time: 10.5 ms, GC Workers: 8]
   [Eden: 200.0M(200.0M)->0.0B(180.0M) Survivors: 20.0M->30.0M Heap: 256.0M(1024.0M)->45.0M(1024.0M)]
```

#### 并发标记触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. 堆占用率达到阈值                      │
│    ├─ 默认：45%                          │
│    ├─ 参数：-XX:InitiatingHeapOccupancyPercent=45 │
│    └─ 触发并发标记周期                   │
│                                         │
│ 2. 上次Mixed GC后堆占用率仍然高          │
│    └─ 继续触发并发标记                   │
│                                         │
│ 3. Humongous对象分配                    │
│    ├─ 大对象分配可能触发                 │
│    └─ 提前启动并发标记                   │
└─────────────────────────────────────────┘

并发标记阶段：
1. Initial Mark（初始标记）- STW，通常与Young GC一起
2. Root Region Scan（根区域扫描）- 并发
3. Concurrent Mark（并发标记）- 并发
4. Remark（重新标记）- STW
5. Cleanup（清理）- 部分STW

GC日志示例：
[GC pause (G1 Evacuation Pause) (young) (initial-mark) 512M->256M(1024M), 0.0234 secs]
                                         ↑
                                  初始标记与Young GC一起
[GC concurrent-root-region-scan-start]
[GC concurrent-root-region-scan-end, 0.0012 secs]
[GC concurrent-mark-start]
[GC concurrent-mark-end, 2.345 secs]
```

#### Mixed GC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. 并发标记完成后                        │
│    ├─ 标记出了垃圾最多的Old Region       │
│    └─ 触发Mixed GC                       │
│                                         │
│ 2. 老年代占用率达到阈值                  │
│    ├─ 参数：-XX:G1MixedGCLiveThresholdPercent=85 │
│    └─ Region存活率 < 85%时才回收         │
│                                         │
│ 3. Mixed GC次数未达到目标                │
│    ├─ 参数：-XX:G1MixedGCCountTarget=8  │
│    └─ 继续触发Mixed GC                   │
└─────────────────────────────────────────┘

特点：
- 回收所有Eden Region
- 回收所有Survivor Region
- 回收部分Old Region（垃圾最多的）
- 分多次进行，不是一次回收所有Old Region

GC日志示例：
[GC pause (G1 Evacuation Pause) (mixed) 512M->256M(1024M), 0.0234567 secs]
   [Eden: 200.0M(200.0M)->0.0B(180.0M) Survivors: 30.0M->20.0M Heap: 512.0M(1024.0M)->256.0M(1024.0M)]
```

#### Full GC触发条件（应该避免）

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. Evacuation Failure（疏散失败）        │
│    ├─ to-space exhausted                │
│    ├─ 没有足够空间容纳存活对象           │
│    └─ 触发Full GC                        │
│                                         │
│ 2. Humongous对象分配失败                 │
│    ├─ 找不到足够的连续Region             │
│    └─ 触发Full GC整理内存                │
│                                         │
│ 3. Mixed GC无法跟上对象分配速度          │
│    ├─ 老年代持续增长                     │
│    └─ 触发Full GC                        │
│                                         │
│ 4. 元空间不足                            │
│    └─ 触发Full GC                        │
│                                         │
│ 5. System.gc()                          │
│    └─ 触发Full GC                        │
└─────────────────────────────────────────┘

特点：
- 使用Serial Old算法
- 单线程
- STW时间很长（秒级）
- 应该尽量避免

GC日志示例：
[Full GC (Allocation Failure) 1024M->512M(1024M), 2.3456789 secs]
   [Eden: 0.0B(100.0M)->0.0B(100.0M) Survivors: 0.0B->0.0B Heap: 1024.0M(1024.0M)->512.0M(1024.0M)], [Metaspace: 3456K->3456K(1056768K)]
```

### 12.6 ZGC 触发时机

#### ZGC触发条件

```
触发时机：
┌─────────────────────────────────────────┐
│ 1. 定时触发                              │
│    ├─ 参数：-XX:ZCollectionInterval=<秒> │
│    └─ 默认：不定时，按需触发             │
│                                         │
│ 2. 堆使用率达到阈值                      │
│    ├─ 自适应算法计算                     │
│    └─ 提前触发，避免分配失败             │
│                                         │
│ 3. 分配速率过快                          │
│    ├─ ZGC预测堆即将满                    │
│    └─ 提前触发GC                         │
│                                         │
│ 4. 元空间分配                            │
│    └─ 触发ZGC                            │
│                                         │
│ 5. 显式触发                              │
│    └─ System.gc()                       │
└─────────────────────────────────────────┘

特点：
- 几乎全程并发
- 停顿时间 < 10ms
- 不区分Young GC和Full GC
- 每次都是全堆回收

ZGC阶段：
1. Pause Mark Start（标记开始）- STW < 1ms
2. Concurrent Mark（并发标记）- 并发
3. Pause Mark End（标记结束）- STW < 1ms
4. Concurrent Prepare for Reloc（并发重定位准备）- 并发
5. Pause Relocate Start（重定位开始）- STW < 1ms
6. Concurrent Relocate（并发重定位）- 并发

GC日志示例：
[0.123s][info][gc] GC(0) Garbage Collection (Warmup)
[0.123s][info][gc,start] GC(0) Pause Mark Start
[0.124s][info][gc] GC(0) Pause Mark Start 0.234ms
[0.124s][info][gc] GC(0) Concurrent Mark
[2.456s][info][gc] GC(0) Concurrent Mark 2.332s
[2.456s][info][gc,start] GC(0) Pause Mark End
[2.457s][info][gc] GC(0) Pause Mark End 0.456ms
```

### 12.8 实战：如何分析GC触发原因

#### 方法1：查看GC日志

```bash
# 启用详细GC日志
java -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -XX:+PrintGCCause \
     -Xloggc:gc.log \
     -jar app.jar

# GC日志示例
2024-03-25T10:00:00.123+0800: [GC (Allocation Failure) [PSYoungGen: 2048K->512K(2560K)] 2048K->520K(9728K), 0.0012345 secs]
                                     ↑
                              触发原因：分配失败（Eden满）

2024-03-25T10:05:00.456+0800: [Full GC (Ergonomics) [PSYoungGen: 512K->0K(2560K)] [ParOldGen: 6144K->5120K(7168K)] 6656K->5120K(9728K), 0.123 secs]
                                        ↑
                              触发原因：人体工程学（自适应调节）
```

#### 方法2：使用JVM参数查看GC原因

```bash
# 常见GC原因
Allocation Failure        # 分配失败（Eden满）
Ergonomics               # 自适应调节
System.gc()              # 显式调用
Metadata GC Threshold    # 元空间阈值
CMS Initial Mark         # CMS初始标记
CMS Final Remark         # CMS重新标记
G1 Evacuation Pause      # G1疏散暂停
G1 Humongous Allocation  # G1大对象分配
Concurrent Mode Failure  # CMS并发失败
Promotion Failed         # 晋升失败
```

```bash
                      ↑
```

### 12.9 优化建议

```
优化策略：

1️⃣ 减少Young GC频率
   ├─ 增大新生代大小
   ├─ 调整Eden/Survivor比例
   └─ 优化对象分配

2️⃣ 避免Full GC
   ├─ 增大堆内存
   ├─ 调整老年代阈值
   ├─ 避免大对象
   └─ 预分配容量

3️⃣ G1优化
   ├─ 调整InitiatingHeapOccupancyPercent
   ├─ 增大Region大小
   └─ 放宽停顿时间目标

4️⃣ CMS优化
   ├─ 降低CMSInitiatingOccupancyFraction
   ├─ 定期整理内存碎片
   └─ 考虑升级到G1

5️⃣ 监控告警
   ├─ Full GC频率 > 1次/小时 → 告警
   ├─ GC停顿时间 > 1秒 → 告警
   └─ 老年代使用率 > 90% → 告警
```
