# 垃圾回收基础与算法

## 💡 大白话精华总结

**垃圾回收（GC）是什么？**
- 就是JVM自动帮你清理不用的对象，释放内存
- 就像家里的保姆，自动帮你扔垃圾，你不用管

**怎么判断对象是垃圾？**
- **引用计数法**：数有多少人在用这个对象，没人用就是垃圾（Python用这个）
  - 问题：循环引用无法回收（A引用B，B引用A）
- **可达性分析**：从"根"开始找，找不到的就是垃圾（Java用这个）
  - 根（GC Roots）：主线程、静态变量、JNI引用等

**4种垃圾回收算法：**

**1. 标记-清除（Mark-Sweep）**
```
标记垃圾 → 清除垃圾
优点：简单
缺点：内存碎片多（像拼图缺了很多块）
```

**2. 标记-复制（Mark-Copy）**
```
把活的对象复制到另一块内存 → 清空原内存
优点：无碎片，速度快
缺点：浪费一半内存
适用：新生代（对象死得快，活的少）
```

**3. 标记-整理（Mark-Compact）**
```
标记垃圾 → 把活的对象往一边挤 → 清除另一边
优点：无碎片，不浪费内存
缺点：慢（要移动对象）
适用：老年代（对象活得久）
```

**4. 分代收集（Generational）**
```
新生代：用复制算法（对象死得快）
老年代：用标记-整理（对象活得久）
这是现代JVM的主流方案！
```

**引用的4种类型（从强到弱）：**
2. **软引用**：内存不够时回收 - 适合缓存
3. **弱引用**：GC时就回收 - 适合临时对象
4. **虚引用**：随时可能被回收 - 用于跟踪对象回收

**一句话记住：**
> GC就是找出垃圾对象并回收内存，新生代用复制算法（快），老年代用标记-整理（省空间）！

## 一、垃圾回收概述

### 1.1 什么是垃圾回收

垃圾回收是自动内存管理机制，负责：
1. **识别**不再使用的对象
2. **回收**这些对象占用的内存
3. **整理**内存碎片

```
手动内存管理（C/C++）：
程序员负责 malloc/free、new/delete

自动内存管理（Java）：
JVM负责自动回收不再使用的对象
```

### 1.2 为什么需要垃圾回收

#### 没有GC的问题

```c
// C语言示例
void createObjects() {
    Object* obj = malloc(sizeof(Object));
    // 使用obj
    // 忘记调用free(obj) -> 内存泄漏
}

// 多次调用后，内存耗尽
for (int i = 0; i < 1000000; i++) {
    createObjects();
}
```

#### 有GC的好处

```java
// Java示例
void createObjects() {
    Object obj = new Object();
    // 使用obj
    // 方法结束后，obj自动被GC回收
}

// 不会内存泄漏
for (int i = 0; i < 1000000; i++) {
    createObjects();
}
```

### 1.3 GC的优势与代价

| 方面 | 优势 | 代价 |
|------|------|------|
| **开发效率** | 无需手动管理内存 | - |
| **安全性** | 避免内存泄漏、悬空指针 | - |
| **性能** | - | GC暂停（Stop The World） |
| **可控性** | - | 回收时机不可控 |

## 二、对象存活判定算法

### 2.1 引用计数法

#### 原理

为每个对象维护一个引用计数器，当引用数为0时回收。

```
对象A：引用计数 = 2
  ↑       ↑
  |       |
引用1   引用2

引用1断开 -> 引用计数 = 1
引用2断开 -> 引用计数 = 0 -> 可回收
```

#### 实现示例

```java
class RefCountObject {
    private int refCount = 0;  // 引用计数
    
    public void addRef() {
        refCount++;
    }
    
    public void release() {
        refCount--;
        if (refCount == 0) {
            // 回收对象
            System.out.println("对象被回收");
        }
    }
}

// 使用
RefCountObject obj = new RefCountObject();
obj.addRef();  // refCount = 1

RefCountObject obj2 = obj;
obj.addRef();  // refCount = 2

obj2 = null;
obj.release();  // refCount = 1

obj = null;
obj.release();  // refCount = 0，对象被回收
```

#### 优点

- ✅ 实现简单
- ✅ 回收及时（引用数为0立即回收）
- ✅ 不需要暂停程序

#### 缺点

- ❌ **无法解决循环引用问题**
- ❌ 每次引用变化都要更新计数器，开销大
- ❌ 需要额外的内存存储计数器

#### 循环引用问题

```java
class Node {
    Node next;
}

// 创建循环引用
Node a = new Node();
Node b = new Node();
a.next = b;  // a引用b
b.next = a;  // b引用a

// 断开外部引用
a = null;
b = null;

// 问题：a和b互相引用，引用计数都不为0
// 但实际上已经无法访问，应该被回收
```

```
循环引用示意图：

外部引用断开前：
main -> a <-> b
       (1)   (1)

外部引用断开后：
main   a <-> b
      (1)   (1)

引用计数都是1，但都无法访问
```

### 2.2 可达性分析算法

#### 原理

从**GC Roots**出发，通过引用链搜索，能够到达的对象是存活的，不能到达的对象是可回收的。

```
GC Roots
   ↓
  对象A → 对象B → 对象C
   ↓
  对象D

对象E（无引用链）

结果：
- A、B、C、D 存活
- E 可回收
```

#### GC Roots包括

| GC Root类型 | 说明 | 示例 |
|------------|------|------|
| **虚拟机栈中的引用** | 方法的局部变量 | `Object obj = new Object();` |
| **方法区中的静态变量** | 类的静态字段 | `static Object obj;` |
| **方法区中的常量** | final常量 | `static final Object obj;` |
| **本地方法栈中的引用** | JNI引用 | Native方法中的对象 |
| **活跃线程** | 正在运行的线程 | `Thread.currentThread()` |

#### 示例

```java
public class GCRootsDemo {
    
    // GC Root 1: 方法区中的静态变量
    private static Object staticObj = new Object();
    
    // GC Root 2: 方法区中的常量
    private static final Object CONSTANT_OBJ = new Object();
    
    public void method() {
        // GC Root 3: 虚拟机栈中的引用
        Object localObj = new Object();
        
        // 这个对象可以通过localObj到达，所以存活
        Object reachableObj = new Object();
        localObj = reachableObj;
        
        // 这个对象无法从GC Roots到达，可回收
        Object unreachableObj = new Object();
        unreachableObj = null;
    }
}
```

#### 可达性分析流程

```
1. 标记阶段
   ├─ 从GC Roots开始遍历
   ├─ 标记所有可达对象
   └─ 未标记的对象即为垃圾

2. 清除阶段
   └─ 回收未标记的对象
```

#### 优点

- ✅ 解决了循环引用问题
- ✅ 准确判断对象是否存活
- ✅ Java、C#等主流语言都采用

#### 缺点

- ❌ 需要暂停程序（Stop The World）
- ❌ 遍历对象图有性能开销

## 三、垃圾回收算法

### 3.1 标记-清除算法（Mark-Sweep）

#### 原理

1. **标记阶段**：标记所有需要回收的对象
2. **清除阶段**：回收被标记的对象

```
初始状态：
[对象A][对象B][对象C][对象D][对象E]

标记阶段（B、D可回收）：
[对象A][对象B✗][对象C][对象D✗][对象E]

清除阶段：
[对象A][空闲][对象C][空闲][对象E]
```

#### 实现示例

```java
public class MarkSweepGC {
    
    private List<Object> heap = new ArrayList<>();
    private Set<Object> marked = new HashSet<>();
    
    // 标记阶段
    public void mark(Object root) {
        if (marked.contains(root)) {
            return;
        }
        
        marked.add(root);
        
        // 递归标记所有可达对象
        for (Object ref : getReferences(root)) {
            mark(ref);
        }
    }
    
    // 清除阶段
    public void sweep() {
        Iterator<Object> it = heap.iterator();
        while (it.hasNext()) {
            Object obj = it.next();
            if (!marked.contains(obj)) {
                // 未标记的对象，回收
                it.remove();
                System.out.println("回收对象: " + obj);
            }
        }
        
        // 清空标记集合
        marked.clear();
    }
    
    // 执行GC
    public void gc(Object root) {
        System.out.println("开始GC...");
        mark(root);
        sweep();
        System.out.println("GC完成");
    }
    
    private List<Object> getReferences(Object obj) {
        // 获取对象的所有引用
        return new ArrayList<>();
    }
}
```

#### 优点

- ✅ 实现简单
- ✅ 不需要移动对象

#### 缺点

- ❌ **产生内存碎片**
- ❌ 标记和清除效率都不高
- ❌ 需要维护空闲列表

#### 内存碎片问题

```
多次GC后的内存状态：
[对象][空闲][对象][空闲][空闲][对象][空闲]

问题：
- 虽然总空闲空间足够
- 但没有连续的大块空间
- 无法分配大对象
```

### 3.2 标记-复制算法（Mark-Copy）

#### 原理

1. 将内存分为两块：From区和To区
2. 对象分配在From区
3. GC时，将From区存活对象复制到To区
4. 清空From区
5. 交换From区和To区

```
初始状态（From区）：
[对象A][对象B][对象C][对象D][对象E]
        (死)           (死)

复制存活对象到To区：
From: [对象A][对象B][对象C][对象D][对象E]
To:   [对象A][对象C][对象E]

清空From区，交换角色：
From: [对象A][对象C][对象E]
To:   [空闲空间]
```

#### 实现示例

```java
public class CopyingGC {
    
    private Object[] fromSpace;
    private Object[] toSpace;
    private int fromIndex = 0;
    private int toIndex = 0;
    
    public CopyingGC(int size) {
        fromSpace = new Object[size];
        toSpace = new Object[size];
    }
    
    // 分配对象
    public int allocate(Object obj) {
        if (fromIndex >= fromSpace.length) {
            gc();
        }
        fromSpace[fromIndex] = obj;
        return fromIndex++;
    }
    
    // 执行GC
    public void gc() {
        System.out.println("开始复制GC...");
        
        toIndex = 0;
        
        // 复制存活对象
        for (int i = 0; i < fromIndex; i++) {
            Object obj = fromSpace[i];
            if (isAlive(obj)) {
                toSpace[toIndex++] = obj;
                System.out.println("复制对象: " + obj);
            }
        }
        
        // 交换空间
        Object[] temp = fromSpace;
        fromSpace = toSpace;
        toSpace = temp;
        fromIndex = toIndex;
        
        // 清空To空间
        Arrays.fill(toSpace, null);
        
        System.out.println("GC完成，存活对象: " + toIndex);
    }
    
    private boolean isAlive(Object obj) {
        // 判断对象是否存活
        return obj != null;
    }
}
```

#### 优点

- ✅ **没有内存碎片**
- ✅ 分配内存简单高效（指针碰撞）
- ✅ 适合存活对象少的场景

#### 缺点

- ❌ **浪费50%的内存空间**
- ❌ 存活对象多时，复制开销大
- ❌ 需要额外空间担保

#### 改进：不对等分配

```
Eden区（80%） + Survivor0（10%） + Survivor1（10%）

正常情况：
- 对象分配在Eden区
- GC时，Eden和Survivor0的存活对象复制到Survivor1
- 清空Eden和Survivor0

下次GC：
- Eden和Survivor1的存活对象复制到Survivor0
- 清空Eden和Survivor1
```

### 3.3 标记-整理算法（Mark-Compact）

#### 原理

1. **标记阶段**：标记所有存活对象
2. **整理阶段**：将存活对象移动到内存一端
3. **清除阶段**：清理边界外的内存

```
初始状态：
[对象A][对象B][对象C][对象D][对象E]
        (死)           (死)

标记存活对象：
[对象A✓][对象B✗][对象C✓][对象D✗][对象E✓]

整理（移动存活对象）：
[对象A][对象C][对象E][空闲空间]
```

#### 实现示例

```java
public class CompactingGC {
    
    private Object[] heap;
    private boolean[] marked;
    private int nextFree = 0;
    
    public CompactingGC(int size) {
        heap = new Object[size];
        marked = new boolean[size];
    }
    
    // 标记阶段
    public void mark(Set<Integer> roots) {
        Arrays.fill(marked, false);
        
        for (int root : roots) {
            markObject(root);
        }
    }
    
    private void markObject(int index) {
        if (index < 0 || index >= heap.length || marked[index]) {
            return;
        }
        
        marked[index] = true;
        
        // 递归标记引用的对象
        for (int ref : getReferences(index)) {
            markObject(ref);
        }
    }
    
    // 整理阶段
    public void compact() {
        int writeIndex = 0;
        
        // 移动存活对象
        for (int i = 0; i < heap.length; i++) {
            if (marked[i]) {
                if (i != writeIndex) {
                    heap[writeIndex] = heap[i];
                    heap[i] = null;
                    System.out.println("移动对象: " + heap[writeIndex] + 
                                     " 从 " + i + " 到 " + writeIndex);
                }
                writeIndex++;
            }
        }
        
        // 清理剩余空间
        for (int i = writeIndex; i < heap.length; i++) {
            heap[i] = null;
        }
        
        nextFree = writeIndex;
        System.out.println("整理完成，下一个空闲位置: " + nextFree);
    }
    
    // 执行GC
    public void gc(Set<Integer> roots) {
        System.out.println("开始标记-整理GC...");
        mark(roots);
        compact();
        System.out.println("GC完成");
    }
    
    private List<Integer> getReferences(int index) {
        // 获取对象的引用
        return new ArrayList<>();
    }
}
```

#### 优点

- ✅ **没有内存碎片**
- ✅ **不浪费内存空间**
- ✅ 适合存活对象多的场景

#### 缺点

- ❌ **移动对象开销大**
- ❌ 需要更新所有引用
- ❌ 需要暂停时间更长

### 3.4 三种算法对比

| 算法 | 内存碎片 | 内存利用率 | 移动对象 | 适用场景 |
|------|---------|-----------|---------|---------|
| **标记-清除** | 有 | 高 | 否 | 对象存活率高 |
| **标记-复制** | 无 | 低（50%） | 是 | 对象存活率低 |
| **标记-整理** | 无 | 高 | 是 | 对象存活率高 |

#### 选择建议

```
新生代（对象存活率低）：
→ 标记-复制算法

老年代（对象存活率高）：
→ 标记-清除 或 标记-整理
```

## 四、引用类型

### 4.1 引用类型概述

Java提供了4种引用类型，用于更灵活地控制对象的生命周期。

```
强引用 > 软引用 > 弱引用 > 虚引用
（引用强度从强到弱）
```

### 4.2 强引用（Strong Reference）

#### 定义

最常见的引用类型，只要强引用存在，对象就不会被回收。

```java
// 强引用
Object obj = new Object();

// 只要obj还指向对象，对象就不会被GC
// 即使内存不足，也不会回收，而是抛出OOM
```

#### 特点

- ✅ 最常用的引用类型
- ✅ 只要强引用存在，对象永远不会被回收
- ❌ 可能导致内存泄漏

#### 示例

```java
public class StrongReferenceDemo {
    public static void main(String[] args) {
        // 创建强引用
        Object obj = new Object();
        System.out.println("对象: " + obj);
        
        // 即使内存不足，也不会回收obj
        // 只有当obj = null时，对象才可能被回收
        
        // 断开强引用
        obj = null;
        
        // 此时对象可以被GC回收
        System.gc();
    }
}
```

### 4.3 软引用（Soft Reference）

#### 定义

内存充足时保留，内存不足时回收。

```java
import java.lang.ref.SoftReference;

// 创建软引用
Object obj = new Object();
SoftReference<Object> softRef = new SoftReference<>(obj);
obj = null;  // 断开强引用

// 获取对象
Object retrieved = softRef.get();
if (retrieved != null) {
    System.out.println("对象还在");
} else {
    System.out.println("对象已被回收");
}
```

#### 特点

- ✅ 内存充足时不回收
- ✅ 内存不足时回收
- ✅ 适合实现缓存

#### 应用场景：图片缓存

```java
public class ImageCache {
    
    private Map<String, SoftReference<Image>> cache = new HashMap<>();
    
    public Image getImage(String path) {
        // 从缓存获取
        SoftReference<Image> ref = cache.get(path);
        if (ref != null) {
            Image image = ref.get();
            if (image != null) {
                System.out.println("从缓存获取图片: " + path);
                return image;
            }
        }
        
        // 缓存中没有，加载图片
        Image image = loadImage(path);
        cache.put(path, new SoftReference<>(image));
        System.out.println("加载图片: " + path);
        return image;
    }
    
    private Image loadImage(String path) {
        // 从磁盘加载图片
        return new Image(path);
    }
    
    static class Image {
        private String path;
        private byte[] data = new byte[1024 * 1024];  // 1MB
        
        public Image(String path) {
            this.path = path;
        }
    }
    
    public static void main(String[] args) {
        ImageCache cache = new ImageCache();
        
        // 加载图片
        cache.getImage("image1.jpg");
        cache.getImage("image2.jpg");
        
        // 再次获取（从缓存）
        cache.getImage("image1.jpg");
        
        // 模拟内存不足
        // 软引用的图片会被回收
    }
}
```

### 4.4 弱引用（Weak Reference）

#### 定义

只要发生GC，就会被回收，无论内存是否充足。

```java
import java.lang.ref.WeakReference;

// 创建弱引用
Object obj = new Object();
WeakReference<Object> weakRef = new WeakReference<>(obj);
obj = null;  // 断开强引用

// 此时对象只有弱引用
System.gc();  // 触发GC

// 对象被回收
Object retrieved = weakRef.get();
System.out.println(retrieved);  // null
```

#### 特点

- ✅ 不影响对象的回收
- ✅ 适合实现规范化映射
- ❌ 生命周期很短

#### 应用场景：ThreadLocal

```java
public class ThreadLocal<T> {
    
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;
        
        Entry(ThreadLocal<?> k, Object v) {
            super(k);  // ThreadLocal作为弱引用
            value = v;
        }
    }
    
    // ThreadLocal作为key，使用弱引用
    // 当ThreadLocal没有强引用时，会被GC回收
    // 避免内存泄漏
}
```

#### WeakHashMap示例

```java
import java.util.WeakHashMap;

public class WeakHashMapDemo {
    public static void main(String[] args) {
        WeakHashMap<Object, String> map = new WeakHashMap<>();
        
        Object key1 = new Object();
        Object key2 = new Object();
        
        map.put(key1, "value1");
        map.put(key2, "value2");
        
        System.out.println("GC前: " + map.size());  // 2
        
        // 断开key1的强引用
        key1 = null;
        
        // 触发GC
        System.gc();
        
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        
        System.out.println("GC后: " + map.size());  // 1
        // key1被回收，对应的entry也被删除
    }
}
```

### 4.5 虚引用（Phantom Reference）

#### 定义

最弱的引用类型，无法通过虚引用获取对象，主要用于跟踪对象被回收的状态。

```java
import java.lang.ref.PhantomReference;
import java.lang.ref.ReferenceQueue;

// 创建虚引用
Object obj = new Object();
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);

// 无法通过虚引用获取对象
Object retrieved = phantomRef.get();
System.out.println(retrieved);  // 总是null
```

#### 特点

- ✅ 跟踪对象回收状态
- ✅ 可以在对象回收前做清理工作
- ❌ 无法通过虚引用获取对象

#### 应用场景：资源清理

```java
import java.lang.ref.*;
import java.util.HashMap;
import java.util.Map;

public class ResourceCleanup {
    
    private static ReferenceQueue<Object> queue = new ReferenceQueue<>();
    private static Map<PhantomReference<Object>, String> resources = new HashMap<>();
    
    public static void track(Object obj, String resourceName) {
        PhantomReference<Object> ref = new PhantomReference<>(obj, queue);
        resources.put(ref, resourceName);
    }
    
    public static void cleanup() {
        Reference<?> ref;
        while ((ref = queue.poll()) != null) {
            String resourceName = resources.remove(ref);
            System.out.println("清理资源: " + resourceName);
            // 执行清理操作
        }
    }
    
    public static void main(String[] args) throws InterruptedException {
        // 创建对象并跟踪
        Object obj1 = new Object();
        track(obj1, "Resource1");
        
        Object obj2 = new Object();
        track(obj2, "Resource2");
        
        // 断开引用
        obj1 = null;
        obj2 = null;
        
        // 触发GC
        System.gc();
        Thread.sleep(100);
        
        // 清理资源
        cleanup();
    }
}
```

### 4.6 引用类型对比

| 引用类型 | 回收时机 | 用途 | get()返回值 |
|---------|---------|------|------------|
| **强引用** | 永不回收 | 普通对象引用 | 对象本身 |
| **软引用** | 内存不足时 | 缓存 | 对象或null |
| **弱引用** | 下次GC时 | 规范化映射 | 对象或null |
| **虚引用** | 下次GC时 | 跟踪回收 | 总是null |

## 五、GC相关概念

### 5.1 Stop The World（STW）

#### 定义

STW 是指 JVM 在执行 GC 时，**强制暂停所有应用（Mutator）线程**，直到 GC 完成后再统一恢复。无论使用哪种 GC 收集器，都无法完全消除 STW，只能尽量缩短其持续时间。

```
时间轴：

应用线程1：  ████████ | 暂停 | ████████████
应用线程2：  ████████ | 暂停 | ████████████
应用线程3：  ████████ | 暂停 | ████████████
GC 线程：            |█████|
                     ↑     ↑
                  STW开始  STW结束（GC完成）
```

---

#### 为什么必须 STW

**核心原因：对象图在 GC 标记过程中不能被修改。**

```
问题场景（无 STW 的并发标记）：

时刻1：GC 开始标记，对象 A 已标记为存活，A.ref → B
时刻2：应用线程执行 A.ref = null；C.ref = B
时刻3：GC 继续，B 通过 A 不可达，误判为垃圾 → B 被错误回收！

这就是"漏标"问题（floating garbage 的极端情况）
```

```
漏标的两个必要条件（三色标记理论）：
  条件1：黑色对象（已扫描完）新增了对白色对象（未扫描）的引用
  条件2：白色对象原来通过灰色对象可达的引用被删除

两个条件同时满足 → 白色对象被漏标 → 存活对象被错误回收（严重Bug）
```

**STW 通过暂停所有应用线程，确保标记期间对象图静止不变，从根本上避免漏标。**

---

#### STW 的触发时机

不同 GC 阶段的 STW 情况：

| GC 阶段 | Serial/Parallel | CMS | G1 | ZGC/Shenandoah |
|--------|----------------|-----|----|---------------|
| **初始标记** | STW | STW（短） | STW（短） | STW（极短） |
| **并发标记** | STW | 并发（无STW） | 并发 | 并发 |
| **重新标记** | STW | STW（短） | STW（短） | STW（极短） |
| **对象复制/整理** | STW | STW（Full GC时） | STW（混合GC） | 并发 |

---

#### STW 的测量与监控

```bash
# 开启 GC 日志，观察 STW 时间
java -Xlog:gc*:file=gc.log:time,uptime,level,tags \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     MyApp

# GC 日志中的 STW 时间示例（G1）：
# [GC pause (G1 Evacuation Pause) (young)
#   Pause Time: 12.3 ms    ← STW 持续时间
# ]
```

```java
// 代码中监控 STW（通过 GC 通知机制）

    ManagementFactory.getGarbageCollectorMXBeans();

for (GarbageCollectorMXBean gcBean : gcBeans) {
    System.out.printf("GC: %-20s 次数: %d 总耗时: %d ms%n",
        gcBean.getName(),

        gcBean.getCollectionTime());  // 包含 STW 时间

```

---

#### 减少 STW 的核心手段

```
1. 并发标记（Concurrent Marking）
   → CMS、G1、ZGC 将标记阶段与应用线程并行执行
   → 只在初始标记和重新标记时短暂 STW

2. 增量式 GC（Incremental GC）
   → 将 GC 工作分成小批次，穿插在应用执行中
   → 每次 STW 时间极短

3. 减小堆分区（Region-based Heap）
   → G1 将堆分成 2048 个 Region
   → 每次只回收部分 Region，缩短单次 STW

4. 并发整理（Concurrent Compaction）
   → ZGC / Shenandoah 实现并发对象移动
   → 整理阶段几乎无 STW

5. 调整 GC 参数
   -XX:MaxGCPauseMillis=200   # 设置期望最大暂停时间（G1目标值）
   -XX:GCPauseIntervalMillis=1000  # GC 间隔目标
```

---

#### STW 耗时过长的排查

```
STW > 500ms 的常见原因：

1. 堆内存过大
   → 老年代几十 GB，Full GC 时标记耗时长
   → 解决：使用 G1/ZGC，或控制堆大小

2. 系统 swap 导致 GC 读取对象时 page fault
   → 解决：禁用 swap，或确保 JVM 堆内存充足

3. JNI 临界区（Critical Section）阻塞 STW
   → JNI 线程在 Critical Section 内无法响应 STW 请求
   → 解决：减少 JNI 临界区时长

4. 大量对象晋升老年代（Promotion Failure）
   → 触发 Full GC，STW 时间暴增
   → 解决：调整晋升阈值和 Survivor 区大小
```

### 5.2 安全点（Safepoint）

#### 定义

安全点是程序执行流中**特定的代码位置**，JVM 只有在所有线程都到达安全点之后，才能开始执行 STW 操作（如 GC、偏向锁撤销、类卸载等）。在安全点处，JVM 可以安全地枚举所有 GC Roots，因为此时线程的寄存器状态、栈帧信息均已知且稳定。

---

#### 安全点的设置位置

HotSpot JIT 编译器在以下位置插入安全点检查：

| 位置 | 说明 |
|------|------|
| **方法调用处** | 每次方法调用前/后均可作为安全点 |
| **循环跳转处** | 循环回跳指令（counted loop back-edge） |
| **异常跳转处** | 抛出或捕获异常时 |

```
注意：解释器执行的每条字节码之间天然是安全点；
     JIT 编译后的本地代码，安全点由编译器在上述位置插入检查代码。
```

```java
public void method() {
    int sum = 0;
    for (int i = 0; i < 1000000; i++) {  // ← 循环跳转：安全点
        sum += i;
        if (i % 100 == 0) {
            doSomething();  // ← 方法调用：安全点
        }
    }
}
```

**不会插入安全点的地方**：

```java
// 计数循环（counted loop）在 JIT 优化后可能不插入安全点
// 导致线程长时间无法响应 STW 请求（TTSP 过长）
for (int i = 0; i < Integer.MAX_VALUE; i++) {
    // 纯计算，无方法调用、无循环跳转
    sum += i;  // JIT 可能将此循环整体优化，不插入安全点
}
// 解决：-XX:+UseCountedLoopSafepoints（JDK 8u92+ 支持）
```

---

#### OopMap —— 安全点的核心数据结构

GC 需要知道栈帧和寄存器中哪些位置存放的是对象引用（OOP = Ordinary Object Pointer）。**OopMap** 就是 JIT 在安全点处生成的映射表，记录了这些信息。

```
OopMap 示例（某方法调用安全点处）：

栈帧布局：
  [本地变量0] → Object  ← OopMap 标记为引用
  [本地变量1] → int     ← OopMap 标记为非引用
  [本地变量2] → String  ← OopMap 标记为引用
  [寄存器 rax] → Object ← OopMap 标记为引用

GC 根据 OopMap 精确找到所有引用 → 逐一更新/标记
```

```
没有 OopMap 会怎样？
→ GC 只能保守扫描（Conservative GC）
→ 把所有可能是引用的值都当作引用
→ 大量误报，无法移动对象（因为不确定某个值是引用还是整数）
→ Java 采用精确 GC（Exact GC）依赖 OopMap
```

---

#### 线程如何响应安全点请求

JVM 让线程"主动"到达安全点，有两种实现方式：

**方式1：轮询标志位（HotSpot 实际采用）**

```
JIT 编译后的安全点检查伪代码：

loop_back:
    ; ... 循环体 ...
    test [safepoint_page], eax   ; 读取安全点标志页
    jnz  safepoint_handler       ; 如果页被标记为不可访问，触发信号
    jmp  loop_back

触发原理：
- 正常运行：safepoint_page 可读，test 指令正常执行，继续
- GC 请求：JVM 将 safepoint_page 设置为不可访问（mprotect）
           test 指令触发 SIGSEGV → JVM 信号处理器捕获 → 线程挂起
```

**方式2：抢先式中断（Preemptive Suspension，已基本废弃）**

```
强制中断所有线程 → 检查是否在安全点 → 不在则恢复继续执行
→ 开销大，实现复杂，HotSpot 不采用
```

---

#### TTSP（Time To Safepoint）问题

TTSP 是从 JVM 发出安全点请求到**所有线程都到达安全点**的时间。

```
问题场景：

线程 A：正在执行一个没有安全点的长循环（JIT 优化后的计数循环）
GC 请求安全点 → 等待线程 A → 线程 A 执行完整个循环才到达安全点
→ STW 延迟高（即使 GC 本身很快，等待阶段就花了几百毫秒）
```

```bash
# 诊断 TTSP 问题
-XX:+PrintSafepointStatistics          # 打印安全点统计
-XX:PrintSafepointStatisticsCount=1    # 每次都打印

# 输出示例：
# Safepoint "GenCollectForAllocation"
#   Total: 350.4 ms
#   Spin:  320.1 ms   ← 等待线程到达安全点耗时（TTSP）
#   Block: 30.3 ms    ← GC 实际执行时间
#
# Spin 时间远大于 Block 时间 → 典型 TTSP 问题

# 解决：
-XX:+UseCountedLoopSafepoints  # 在计数循环中也插入安全点
```

---

#### 安全点与安全区域的协作总结

```
GC 触发时各类线程的处理方式：

Running 线程   → 执行到下一个安全点后主动挂起
Sleep 线程     → 已在安全区域，直接认为安全，GC 无需等待
Blocked 线程   → 同上，已在安全区域
JNI 线程       → 执行完当前 JNI 调用后在返回时挂起
               （JNI 临界区例外：必须等待临界区结束）
```

### 5.3 安全区域（Safe Region）

#### 定义

安全区域是对安全点机制的补充。**安全点要求线程主动执行到某个位置后挂起**，但处于 `Sleep`、`Blocked`、`Waiting` 等状态的线程根本无法执行代码，自然也无法"走到"安全点。安全区域解决的就是这个问题：

> 安全区域是代码中的某一段范围，在这段范围内，**对象引用关系不会发生变化**，因此 GC 可以在任意时刻安全地对其中的线程执行标记操作，无需等待这些线程主动到达安全点。

---

#### 安全区域的工作流程

```
线程进入安全区域的完整协议：

┌─────────────────────────────────────────────────────┐
│                   线程执行流程                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 线程即将进入 Sleep/Blocked/Wait：               │
│     → 主动标记自己"已进入安全区域"                   │
│       （设置线程状态标志位）                          │
│                                                     │
│  2. GC 发出安全点请求：                              │
│     → 检查所有线程状态                               │
│     → 处于安全区域的线程：直接跳过，无需等待          │
│     → Running 线程：等待其到达安全点                  │
│                                                     │
│  3. GC 执行完毕：                                   │
│     → 广播"GC 完成"信号                             │
│                                                     │
│  4. 线程准备离开安全区域（恢复执行）：                │
│     → 检查是否有未完成的 GC 操作                     │
│     → 若 GC 仍在进行 → 线程阻塞等待 GC 完成          │
│     → 若 GC 已完成  → 清除标志，正常继续执行          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

---

#### 代码示例：安全区域的进入与离开

```java
// 模拟线程进入和离开安全区域的逻辑（概念示意）
public void run() {
    // --- 正常执行阶段（非安全区域）---
    doWork();

    // 进入安全区域（即将 sleep）
    JVM.markThreadInSafeRegion(this);    // 标记：我现在引用不会变化
    Thread.sleep(1000);                  // sleep 期间 GC 可随时执行
    // 准备离开安全区域
    JVM.checkAndWaitForGC();             // 如果 GC 还在进行，等待完成
    JVM.clearSafeRegionMark(this);       // 清除标记

    // --- 继续正常执行 ---
    doMoreWork();
}
```

```
关键点：线程离开安全区域时，必须检查 GC 状态

场景1：GC 已完成
  线程正常恢复执行，对象引用已被 GC 更新（如对象被移动），
  JVM 已修正所有引用 → 线程直接继续

场景2：GC 仍在进行（线程苏醒时 GC 还未结束）
  线程必须等待 GC 完成后才能继续
  原因：GC 可能正在移动对象或更新引用，此时继续执行会读到无效引用
```

---

#### 安全区域 vs 安全点

| 维度 | 安全点（Safepoint） | 安全区域（Safe Region） |
|------|-------------------|----------------------|
| **适用线程** | Running（执行中）线程 | Sleep/Blocked/Waiting 线程 |
| **触发方式** | 线程执行到特定位置时主动检查 | 线程进入阻塞状态前主动声明 |
| **GC 等待行为** | 等待所有 Running 线程到达安全点 | 对安全区域内线程无需等待 |
| **离开时机** | GC 完成后自动恢复 | 线程主动检查 GC 是否完成后再离开 |
| **典型场景** | 正常业务代码执行期间 | `Thread.sleep()`、`Object.wait()`、I/O 阻塞 |

---

#### 实际影响：为什么长时间 sleep 的线程不会阻塞 GC

```
误解：线程 sleep 很久，GC 会一直等它？

真相：
  线程进入 sleep 前已标记自己"在安全区域"
  GC 不需要等待该线程 → 直接开始执行
  线程 sleep 结束后，检查 GC 状态，再决定是否等待

结论：
  大量 sleep/blocked 线程的存在 ✅ 不会影响 STW 时间
  大量 running 线程的存在 ❌ 需要等待所有人到达安全点（影响 TTSP）
```

### 5.4 记忆集（Remembered Set）

#### 定义

记忆集（Remembered Set，RSet）是一种**辅助数据结构**，用于记录**从堆的其他区域指向当前被收集区域的引用**，从而在局部 GC（如 Young GC）时，无需扫描整个堆，只扫描记忆集即可找到所有跨区域引用，大幅减少 GC 扫描范围。

---

#### 为什么需要记忆集——跨代引用问题

```
分代堆结构：
┌──────────────────────┬──────────────────────────────┐
│       新生代          │           老年代              │
│  Eden  S0  S1        │                              │
└──────────────────────┴──────────────────────────────┘

Young GC 只回收新生代，GC Roots 除了栈/方法区之外，
还必须包含所有"老年代 → 新生代"的引用：

                老年代对象 A
                    │
                    │ 持有引用
                    ↓
                新生代对象 B  ← 如果 B 只被 A 引用，没有 RSet 的话
                              B 会被误判为垃圾，导致 A 持有悬空指针！
```

```
没有记忆集的方案（全堆扫描）：
  Young GC 开始 → 扫描整个老年代寻找跨代引用
  老年代几百 MB ~ 几 GB → 扫描极慢 → Young GC 失去"快速"的意义

有记忆集的方案：
  写屏障维护记忆集，记录"老年代区域 X 持有新生代引用"
  Young GC 只扫描记忆集中的区域 → 扫描范围大幅缩小
```

---

#### 记忆集的三种精度级别

记忆集只需要找到"哪些区域持有跨代引用"，精度越高越准确但开销越大：

| 精度 | 记录粒度 | 内存开销 | 扫描精度 | 实际使用 |
|------|---------|---------|---------|---------|
| **字长精度（Word）** | 每个机器字（8字节） | 极高 | 最精确 | 基本不用 |
| **对象精度（Object）** | 每个对象 | 高 | 精确 | 较少使用 |
| **卡精度（Card）** | 每 512 字节内存块 | 极低（堆的1/512） | 存在冗余扫描 | **HotSpot 主流方案（卡表）** |

---

#### 记忆集的记录方向

记忆集可以从两个方向记录跨区域引用：

```
方向1：Point-out（指出方向）—— 记录"我引用了谁"
  → 老年代区域 X 的记忆集 = { 新生代中被 X 内对象引用的所有对象 }
  → Young GC 时，需要遍历所有老年代区域的记忆集
  → 查找成本高，较少采用

方向2：Point-in（指入方向）—— 记录"谁引用了我"（HotSpot 采用）
  → 新生代的记忆集 = { 老年代中持有新生代引用的所有区域/卡片 }
  → Young GC 时，直接使用新生代的记忆集作为额外 GC Root 来源
  → 查找成本低，HotSpot 卡表即采用此方向
```

---

#### HotSpot 中记忆集的实现演进

**Serial / Parallel / CMS GC：使用卡表（Card Table）**

```
整个堆只有一个卡表（全局数组）：
  cardTable[i] = DIRTY  表示第 i 个卡片（512字节）内存在跨代引用

适用场景：两代堆（新生代 + 老年代），只需记录老年代→新生代
局限性：不适合 G1 的多 Region 场景
```

**G1 GC：每个 Region 独立维护 RSet**

```
G1 堆结构（每个格子是一个 Region，约 1~32MB）：
┌────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ O  │ E  │ H  │
└────┴────┴────┴────┴────┴────┘
  ↑
  每个 Region 有自己的 RSet，记录"哪些其他 Region 的卡片引用了我"

Region A 的 RSet（Point-in 方向）：
  { Region B 的卡片 3, Region C 的卡片 7, Region D 的卡片 1 }

GC 回收 Region A 时：
  只扫描 RSet 中记录的 Region B/C/D 的对应卡片
  而非扫描整个堆
```

```
G1 RSet 内部结构（三级精度，随引用数量自动升级）：

引用数少  → 稀疏表（Sparse Table）：哈希表存储 Region+Card 索引，内存小
引用数中  → 细粒度表（Fine-grained）：位图表示 Card 级别
引用数多  → 粗粒度表（Coarse-grained）：位图表示 Region 级别（精度下降）
```

#### 记忆集的维护时机与写屏障

记忆集由**写屏障（Write Barrier）** 在对象引用赋值时同步维护：

```java
// 应用代码：老年代对象引用新生代对象
oldGenObj.field = youngGenObj;

// JIT 插入的写后屏障（Post-Write Barrier）伪代码：
void postWriteBarrier(Object src, Object dst) {
    if (isOldGen(src) && isYoungGen(dst)) {
        // 将 src 所在的卡片标记为脏（通知 GC 该区域有跨代引用）
        markCardDirty(src);
        // G1 还会更新 dst 所在 Region 的 RSet：
        // dst.region.rset.add(cardOf(src));
    }
}
```

**写屏障的性能影响**：

```
每次引用赋值都触发写屏障 → 带来额外开销

HotSpot 优化手段：
1. 条件写入（-XX:+UseCondCardMark）
   if (cardTable[i] != DIRTY) cardTable[i] = DIRTY;
   避免重复标记同一卡片，减少缓存失效

2. G1 的 Refinement 线程
   写屏障只将脏卡片加入队列（极轻量）
   由后台 Refinement 线程异步处理 RSet 更新
   → 将 RSet 维护开销从应用线程转移到后台线程
```

#### 记忆集的 GC 使用流程

```
Young GC 完整流程（含记忆集）：

初始 GC Roots：
  ├─ 虚拟机栈中的引用
  ├─ 方法区静态变量
  └─ 记忆集/卡表中标记为脏的区域  ← 新增的跨代引用根

扫描脏卡片/RSet：
  for each dirty_card in remembered_set:
      scan(dirty_card)    // 扫描 512 字节，找出引用新生代的对象
      add_to_gc_roots()   // 将这些老年代对象加入 GC Root 集合
      dirty_card.clean()  // 清除脏标记

标记与复制：
  从 GC Roots 标记所有新生代可达对象
  将存活对象复制到 Survivor 或晋升老年代

更新记忆集：
  晋升到老年代的对象若持有新生代引用
  → 重新标记对应卡片为脏，确保下次 GC 不遗漏
```

#### 各 GC 收集器的记忆集对比

| GC 收集器 | 记忆集实现 | 精度 | 维护方式 |
|----------|-----------|------|---------|
| Serial GC | 全局卡表 | 卡片级（512B） | 同步写屏障 |
| Parallel GC | 全局卡表 | 卡片级（512B） | 同步写屏障 |
| CMS GC | 全局卡表 | 卡片级（512B） | 同步写屏障 |
| G1 GC | 每 Region 独立 RSet + 卡表 | 卡片级，自适应三级 | 异步 Refinement 线程 |

| Shenandoah | 连接矩阵（Connection Matrix） | Region 级 | 写屏障 |

### 5.5 卡表（Card Table）

#### 定义

卡表是**记忆集（Remembered Set）的一种具体实现**，由 HotSpot JVM 采用。它将整个 Java 堆划分为一组固定大小的**卡片（Card）**，每张卡片对应卡表中的一个字节标志位，用于快速定位哪些堆区域存在**跨代引用**，从而避免 Young GC 时扫描整个老年代。

---

#### 为什么需要卡表？——跨代引用问题

```
问题场景：Young GC（只回收新生代）

新生代 GC 的 GC Roots 除了栈/方法区，还包括：
  老年代对象 → 新生代对象   ← 这条跨代引用！

如果不处理：
  必须扫描整个老年代来找到所有指向新生代的引用
  老年代通常很大（几百 MB ~ 几 GB），扫描极慢

解决方案：卡表
  只记录"哪些老年代区域存在跨代引用"
  Young GC 时只扫描这些脏区域，而非整个老年代
```

---

#### 卡表结构与卡片大小

HotSpot 中每张卡片覆盖 **512 字节**的堆内存，对应卡表中的 **1 个字节**。

```
堆内存（示例，总大小 2KB）：
┌──────────┬──────────┬──────────┬──────────┐
│  Card 0  │  Card 1  │  Card 2  │  Card 3  │
│ 512 字节 │ 512 字节 │ 512 字节 │ 512 字节 │
└──────────┴──────────┴──────────┴──────────┘
     ↕           ↕           ↕           ↕
卡表（每项 1 字节）：
┌────┬────┬────┬────┐
│ 0  │ 1  │ 0  │ 1  │
└────┴────┴────┴────┘
  干净  脏    干净  脏
       ↑           ↑
   Card1 内有      Card3 内有
   老年代对象      老年代对象
   指向新生代      指向新生代
```

**卡表项的值**：
- `0`（`CARD_TABLE_ENTRY_CLEAN`）：该卡片内无跨代引用
- `1`（非零，即"脏"，`CARD_TABLE_ENTRY_DIRTY`）：该卡片内存在跨代引用，GC 时需扫描

**内存开销计算**：
```
卡片大小 = 512 字节 = 2^9 字节
卡表大小 = 堆大小 / 512

示例：堆 = 4GB
  卡表大小 = 4GB / 512 = 8MB
  卡表仅占堆的 1/512 ≈ 0.2%，开销极小
```

---

#### 卡表在 HotSpot 源码中的实现

```
HotSpot 源码路径（OpenJDK）：
  src/hotspot/share/gc/shared/cardTable.hpp
  src/hotspot/share/gc/shared/cardTable.cpp
```

核心数据结构（简化）：

```cpp
class CardTable {
  // 卡表字节数组，每字节对应堆中 512 字节
  jbyte* _byte_map;

  // 堆起始地址（用于计算卡表索引）
  HeapWord* _whole_heap_start;

  // 根据堆地址计算对应的卡表项地址
  jbyte* byte_for(const void* p) const {
    // 地址右移 9 位（÷512）得到卡表索引
    return &_byte_map[uintptr_t(p) >> card_shift];
  }

  // 卡片大小：512 字节 = 2^9
  static const int card_shift = 9;
  static const int card_size  = 1 << card_shift;  // 512
};
```

---

#### 写屏障（Write Barrier）—— 卡表的维护机制

卡表需要在**每次引用赋值时**被更新，这通过**写屏障**实现。写屏障是 JIT 编译器在对象引用赋值处插入的一小段额外代码。

**写屏障工作流程**：

```
Java 代码：
  oldGenObject.field = youngGenObject;  // 老年代对象引用新生代对象

JIT 编译后（插入写屏障）：
  oldGenObject.field = youngGenObject;
  // ↓ 写屏障（post-write barrier）
  card_table[address_of(oldGenObject) >> 9] = DIRTY;
```

**伪代码示意**：

```java
// 对象引用赋值（以 obj.field = value 为例）
void writeBarrier(Object obj, Object value) {
    // 1. 执行实际赋值
    obj.field = value;

    // 2. 如果赋值源（obj）在老年代，且 value 在新生代
    //    则将 obj 所在的卡片标记为"脏"
    if (isOldGen(obj) && isYoungGen(value)) {
        // 计算 obj 地址对应的卡表索引（地址 >> 9）
        int cardIndex = addressOf(obj) >>> 9;
        cardTable[cardIndex] = DIRTY;  // 标记为脏
    }
}
```

**HotSpot 实际写屏障（Serial/Parallel GC，简化 JIT 输出）**：

```asm
; Java：obj.field = newVal
mov [obj + field_offset], newVal    ; 实际赋值

; 写屏障：将 obj 地址右移 9 位得到卡表索引，标记为脏
shr rax, 9                          ; rax = obj地址 >> 9
mov byte ptr [card_table_base + rax], 0  ; 标记脏（值为0在HotSpot中表示dirty！）
```

> ⚠️ **注意**：HotSpot 实际用 `0` 表示 dirty（脏），用非零值表示 clean（干净），与直觉相反。具体数值定义在 `cardTableBarrierSet.hpp` 中。

#### Young GC 如何使用卡表

```
Young GC 执行步骤：

1. 暂停应用线程（STW）

2. 确定扫描范围（GC Roots）：
   ├─ 虚拟机栈引用
   ├─ 方法区静态变量
   └─ 卡表中所有脏卡片对应的老年代区域  ← 卡表在此发挥作用

3. 遍历卡表，找出所有脏卡片：
   for (int i = 0; i < cardTable.length; i++) {
       if (cardTable[i] == DIRTY) {
           // 扫描该卡片覆盖的 512 字节堆内存
           // 找出其中所有指向新生代的引用，加入 GC Roots
           scanCard(heapBase + i * 512, 512);
           // 扫描完毕，清除脏标记
           cardTable[i] = CLEAN;
       }
   }

4. 从 GC Roots 出发，标记新生代存活对象

5. 复制存活对象到 Survivor 区或晋升老年代

6. 恢复应用线程
```

#### 卡表的问题：伪共享（False Sharing）

**问题描述**：

现代 CPU 缓存以**缓存行（Cache Line）** 为单位操作，通常 64 字节。卡表每项 1 字节，因此一条缓存行对应 64 个卡表项，即 64 × 512 = **32KB 的堆内存**。

```
CPU 缓存行（64 字节）覆盖卡表项 0~63：
┌──┬──┬──┬──┬──┬── ... ──┬──┐
│0 │1 │2 │3 │4 │   ...   │63│  ← 64 个卡表项（1 缓存行）
└──┴──┴──┴──┴──┴── ... ──┴──┘
  ↕    ↕    ↕
Card0 Card1 Card2 ...（每卡 512 字节）

问题：
  线程 A 修改 Card0 → 标记卡表项 0 → 使整条缓存行失效
  线程 B 修改 Card32 → 标记卡表项 32 → 需要重新加载缓存行
  两个线程修改的是不同内存地址，却因共享同一缓存行而相互干扰
  → 伪共享，多核下性能下降
```

**JDK 解决方案（JEP 中引入，G1/ZGC 等新 GC 采用）**：

将每个卡表项从 1 字节扩展，使其独占一个缓存行，或通过对齐/填充避免伪共享：

```
-XX:+UseCondCardMark  （条件性卡表标记）

原来（无条件写入）：
  cardTable[i] = DIRTY;  // 无论原值是什么，都写入，导致缓存失效

优化后（条件写入）：
  if (cardTable[i] != DIRTY) {  // 先检查，避免不必要的写操作
      cardTable[i] = DIRTY;
  }
  // 已经是 DIRTY 则跳过写入，减少缓存行失效次数
```

#### 卡表 vs 其他记忆集实现对比

| 实现方式 | 精度 | 内存开销 | 维护成本 | 使用场景 |
|---------|------|---------|---------|---------|
| **卡表（Card Table）** | 卡片级（512 字节） | 极低（堆的 1/512） | 低（写屏障一条指令） | Serial、Parallel、CMS、G1（老年代→新生代） |
| **记忆集精确版（Point-in）** | 对象级 | 高 | 高 | 学术研究 |
| **位图（Bitmap）** | 字级（8 字节） | 低 | 低 | G1 内部 Region 间引用 |
| **RSet（Remembered Set）** | Region 级 | 中 | 中 | G1 GC（每个 Region 维护一个 RSet） |

> G1 GC 在卡表之上再封装了 **Remembered Set（RSet）**：每个 Region 有一个 RSet，记录有哪些其他 Region 的卡片引用了本 Region，是卡表的进一步组织抽象。

```

- 对象存活率低（10%左右）
- 使用标记-复制算法
- Eden + Survivor0 + Survivor1

老年代：
- 对象存活率高（90%以上）
- 使用标记-清除或标记-整理
- 根据具体GC器选择
```
