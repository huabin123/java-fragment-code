# 第二章：双亲委派模型

## 2.1 类加载器的层次结构

`ClassLoadingDemo.java → demonstrateClassLoaderHierarchy()` 展示了这个层次：

```
Bootstrap ClassLoader（启动类加载器）
  └── 加载：JDK 核心类（rt.jar / jdk.boot），如 java.lang.*、java.util.*
  └── 实现：C++ 实现，Java 中表示为 null

Extension ClassLoader（扩展类加载器，JDK 9+ 改名 Platform ClassLoader）
  └── 加载：JDK 扩展目录（lib/ext 或 java.ext.dirs），如 javax.*
  └── 父加载器：Bootstrap

Application ClassLoader（应用类加载器 / 系统类加载器）
  └── 加载：用户 classpath 下的类（自己写的代码）
  └── 父加载器：Extension
  └── 获取：ClassLoader.getSystemClassLoader()

自定义 ClassLoader
  └── 加载：从网络、数据库、加密文件等自定义来源加载类
  └── 父加载器：通常是 Application ClassLoader
```

## 2.2 双亲委派的工作流程

```java
// ClassLoader.loadClass() 的核心逻辑（简化）
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 先检查是否已经加载过
    Class<?> c = findLoadedClass(name);

    if (c == null) {
        // 2. 委派给父加载器（双亲委派的核心）
        if (parent != null) {
            c = parent.loadClass(name, false);  // 递归委派
        } else {
            c = findBootstrapClassOrNull(name);  // 到达 Bootstrap
        }

        // 3. 父加载器加载不到，自己尝试加载
        if (c == null) {
            c = findClass(name);  // 子类重写此方法实现自定义加载
        }
    }
    return c;
}
```

**流程图**：
```
AppClassLoader.loadClass("com.example.MyClass")
  → 委派给 ExtClassLoader
    → 委派给 BootstrapClassLoader
      → Bootstrap 找不到 → 返回 null
    → ExtClassLoader 找不到 → 返回 null
  → AppClassLoader 自己加载 → 找到！返回 Class 对象
```

## 2.3 双亲委派的两大目的

**目的1：安全性**
```java
// 如果没有双亲委派，攻击者可以自定义一个 java.lang.String 类
// 替换 JDK 的 String，在其中植入恶意代码

// 有了双亲委派：
// 自定义 ClassLoader 加载 "java.lang.String"
//   → 委派给 Bootstrap → Bootstrap 先找到真正的 String.class
//   → 直接返回，自定义的 String 永远不会被加载
```

**目的2：唯一性**
```java
// 保证同一个类只有一份（由同一个 ClassLoader 加载的 Class 才是"同一个类"）
// 避免多份 Class 对象导致 instanceof 失败等问题
ClassLoader cl1 = new MyClassLoader();
ClassLoader cl2 = new MyClassLoader();
Class<?> c1 = cl1.loadClass("com.example.Foo");
Class<?> c2 = cl2.loadClass("com.example.Foo");
c1 == c2;  // false！不同 ClassLoader 加载的是两个独立的 Class
c1.newInstance() instanceof c2;  // false！
```

## 2.4 双亲委派的破坏场景

并非所有场景都适合双亲委派，有三次典型的"破坏"：

**第一次破坏：JDK 1.2 之前**
```
JDK 1.2 引入双亲委派之前，ClassLoader 的 loadClass() 可以被随意重写
为了兼容旧代码，ClassLoader 允许子类重写 loadClass()（但建议只重写 findClass()）
```

**第二次破坏：SPI 机制（如 JDBC）**
```java
// 问题：Bootstrap ClassLoader 加载了 java.sql.Driver（接口）
//       但 JDBC 驱动（如 MySQL 的 com.mysql.Driver）在 classpath，只有 AppClassLoader 能加载
//       Bootstrap 无法"向下"委派给 AppClassLoader

// 解决：线程上下文类加载器（Thread Context ClassLoader）
// Bootstrap 通过 Thread.currentThread().getContextClassLoader() 获取 AppClassLoader
// 打破委派，用 AppClassLoader 加载 SPI 实现

ClassLoader contextClassLoader = Thread.currentThread().getContextClassLoader();
// ServiceLoader 内部就是这样加载 SPI 实现的
```

**第三次破坏：OSGI / 热部署**
```
OSGI 的 Bundle 有自己的 ClassLoader，类加载是网状结构而非树状
每个 Bundle 优先在自己的 ClassLoader 中查找，再按网状关系委派
实现了模块化和动态部署（Tomcat 也采用类似机制）
```

## 2.5 本章总结

- **三层结构**：Bootstrap（核心类）→ Extension（扩展类）→ Application（用户类）
- **双亲委派流程**：先委派父，父找不到自己找；子类只重写 `findClass()`
- **两大目的**：安全性（防止替换核心类）、唯一性（同类只有一份）
- **三次破坏**：兼容旧代码、SPI 线程上下文类加载器、OSGI 网状委派
