# 第五章：类加载最佳实践

## 5.1 利用类初始化实现线程安全的延迟加载

```java
// 静态内部类单例：利用类初始化的线程安全性（JVM 保证 <clinit> 只执行一次）
public class Singleton {
    private Singleton() {}

    private static class Holder {
        // Holder 类在第一次被使用时才初始化（延迟加载）
        // JVM 保证 <clinit> 在多线程下只执行一次（线程安全）
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;  // 触发 Holder 初始化
    }
}

// 优于 Double-Checked Locking：
// - 无需 volatile
// - 无需 synchronized
// - 利用 JVM 机制保证线程安全和延迟初始化
```

## 5.2 避免类加载器内存泄漏

```java
// 常见泄漏场景：自定义 ClassLoader 被静态变量或 ThreadLocal 持有
// ❌ 静态 Map 持有 ClassLoader 加载的对象
static Map<String, Object> globalCache = new HashMap<>();
globalCache.put("key", customLoader.loadClass("Foo").newInstance());
// globalCache 不清理 → 对象不释放 → ClassLoader 不释放 → 所有类不卸载

// ❌ ThreadLocal 未 remove（线程池场景）
ThreadLocal<Object> tl = new ThreadLocal<>();
tl.set(customLoader.loadClass("Bar").newInstance());
// 线程池线程不销毁 → ThreadLocal 不清理 → ClassLoader 泄漏

// ✅ 热部署后清理所有对旧 ClassLoader 的引用
public void undeploy() {
    globalCache.clear();              // 清理静态缓存
    ThreadLocalCleaner.cleanAll();    // 清理 ThreadLocal
    oldClassLoader = null;            // 移除强引用
    System.gc();                      // 建议（不强制）GC 回收
}
```

## 5.3 Class.forName() vs ClassLoader.loadClass()

```java
// 两者的区别：
// Class.forName()：加载 + 初始化（执行 static 块）
Class<?> c1 = Class.forName("com.example.MyClass");
// MyClass 的 static 块会被执行

// ClassLoader.loadClass()：只加载，不初始化
Class<?> c2 = ClassLoader.getSystemClassLoader().loadClass("com.example.MyClass");
// MyClass 的 static 块不执行，直到第一次使用才初始化

// JDBC 驱动为何用 Class.forName()：
Class.forName("com.mysql.cj.jdbc.Driver");
// 驱动的 static 块会向 DriverManager 注册自己
// 如果用 loadClass() 则 static 块不执行，驱动注册失败
```

## 5.4 JDK 9+ 模块化兼容问题处理

```bash
# Spring、Hibernate 等框架需要通过反射访问 JDK 内部类
# JDK 9+ 模块化后，未导出的包默认不允许反射访问，需要显式开放

# 方案1：JVM 启动参数 --add-opens
java --add-opens java.base/java.lang=ALL-UNNAMED \
     --add-opens java.base/java.util=ALL-UNNAMED \
     -jar app.jar

# 方案2：module-info.java 中声明 opens
module com.example.myapp {
    opens com.example.internal to spring.core;  // 只对 spring.core 开放
}

# Spring Boot 3.x 已完全支持 JDK 17+ 模块化，无需额外配置
# 旧版框架（Spring Boot 2.x on JDK 17）通常需要添加 --add-opens 参数
```

## 5.5 本章总结

**四条核心实践**：
1. **延迟加载单例**：静态内部类 + JVM 类初始化机制，比 DCL 更简洁安全
2. **防止 ClassLoader 泄漏**：热部署后清理所有强引用（静态 Map、ThreadLocal）
3. **forName vs loadClass**：需要执行 `static` 块（如 JDBC 注册）用 `forName`；只需加载类结构用 `loadClass`
4. **JDK 9+ 模块化**：需要反射访问内部类时用 `--add-opens`，或在 `module-info.java` 中声明 `opens`
