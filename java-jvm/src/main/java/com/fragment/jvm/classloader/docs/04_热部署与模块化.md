# 第四章：热部署与模块化

## 4.1 热部署的原理

热部署 = 不重启 JVM，动态替换类的实现。核心原理：

```
JVM 中，一个类由"类加载器 + 全限定名"唯一确定
要替换一个类，不能卸载/重新加载同一个 ClassLoader 中的类
（类卸载条件：ClassLoader 被回收 → 其加载的所有类都被卸载）

热部署步骤：
1. 抛弃旧 ClassLoader（让其失去所有强引用，等待 GC 回收）
2. 创建新 ClassLoader（读取磁盘上新版本的 .class 文件）
3. 用新 ClassLoader 加载类，创建新实例

本质：不是"重新加载"同一个类，而是用新 ClassLoader 加载新版本类
```

---

## 4.2 简单热部署实现

`HotDeployDemo.java` 演示了完整的热部署流程：

```java
// 热部署的核心代码结构
public class HotDeployManager {
    private volatile ClassLoader currentLoader;
    private volatile Object currentInstance;

    public void reload(String classDir, String className) throws Exception {
        // 1. 创建新 ClassLoader（指向新的 .class 文件目录）
        ClassLoader newLoader = new FileClassLoader(classDir);

        // 2. 加载新版本的类
        Class<?> newClass = newLoader.loadClass(className);

        // 3. 创建新实例
        Object newInstance = newClass.getDeclaredConstructor().newInstance();

        // 4. 替换（volatile 保证可见性）
        Object oldInstance = currentInstance;
        currentInstance = newInstance;
        currentLoader = newLoader;

        // 5. 旧 ClassLoader 和实例失去强引用 → 等待 GC 回收
        // oldInstance 和旧 ClassLoader 在此方法结束后无强引用
        System.out.println("热部署完成，旧版本等待 GC 回收");
    }
}
```

## 4.3 Tomcat 的类加载机制

Tomcat 打破了标准双亲委派，实现 WebApp 隔离：

```
Bootstrap ClassLoader（JDK 核心类）
  └── Extension ClassLoader
        └── Application ClassLoader（Tomcat 自身类）
              └── Common ClassLoader（Tomcat + 所有 WebApp 共享）
                    ├── Catalina ClassLoader（Tomcat 内部，WebApp 不可见）
                    └── Shared ClassLoader（所有 WebApp 共享但不可访问 Tomcat）
                          ├── WebApp ClassLoader（app1，独立）
                          └── WebApp ClassLoader（app2，独立）

WebApp ClassLoader 的特殊行为：
  优先从自己的 WEB-INF/classes 和 WEB-INF/lib 加载（打破双亲委派）
  找不到时才委派给父加载器
  → 实现 app1 和 app2 的类完全隔离
  → 同一个 Tomcat 可以运行使用不同版本 Spring 的两个 app
```

## 4.4 JDK 9 模块化系统（JPMS）

```
JDK 9 引入模块系统（Java Platform Module System），重构了类加载器：

JDK 8：Bootstrap + Extension + Application
JDK 9+：Bootstrap + Platform + System（名称变化，机制类似）

模块（module）：
  - 每个模块有 module-info.java，声明 requires（依赖）和 exports（对外暴露）
  - 只有 exports 的包才能被其他模块访问（强封装）
  - 没有 exports 的包，即使 public 类也无法从外部反射访问

// module-info.java 示例
module com.example.myapp {
    requires java.base;           // 依赖 JDK 基础模块
    requires java.sql;            // 依赖 SQL 模块
    exports com.example.api;      // 对外暴露 API 包
    // com.example.internal 不导出，外部无法访问
}

影响：
  - 反射访问模块内部类需要 --add-opens（如 Spring / Hibernate 需要配置）
  - 类路径仍然支持（unnamed module），老项目不用立即迁移
```

## 4.5 本章总结

- **热部署原理**：不是"重新加载"，而是新建 ClassLoader 加载新版本，旧 ClassLoader 等待 GC
- **实现关键**：旧 ClassLoader 必须失去所有强引用才能被 GC 回收，否则类无法卸载
- **Tomcat 隔离**：每个 WebApp 有独立 ClassLoader，且优先自加载，实现版本隔离
- **JDK 9 模块化**：强封装（未导出的包外部不可访问），对 Spring 等框架有影响（需 --add-opens）
