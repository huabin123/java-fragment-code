# 内存泄漏排查

## 💡 大白话精华总结

**内存泄漏是什么？**
- 对象不用了，但还被引用着，GC回收不了，内存越来越少
- 就像房间里的垃圾，虽然不用了但还堆在那，越堆越多

**内存泄漏 vs 内存溢出：**
```
内存泄漏（Memory Leak）：
- 慢性病，内存慢慢变少
- 对象不用了但还被引用
- 最终导致OOM

内存溢出（Out Of Memory）：
- 急性病，内存突然不够
- 对象太多，内存真的不够用
- 直接OOM
```

**内存泄漏的5大常见原因：**

**1. 静态集合**
```java
// 问题代码
public class Cache {
    private static Map<String, Object> cache = new HashMap<>();
    
    public void add(String key, Object value) {
        cache.put(key, value);  // 永远不清理，越来越多
    }
}

// 解决方案
使用WeakHashMap或定期清理
```

**2. 未关闭的资源**
```java
// 问题代码
Connection conn = getConnection();
// 忘记关闭，连接泄漏

// 解决方案
try (Connection conn = getConnection()) {
    // 自动关闭
}
```

**3. ThreadLocal未清理**
```java
// 问题代码
ThreadLocal<User> userHolder = new ThreadLocal<>();
userHolder.set(user);  // 线程池复用时，user一直在

// 解决方案
finally {
    userHolder.remove();  // 用完必须清理
}
```

**4. 监听器未移除**
```java
// 问题代码
button.addListener(listener);  // 添加监听器
// 忘记移除，listener一直被引用

// 解决方案
button.removeListener(listener);
```

**5. 内部类持有外部类引用**
```java
// 问题代码
public class Outer {
    class Inner {
        // 自动持有Outer的引用
    }
}

// 解决方案
使用静态内部类
```

**排查内存泄漏的4步法：**
```bash
# 1. 导出堆dump
jmap -dump:live,format=b,file=heap.hprof 12345

# 2. 使用MAT分析
# 下载Eclipse MAT，打开heap.hprof

# 3. 查看Leak Suspects（泄漏嫌疑）
# MAT会自动分析并给出报告

# 4. 查看Dominator Tree（支配树）
# 找到占用内存最多的对象
```

**MAT分析要点：**
```
1. Histogram（直方图）
   - 看哪个类的实例最多
   
2. Dominator Tree（支配树）
   - 看哪个对象占用内存最大
   
3. Leak Suspects（泄漏嫌疑）
   - MAT自动分析的可疑对象
   
4. Path to GC Roots（GC根路径）
   - 看对象被谁引用，为什么回收不了
```

**预防内存泄漏的最佳实践：**
- ✅ 使用try-with-resources关闭资源
- ✅ ThreadLocal用完必须remove
- ✅ 集合用完及时clear
- ✅ 监听器用完及时移除
- ✅ 使用弱引用（WeakReference）
- ✅ 定期review代码，查找潜在泄漏

**一句话记住：**
> 内存泄漏就是对象不用了还被引用，用jmap导dump，MAT找泄漏！

## 一、什么是内存泄漏

### 1.1 内存泄漏的定义

```
内存泄漏（Memory Leak）：
- 对象不再被使用
- 但仍然被引用
- 无法被GC回收
- 导致内存持续增长
```

### 1.2 为什么Java会有内存泄漏

```
问题：Java不是有GC吗？

解答：
GC只能回收"不可达"的对象
如果对象仍然可达（有引用），GC无法回收

示例：
静态集合持有对象
    ↓
对象可达（有引用）
    ↓
GC无法回收
    ↓
内存泄漏
```

### 1.3 内存泄漏 vs 内存溢出

| 对比项 | 内存泄漏 | 内存溢出 |
|-------|---------|---------|
| **定义** | 对象无法被回收 | 内存不足 |
| **原因** | 不当的引用 | 内存真的不够 |
| **表现** | 内存持续增长 | 立即OOM |
| **GC行为** | GC无效 | GC后仍不足 |
| **解决** | 修复代码 | 增加内存 |

### 1.4 内存泄漏的危害

```
危害1：内存持续增长
- 可用内存越来越少
- 最终导致OOM

危害2：频繁Full GC
- 老年代持续增长
- 触发频繁Full GC
- 应用响应变慢

危害3：系统不稳定
- 应用崩溃
- 服务不可用
- 影响用户体验

危害4：难以排查
- 问题隐蔽
- 可能几天后才出现
- 排查成本高
```

## 二、内存泄漏的常见原因

### 2.1 静态集合持有对象

#### 为什么会泄漏？

```
问题：
1. 静态变量生命周期 = JVM生命周期
2. 静态集合持有对象
3. 对象无法被GC
4. 内存泄漏

流程：
创建对象 → 添加到静态集合 → 对象不再使用 → 仍被集合引用 → 无法GC → 泄漏
```

#### 问题代码

```java
public class CacheManager {
    
    // 问题：静态集合持有对象
    private static List<Object> cache = new ArrayList<>();
    
    public void addToCache(Object obj) {
        cache.add(obj);
        // 问题：对象永远不会被移除
    }
    
    public static void main(String[] args) {
        CacheManager manager = new CacheManager();
        
        // 持续添加对象
        for (int i = 0; i < 1000000; i++) {
            manager.addToCache(new byte[1024]);  // 每次1KB
        }
        
        // 结果：
        // - cache持有100万个对象
        // - 占用约1GB内存
        // - 对象无法被GC
        // - 最终OOM
    }
}
```

#### 内存增长趋势

```
内存使用
  ↑
  │                                    ╱
  │                                  ╱
  │                                ╱
  │                              ╱
  │                            ╱
  │                          ╱
  │                        ╱
  │                      ╱
  │                    ╱
  │                  ╱
  │                ╱
  │              ╱
  │            ╱
  │          ╱
  │        ╱
  │      ╱
  │    ╱
  │  ╱
  │╱
  └────────────────────────────────────→ 时间
  
特征：
- 内存持续线性增长
- Full GC后内存不下降
- 最终OOM
```

#### 解决方案

```java
public class CacheManager {
    
    // 方案1：使用WeakHashMap
    private static Map<Object, Object> cache = new WeakHashMap<>();
    
    // 方案2：使用软引用
    private static Map<String, SoftReference<Object>> cache2 = new HashMap<>();
    
    public void addToCache2(String key, Object obj) {
        cache2.put(key, new SoftReference<>(obj));
    }
    
    // 方案3：设置缓存大小限制
    private static final int MAX_CACHE_SIZE = 10000;
    private static LinkedHashMap<String, Object> cache3 = new LinkedHashMap<String, Object>(16, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(Map.Entry<String, Object> eldest) {
            return size() > MAX_CACHE_SIZE;
        }
    };
    
    // 方案4：使用Guava Cache（推荐）
    private static Cache<String, Object> cache4 = CacheBuilder.newBuilder()
        .maximumSize(10000)
        .expireAfterWrite(10, TimeUnit.MINUTES)
        .build();
    
    // 方案5：定期清理
    private static List<Object> cache5 = new ArrayList<>();
    
    public void clearCache() {
        cache5.clear();
    }
    
    // 启动定时任务
    @PostConstruct
    public void init() {
        ScheduledExecutorService executor = Executors.newScheduledThreadPool(1);
        executor.scheduleAtFixedRate(this::clearCache, 1, 1, TimeUnit.HOURS);
    }
}
```

### 2.2 监听器未注销

#### 为什么会泄漏？

```
问题：
1. 注册监听器
2. 监听器持有对象引用
3. 对象不再使用
4. 忘记注销监听器
5. 对象仍被监听器持有
6. 无法GC

流程：
注册监听器 → 监听器持有对象 → 对象不再使用 → 未注销监听器 → 对象仍被引用 → 泄漏
```

#### 问题代码

```java
public class EventBus {
    
    private static List<EventListener> listeners = new ArrayList<>();
    
    public static void register(EventListener listener) {
        listeners.add(listener);
    }
    
    public static void unregister(EventListener listener) {
        listeners.remove(listener);
    }
}

public class UserService {
    
    private EventListener listener;
    
    public UserService() {
        // 注册监听器
        listener = new EventListener() {
            @Override
            public void onEvent(Event event) {
                // 处理事件
            }
        };
        EventBus.register(listener);
    }
    
    // 问题：没有注销监听器
    // UserService对象销毁时，listener仍被EventBus持有
    // 导致UserService无法被GC
}
```

#### 泄漏分析

```
对象引用关系：

EventBus (static)
    ↓
listeners (static List)
    ↓
EventListener (匿名内部类)
    ↓
UserService (外部类引用)
    ↓
其他对象

结果：
- UserService不再使用
- 但仍被EventBus间接持有
- 无法被GC
- 内存泄漏
```

#### 解决方案

```java
public class UserService {
    
    private EventListener listener;
    
    public UserService() {
        listener = new EventListener() {
            @Override
            public void onEvent(Event event) {
                // 处理事件
            }
        };
        EventBus.register(listener);
    }
    
    // 方案1：手动注销
    public void destroy() {
        EventBus.unregister(listener);
        listener = null;
    }
    
    // 方案2：使用@PreDestroy（Spring）
    @PreDestroy
    public void cleanup() {
        EventBus.unregister(listener);
    }
    
    // 方案3：使用弱引用
    public void registerWithWeakReference() {
        WeakReference<EventListener> weakListener = new WeakReference<>(listener);
        EventBus.register(weakListener.get());
    }
}
```

### 2.3 ThreadLocal未清理

#### 为什么会泄漏？

```
问题：
1. ThreadLocal存储线程私有数据
2. 数据存储在Thread的ThreadLocalMap中
3. 线程池中的线程不销毁
4. ThreadLocal未清理
5. 数据一直存在
6. 内存泄漏

ThreadLocal内部结构：
Thread
    ↓
ThreadLocalMap
    ↓
Entry[] (弱引用ThreadLocal，强引用value)
    ↓
value (强引用)

问题：
- ThreadLocal被回收（弱引用）
- 但value仍被强引用
- 线程不销毁，value无法回收
- 内存泄漏
```

#### 问题代码

```java
public class UserContext {
    
    private static ThreadLocal<User> userThreadLocal = new ThreadLocal<>();
    
    public static void setUser(User user) {
        userThreadLocal.set(user);
        // 问题：没有调用remove
    }
    
    public static User getUser() {
        return userThreadLocal.get();
    }
}

// 在线程池中使用
public class UserService {
    
    private ExecutorService executor = Executors.newFixedThreadPool(10);
    
    public void processUser(User user) {
        executor.submit(() -> {
            // 设置用户上下文
            UserContext.setUser(user);
            
            try {
                // 业务处理
                doSomething();
            } finally {
                // 问题：忘记清理ThreadLocal
                // UserContext.remove();
            }
        });
    }
    
    // 结果：
    // - 线程池中的10个线程
    // - 每个线程都持有User对象
    // - User对象无法被GC
    // - 内存泄漏
}
```

#### 泄漏场景

```
场景1：Web应用
- Tomcat线程池
- 请求处理完成
- 线程返回线程池
- ThreadLocal未清理
- 下次请求可能读到脏数据

场景2：异步任务
- 线程池执行任务
- 任务完成
- 线程返回线程池
- ThreadLocal未清理
- 内存泄漏
```

#### 解决方案

```java
public class UserContext {
    
    private static ThreadLocal<User> userThreadLocal = new ThreadLocal<>();
    
    public static void setUser(User user) {
        userThreadLocal.set(user);
    }
    
    public static User getUser() {
        return userThreadLocal.get();
    }
    
    // 方案1：手动清理
    public static void remove() {
        userThreadLocal.remove();
    }
}

public class UserService {
    
    private ExecutorService executor = Executors.newFixedThreadPool(10);
    
    // 方案2：使用try-finally确保清理
    public void processUser(User user) {
        executor.submit(() -> {
            try {
                UserContext.setUser(user);
                doSomething();
            } finally {
                UserContext.remove();  // 确保清理
            }
        });
    }
    
    // 方案3：使用装饰器模式
    public void processUserWithDecorator(User user) {
        executor.submit(new Runnable() {
            @Override
            public void run() {
                try {
                    UserContext.setUser(user);
                    doSomething();
                } finally {
                    UserContext.remove();
                }
            }
        });
    }
    
    // 方案4：使用InheritableThreadLocal（谨慎使用）
    private static InheritableThreadLocal<User> inheritableThreadLocal = new InheritableThreadLocal<>();
}
```

### 2.4 资源未关闭

#### 为什么会泄漏？

```
问题：
1. 打开资源（连接、流、文件）
2. 资源对象持有内存
3. 使用完成后未关闭
4. 资源对象无法被GC
5. 内存泄漏

常见资源：
- 数据库连接
- 文件流
- Socket连接
- HTTP连接
- 缓存连接
```

#### 问题代码

```java
public class DatabaseService {
    
    // 问题代码1：连接未关闭
    public List<User> queryUsers() throws SQLException {
        Connection conn = DriverManager.getConnection(url, user, password);
        Statement stmt = conn.createStatement();
        ResultSet rs = stmt.executeQuery("SELECT * FROM users");
        
        List<User> users = new ArrayList<>();
        while (rs.next()) {
            users.add(new User(rs.getString("name")));
        }
        
        // 问题：忘记关闭连接
        // rs.close();
        // stmt.close();
        // conn.close();
        
        return users;
    }
    
    // 问题代码2：流未关闭
    public String readFile(String path) throws IOException {
        FileInputStream fis = new FileInputStream(path);
        BufferedReader reader = new BufferedReader(new InputStreamReader(fis));
        
        StringBuilder sb = new StringBuilder();
        String line;
        while ((line = reader.readLine()) != null) {
            sb.append(line);
        }
        
        // 问题：忘记关闭流
        // reader.close();
        // fis.close();
        
        return sb.toString();
    }
}
```

#### 泄漏影响

```
影响1：连接池耗尽
- 连接未归还
- 连接池无可用连接
- 新请求阻塞

影响2：文件句柄耗尽
- 文件未关闭
- 文件句柄泄漏
- 无法打开新文件

影响3：内存泄漏
- 资源对象占用内存
- 无法被GC
- 内存持续增长
```

#### 解决方案

```java
public class DatabaseService {
    
    // 方案1：try-finally手动关闭
    public List<User> queryUsersWithFinally() throws SQLException {
        Connection conn = null;
        Statement stmt = null;
        ResultSet rs = null;
        
        try {
            conn = DriverManager.getConnection(url, user, password);
            stmt = conn.createStatement();
            rs = stmt.executeQuery("SELECT * FROM users");
            
            List<User> users = new ArrayList<>();
            while (rs.next()) {
                users.add(new User(rs.getString("name")));
            }
            return users;
        } finally {
            // 确保关闭资源
            if (rs != null) {
                try {
                    rs.close();
                } catch (SQLException e) {
                    // 记录日志
                }
            }
            if (stmt != null) {
                try {
                    stmt.close();
                } catch (SQLException e) {
                    // 记录日志
                }
            }
            if (conn != null) {
                try {
                    conn.close();
                } catch (SQLException e) {
                    // 记录日志
                }
            }
        }
    }
    
    // 方案2：try-with-resources（推荐）
    public List<User> queryUsersWithTryWithResources() throws SQLException {
        try (Connection conn = DriverManager.getConnection(url, user, password);
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SELECT * FROM users")) {
            
            List<User> users = new ArrayList<>();
            while (rs.next()) {
                users.add(new User(rs.getString("name")));
            }
            return users;
        }
        // 自动关闭资源
    }
    
    // 方案3：使用连接池
    private DataSource dataSource;
    
    public List<User> queryUsersWithPool() throws SQLException {
        try (Connection conn = dataSource.getConnection();
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SELECT * FROM users")) {
            
            List<User> users = new ArrayList<>();
            while (rs.next()) {
                users.add(new User(rs.getString("name")));
            }
            return users;
        }
    }
}
```

### 2.5 内部类持有外部类引用

#### 为什么会泄漏？

```
问题：
1. 非静态内部类持有外部类引用
2. 内部类对象生命周期长
3. 外部类对象无法被GC
4. 内存泄漏

引用关系：
外部类
    ↓
内部类（隐式持有外部类引用）
    ↓
如果内部类被长期持有
    ↓
外部类也被长期持有
    ↓
外部类无法被GC
```

#### 问题代码

```java
public class OuterClass {
    
    private byte[] data = new byte[1024 * 1024];  // 1MB
    
    // 问题：非静态内部类
    public class InnerClass {
        public void doSomething() {
            // 可以访问外部类的成员
            System.out.println(data.length);
        }
    }
    
    public InnerClass getInner() {
        return new InnerClass();
    }
}

public class LeakDemo {
    
    private static List<Object> cache = new ArrayList<>();
    
    public static void main(String[] args) {
        for (int i = 0; i < 1000; i++) {
            OuterClass outer = new OuterClass();
            OuterClass.InnerClass inner = outer.getInner();
            
            // 问题：只缓存内部类对象
            cache.add(inner);
            
            // 但内部类持有外部类引用
            // 导致外部类也无法被GC
            // 1000个外部类 × 1MB = 1GB内存泄漏
        }
    }
}
```

#### 字节码分析

```java
// 编译后的内部类
public class OuterClass$InnerClass {
    
    // 隐式持有外部类引用
    final OuterClass this$0;
    
    public OuterClass$InnerClass(OuterClass outer) {
        this.this$0 = outer;
    }
    
    public void doSomething() {
        // 访问外部类成员
        System.out.println(this$0.data.length);
    }
}
```

#### 解决方案

```java
public class OuterClass {
    
    private byte[] data = new byte[1024 * 1024];
    
    // 方案1：使用静态内部类
    public static class StaticInnerClass {
        public void doSomething() {
            // 无法直接访问外部类成员
            // 不持有外部类引用
        }
    }
    
    // 方案2：使用局部类
    public Object createInner() {
        class LocalInner {
            public void doSomething() {
                // 局部类
            }
        }
        return new LocalInner();
    }
    
    // 方案3：使用匿名类时注意
    public Runnable createRunnable() {
        // 问题：匿名类持有外部类引用
        return new Runnable() {
            @Override
            public void run() {
                // 可以访问外部类成员
                System.out.println(data.length);
            }
        };
    }
    
    // 方案4：使用Lambda（不持有外部类引用）
    public Runnable createRunnableWithLambda() {
        return () -> {
            // Lambda不持有外部类引用
            // 除非显式使用外部类成员
        };
    }
}
```

### 2.6 HashMap的Key未重写hashCode和equals

#### 为什么会泄漏？

```
问题：
1. 使用对象作为HashMap的Key
2. 对象未重写hashCode和equals
3. 对象内容相同但实例不同
4. HashMap中存在重复的Key
5. 旧的Key无法被移除
6. 内存泄漏

示例：
map.put(new Key("a"), value1);
map.put(new Key("a"), value2);

结果：
- 两个Key对象不同
- HashMap中有两个entry
- 旧的entry无法被移除
```

#### 问题代码

```java
public class Key {
    private String name;
    
    public Key(String name) {
        this.name = name;
    }
    
    // 问题：未重写hashCode和equals
}

public class HashMapLeakDemo {
    
    private static Map<Key, String> map = new HashMap<>();
    
    public static void main(String[] args) {
        // 添加数据
        for (int i = 0; i < 1000000; i++) {
            Key key = new Key("key" + (i % 100));
            map.put(key, "value" + i);
        }
        
        // 问题：
        // - 实际只有100个不同的key
        // - 但map中有100万个entry
        // - 因为Key对象不同（即使name相同）
        // - 旧的entry无法被移除
        // - 内存泄漏
        
        System.out.println("Map size: " + map.size());  // 1000000
    }
}
```

#### 解决方案

```java
public class Key {
    private String name;
    
    public Key(String name) {
        this.name = name;
    }
    
    // 方案1：重写hashCode和equals
    @Override
    public int hashCode() {
        return name != null ? name.hashCode() : 0;
    }
    
    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Key key = (Key) obj;
        return Objects.equals(name, key.name);
    }
}

public class HashMapLeakDemo {
    
    private static Map<Key, String> map = new HashMap<>();
    
    public static void main(String[] args) {
        // 添加数据
        for (int i = 0; i < 1000000; i++) {
            Key key = new Key("key" + (i % 100));
            map.put(key, "value" + i);
        }
        
        // 结果：
        // - map中只有100个entry
        // - 相同的key会覆盖旧值
        // - 没有内存泄漏
        
        System.out.println("Map size: " + map.size());  // 100
    }
    
    // 方案2：使用String作为Key
    private static Map<String, String> map2 = new HashMap<>();
    
    public static void demo2() {
        for (int i = 0; i < 1000000; i++) {
            String key = "key" + (i % 100);
            map2.put(key, "value" + i);
        }
        System.out.println("Map size: " + map2.size());  // 100
    }
}
```

## 三、内存泄漏排查流程

### 3.1 排查流程图

```
发现内存问题
    ↓
1. 确认是否内存泄漏
   - 观察内存趋势
   - Full GC后内存是否下降
    ↓
2. 生成堆转储
   - jmap -dump
   - 或OOM时自动生成
    ↓
3. 分析堆转储
   - 使用MAT/JProfiler
   - 找到占用内存最多的对象
    ↓
4. 分析引用链
   - 查看GC Roots
   - 找到泄漏路径
    ↓
5. 定位代码
   - 根据引用链定位代码
   - 分析泄漏原因
    ↓
6. 修复问题
   - 修改代码
   - 验证效果
```

### 3.2 判断是否内存泄漏

#### 方法1：观察内存趋势

```bash
# 持续监控堆内存使用
jstat -gcutil <pid> 1000

  0.00  95.00  55.00  62.00  90.00  85.00  101    1.010    5    2.500    3.510
```

### 3.4 使用MAT分析堆转储

```
File → Open Heap Dump → 选择heap.hprof
```

#### 步骤2：查看Leak Suspects

```
MAT会自动分析可疑的内存泄漏
显示：
- 占用内存最多的对象
- 可能的泄漏路径

```

#### 步骤3：查看Dominator Tree

```
Dominator Tree显示：
```

#### 步骤4：查看GC Roots路径

```
右键对象 → Path to GC Roots → exclude weak references
```

## 四、内存泄漏案例分析

### 案例1：生产环境内存泄漏排查

#### 问题现象

```
现象：
- 应用运行3天后OOM
- 重启后正常，3天后再次OOM
- Full GC频繁，每次耗时5秒+
- 用户投诉系统卡顿
```

#### 排查过程

```
步骤1：观察内存趋势
jstat -gcutil <pid> 1000

发现：
- 老年代使用率持续增长
- Full GC后内存不下降
- 确认内存泄漏

步骤2：生成堆转储
jmap -dump:format=b,file=heap.hprof <pid>

步骤3：使用MAT分析
打开heap.hprof

Leak Suspects显示：
- ConcurrentHashMap占用2GB
- 包含500万个entry
- 被静态字段持有

步骤4：查看引用链
Thread "main"
    ↓
UserSessionManager (static)
    ↓
sessionMap (ConcurrentHashMap)
    ↓
UserSession[] (500万个)

步骤5：分析代码
public class UserSessionManager {
    private static ConcurrentHashMap<String, UserSession> sessionMap = new ConcurrentHashMap<>();
    
    public static void addSession(String sessionId, UserSession session) {
        sessionMap.put(sessionId, session);
        // 问题：session永不过期，永不删除
    }
}
```

#### 解决方案

```java
public class UserSessionManager {
    
    // 方案1：使用Guava Cache（推荐）
    private static Cache<String, UserSession> sessionCache = CacheBuilder.newBuilder()
        .maximumSize(100000)  // 最大10万个session
        .expireAfterWrite(30, TimeUnit.MINUTES)  // 30分钟过期
        .build();
    
    public static void addSession(String sessionId, UserSession session) {
        sessionCache.put(sessionId, session);
    }
    
    // 方案2：定期清理
    private static ConcurrentHashMap<String, UserSession> sessionMap = new ConcurrentHashMap<>();
    
    @Scheduled(fixedRate = 60000)  // 每分钟执行一次
    public void cleanExpiredSessions() {
        long now = System.currentTimeMillis();
        sessionMap.entrySet().removeIf(entry -> 
            now - entry.getValue().getCreateTime() > 30 * 60 * 1000
        );
    }
}
```

#### 效果验证

```
优化后：
- 内存使用稳定在2GB
- Full GC频率：1次/小时
- Full GC时间：< 500ms
- 应用稳定运行30天+
```

## 五、预防内存泄漏

### 5.1 编码规范

```java
/**
 * 内存泄漏预防最佳实践
 */
public class MemoryLeakPrevention {
    
    // 1. 避免使用静态集合
    // 不好
    private static List<Object> cache = new ArrayList<>();
    
    // 好：使用有界缓存
    private static Cache<String, Object> cache2 = CacheBuilder.newBuilder()
        .maximumSize(10000)
        .build();
    
    // 2. 及时注销监听器
    public void registerListener() {
        EventBus.register(listener);
    }
    
    @PreDestroy
    public void cleanup() {
        EventBus.unregister(listener);
    }
    
    // 3. ThreadLocal必须清理
    public void processRequest() {
        try {
            threadLocal.set(value);
            doSomething();
        } finally {
            threadLocal.remove();  // 必须清理
        }
    }
    
    // 4. 资源必须关闭
    public void readFile() throws IOException {
        try (FileInputStream fis = new FileInputStream("file.txt")) {
            // 使用资源
        }  // 自动关闭
    }
    
    // 5. 使用静态内部类
    public static class StaticInner {
        // 不持有外部类引用
    }
    
    // 6. HashMap的Key重写hashCode和equals
    public static class Key {
        private String name;
        
        @Override
        public int hashCode() {
            return name.hashCode();
        }
        
        @Override
        public boolean equals(Object obj) {
            if (this == obj) return true;
            if (!(obj instanceof Key)) return false;
            return Objects.equals(name, ((Key) obj).name);
        }
    }
}
```

### 5.2 Code Review检查清单

```
□ 静态集合
  □ 是否有大小限制
  □ 是否有过期机制
  □ 是否定期清理

□ 监听器
  □ 注册后是否注销
  □ 是否使用弱引用

□ ThreadLocal
  □ 使用后是否remove
  □ 是否在finally中清理

□ 资源
  □ 是否使用try-with-resources
  □ 是否在finally中关闭

□ 内部类
  □ 是否应该使用静态内部类
  □ 匿名类是否持有外部类引用

□ HashMap
  □ Key是否重写hashCode和equals

```

## 六、总结

### 6.1 常见原因

```
1. 静态集合持有对象
2. 监听器未注销
3. ThreadLocal未清理
4. 资源未关闭
5. 内部类持有外部类引用
6. HashMap的Key未重写hashCode和equals
```

### 6.2 排查步骤

```
1. 确认内存泄漏（观察内存趋势）
2. 生成堆转储（jmap -dump）
3. 分析堆转储（MAT）
4. 查看引用链（Path to GC Roots）
5. 定位代码
6. 修复问题
7. 验证效果
```

### 6.3 预防措施

```
1. 遵循编码规范
2. Code Review
3. 使用静态代码分析工具
4. 定期进行内存分析
5. 监控内存使用趋势
```
