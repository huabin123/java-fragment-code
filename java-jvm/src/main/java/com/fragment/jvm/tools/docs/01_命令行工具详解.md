# 第一章：JVM 命令行工具详解

JDK 自带的诊断工具，无需额外安装，是排查线上问题的第一道武器。

```
核心工具速查表：

工具     作用                  一句话说明
jps      查看 Java 进程         找到 PID
jstat    查看 GC 统计           GC 频率、时间、内存使用率
jinfo    查看/修改 JVM 参数     运行时配置查询与动态调整
jmap     导出堆 dump            内存快照，分析内存泄漏
jstack   导出线程 dump          线程快照，分析死锁/CPU 高
jcmd     万能诊断工具           集成以上所有功能，Oracle 推荐
```

---

## 1.1 jps — 查看 Java 进程

```bash
# 基本用法
jps           # 列出 PID + 主类短名
jps -l        # PID + 主类全限定名（最常用）
jps -v        # PID + JVM 启动参数
jps -m        # PID + main 方法参数
jps -lvm      # 全部信息

# 实战
jps -l | grep "com.example.Application"   # 快速找到目标进程
```

```
工作原理：
  扫描 /tmp/hsperfdata_<username>/ 目录
  每个 Java 进程在此目录创建以 PID 命名的文件
  jps 读取这些文件获取进程信息

注意：
  - 只能看到当前用户的进程（root 可看所有）
  - 进程异常退出可能残留文件 → jps 显示已不存在的进程
```

## 1.2 jstat — GC 实时监控

```bash
# 基本语法
jstat [option] <PID> [interval_ms] [count]

# 最常用：GC 使用率百分比（推荐）
jstat -gcutil <PID> 1000 10    # 每秒 1 次，共 10 次

# 输出示例
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
  0.00  76.19  35.79  50.00  96.54  90.00    16    0.242     2    0.068    0.310

# 字段含义（百分比）
S0/S1  Survivor 0/1 区使用率
E      Eden 区使用率
O      老年代使用率
M      Metaspace 使用率
YGC    Young GC 次数      YGCT  Young GC 累计耗时（秒）
FGC    Full GC 次数       FGCT  Full GC 累计耗时（秒）
GCT    GC 总耗时
```

```bash
# GC 容量（KB 绝对值）
jstat -gc <PID> 1000

# 字段：S0C/S1C/S0U/S1U/EC/EU/OC/OU/MC/MU（C=容量, U=使用量）

# 新生代详情
jstat -gcnew <PID>       # 含 TT（晋升阈值）/ MTT（最大阈值）/ DSS（期望 Survivor）

# 编译统计
jstat -compiler <PID>    # JIT 编译任务数、失败数、耗时
```

### GC 监控判断标准

```
指标             正常                    需关注
YGC 频率         几秒~几分钟一次         每秒多次
FGC 频率         几小时~几天一次         几分钟一次
O（老年代占比）   GC 后有回落             持续增长不回落 → 可能泄漏
GCT 占比          <1% 优秀 / 1-3% 良好   >5% 需优化 / >10% 严重
```

```bash
# 技巧：输出到文件
jstat -gcutil <PID> 1000 > gc.log

# 技巧：格式化
jstat -gcutil <PID> 1000 | awk '{printf "Eden: %.2f%%, Old: %.2f%%\n", $3, $4}'

# 技巧：监控多个进程
for pid in $(jps -q); do echo "PID: $pid"; jstat -gcutil $pid; echo "---"; done
```

## 1.3 jinfo — 查看与修改 JVM 参数

```bash
# 查看特定参数
jinfo -flag MaxHeapSize <PID>        # -XX:MaxHeapSize=2147483648
jinfo -flag UseG1GC <PID>            # -XX:+UseG1GC

# 查看所有 JVM 参数
jinfo -flags <PID>

# 查看系统属性（System.getProperty）
jinfo -sysprops <PID>

# 动态修改（仅支持 manageable 参数）
jinfo -flag +PrintGC <PID>           # 开启 GC 日志
jinfo -flag -PrintGC <PID>           # 关闭 GC 日志
jinfo -flag +HeapDumpOnOutOfMemoryError <PID>
```

```bash
# 查看哪些参数可动态修改
java -XX:+PrintFlagsFinal -version | grep manageable

# 常见可动态修改参数
PrintGC / PrintGCDetails / PrintGCTimeStamps
HeapDumpOnOutOfMemoryError / HeapDumpPath
```

## 1.4 jmap — 堆内存分析

```bash
# 生成堆 dump（⚠️ 触发 Full GC，应用暂停）
jmap -dump:live,format=b,file=/tmp/heap.hprof <PID>

# 查看堆内存概况
jmap -heap <PID>

# 对象统计（按内存占用排序）
jmap -histo <PID> | head -20         # 不触发 GC
jmap -histo:live <PID> | head -20    # 触发 Full GC，只统计存活对象
```

### 对象统计输出解读

```
 num     #instances         #bytes  class name
----------------------------------------------
   1:         45678        3654240  [C            ← char[]
   2:         12345        2469000  java.lang.String
   3:          8901        1780200  [B            ← byte[]
   4:           987         197400  com.example.MyClass  ← 关注业务类

分析要点：
  实例数异常多 → 可能泄漏
  业务类占用大 → 检查是否合理
  定期对比 → 观察增长趋势
```

### ⚠️ 注意事项

```
1. jmap -dump 会触发 Full GC + STW → 生产低峰期使用
2. dump 文件大小 ≈ 堆内存大小 → 确保磁盘空间
3. 需要与 Java 进程相同用户或 root 权限
4. 推荐：启动时加 -XX:+HeapDumpOnOutOfMemoryError 自动 dump
```

## 1.5 jstack — 线程分析

```bash
# 基本用法
jstack <PID> > thread_dump.txt

# 包含锁信息
jstack -l <PID>

# 进程挂起时强制输出
jstack -F <PID>
```

### 线程状态速查

```
状态             含义                    常见场景
RUNNABLE         运行中或等待 CPU         正常执行 / CPU 高
BLOCKED          等待获取 synchronized    锁竞争
WAITING          无限期等待               Object.wait() / LockSupport.park()
TIMED_WAITING    有时限等待               Thread.sleep() / wait(timeout)
```

### 实战场景

#### CPU 飙高排查

```bash
# 1. 找到 CPU 高的线程
top -Hp <PID>          # Linux（macOS 无此选项，用 Activity Monitor）

# 2. 线程 ID 转 16 进制
printf "%x\n" <ThreadID>    # 例：1234 → 4d2

# 3. 在 jstack 输出中查找
jstack <PID> | grep "nid=0x4d2" -A 30

# 4. 分析该线程在执行什么代码
```

#### 死锁检测

```bash
jstack <PID> | grep -A 20 "Found one Java-level deadlock"

# 输出示例：
Found one Java-level deadlock:
"Thread-1":
  waiting to lock <0x00000000d5f10000> (a java.lang.Object),
  which is held by "Thread-2"
"Thread-2":
  waiting to lock <0x00000000d5f10010> (a java.lang.Object),
  which is held by "Thread-1"
```

#### 线程状态统计

```bash
jstack <PID> | grep "java.lang.Thread.State" | sort | uniq -c

# 示例输出：
  50 java.lang.Thread.State: RUNNABLE
  30 java.lang.Thread.State: TIMED_WAITING (parking)
  20 java.lang.Thread.State: WAITING (parking)
   5 java.lang.Thread.State: BLOCKED (on object monitor)

# 大量 BLOCKED → 锁竞争
# 大量 WAITING → 线程池空闲或等待资源
# 大量 RUNNABLE → CPU 密集
```

## 1.6 jcmd — 万能诊断工具（推荐）

Oracle 推荐的新一代诊断工具，一个命令替代多个工具。

```bash
# 列出 Java 进程（替代 jps）
jcmd

# 查看可用命令
jcmd <PID> help

# VM 信息
jcmd <PID> VM.version
jcmd <PID> VM.flags            # 替代 jinfo -flags
jcmd <PID> VM.system_properties
jcmd <PID> VM.uptime
jcmd <PID> VM.command_line

# 线程（替代 jstack）
jcmd <PID> Thread.print
jcmd <PID> Thread.print -l     # 含锁信息

# GC（替代 jmap/jstat）
jcmd <PID> GC.heap_info
jcmd <PID> GC.class_histogram  # 替代 jmap -histo
jcmd <PID> GC.heap_dump /tmp/heap.hprof
jcmd <PID> GC.run              # 触发 GC

# 动态日志（JDK 9+）
jcmd <PID> VM.log what=gc output=/tmp/gc.log

# Native Memory（需启动参数 -XX:NativeMemoryTracking=summary）
jcmd <PID> VM.native_memory summary
```

## 1.7 工具组合：问题排查流程

```
发现问题
  ↓
jps -l → 确定进程 PID
  ↓
jinfo -flags → 查看 JVM 配置
  ↓
判断问题类型：
  ├─ CPU 高     → jstack（多次采样对比）
  ├─ 内存增长   → jstat -gcutil → jmap -histo → jmap -dump → MAT
  ├─ GC 频繁    → jstat -gc 持续监控
  ├─ 死锁       → jstack -l（或 jcmd Thread.print）
  └─ 综合诊断   → jcmd（推荐）
```

### CPU 飙高一键排查脚本

```bash
#!/bin/bash
PID=$1
DUMP_DIR="/tmp/cpu_diag_$(date +%Y%m%d_%H%M%S)"
mkdir -p ${DUMP_DIR}

echo "[1/3] Dumping threads..."
jstack ${PID} > ${DUMP_DIR}/threads.txt

echo "[2/3] Collecting GC stats..."
jstat -gcutil ${PID} 1000 5 > ${DUMP_DIR}/gc_stat.txt

echo "[3/3] Collecting JVM info..."
jinfo -flags ${PID} > ${DUMP_DIR}/jvm_flags.txt

echo "✅ Diagnosis saved: ${DUMP_DIR}"
```

### 内存问题一键排查脚本

```bash
#!/bin/bash
PID=$1
DUMP_DIR="/tmp/mem_diag_$(date +%Y%m%d_%H%M%S)"
mkdir -p ${DUMP_DIR}

echo "[1/4] Heap summary..."
jmap -heap ${PID} > ${DUMP_DIR}/heap_summary.txt

echo "[2/4] Object histogram (top 30)..."
jmap -histo ${PID} | head -30 > ${DUMP_DIR}/histo.txt

echo "[3/4] GC stats (10s)..."
jstat -gcutil ${PID} 1000 10 > ${DUMP_DIR}/gc_stat.txt

echo "[4/4] Heap dump..."
jmap -dump:live,format=b,file=${DUMP_DIR}/heap.hprof ${PID}

echo "✅ Diagnosis saved: ${DUMP_DIR}"
echo "Use MAT to analyze: ${DUMP_DIR}/heap.hprof"
```

## 1.8 性能影响评估

```
工具                性能影响    说明
jps                 极小       仅读取临时文件
jstat               小         轮询性能计数器
jinfo               小         读取 JVM 配置
jmap -heap          小         读取内存信息
jmap -histo         中         遍历堆（不触发 GC）
jmap -histo:live    大         触发 Full GC
jmap -dump          大         Full GC + STW + 写文件
jstack              小         快照线程栈
jcmd                小~大      取决于具体子命令
```

---

## 1.9 本章总结

- **jps 找进程，jstat 看 GC，jmap 导内存，jstack 导线程，jcmd 是万能**
- 生产环境优先使用 **jcmd**，功能全、影响小
- **jmap -dump** 会触发 Full GC，低峰期使用
- 启动时就加 `-XX:+HeapDumpOnOutOfMemoryError`，不依赖手动 jmap
- 排查脚本提前准备，出问题时一键保留现场
