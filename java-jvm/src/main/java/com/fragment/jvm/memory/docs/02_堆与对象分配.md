# 第二章：堆与对象分配

## 2.1 堆的内部结构

```
┌─────────────────────────────────────────────────────────────┐
│                           堆                                 │
│                                                             │
│  ┌──────────────────────────────┐  ┌─────────────────────┐ │
│  │           新生代（Young Gen）  │  │   老年代（Old Gen）  │ │
│  │                              │  │                     │ │
│  │  ┌──────┐ ┌───────┬───────┐ │  │   长期存活对象        │ │
│  │  │ Eden │ │  S0   │  S1   │ │  │   大对象直接进入      │ │
│  │  │ (80%)│ │(10%)  │(10%)  │ │  │                     │ │
│  │  └──────┘ └───────┴───────┘ │  └─────────────────────┘ │
│  └──────────────────────────────┘                          │
│                                                             │
│  默认比例：新生代:老年代 = 1:2（-XX:NewRatio=2）             │
└─────────────────────────────────────────────────────────────┘
```

**参数控制**：
```bash
-Xms512m           # 堆初始大小
-Xmx2g             # 堆最大大小（生产环境建议 -Xms == -Xmx，避免动态扩缩开销）
-Xmn256m           # 新生代大小
-XX:NewRatio=2     # 老年代:新生代 = 2:1
-XX:SurvivorRatio=8 # Eden:S0:S1 = 8:1:1
```

## 2.2 对象的分配流程

```
new 一个对象
  ↓
是否超过大对象阈值（-XX:PretenureSizeThreshold）？
  → 是：直接进入老年代
  → 否：
      ↓
    尝试 TLAB（线程本地分配缓冲区）分配（Eden 区的线程私有小区域）
      → 成功：分配完成
      → 失败（TLAB 满了）：
          ↓
        Eden 是否有足够空间？
          → 是：直接在 Eden 分配
          → 否：触发 Minor GC（YGC）
              ↓
            GC 后 Eden 是否够？
              → 是：分配到 Eden
              → 否：直接进入老年代
```

## 2.3 对象晋升老年代的条件

```java
// OOMDemo.java → demonstrateObjectPromotion() 可观察晋升过程

// 条件1：年龄阈值（-XX:MaxTenuringThreshold，默认15）
// 每次 Minor GC 存活，对象 age+1，达到阈值晋升

// 条件2：动态年龄判定
// Survivor 区中相同年龄对象总大小 > Survivor 区容量的 50%
// 则年龄 ≥ 该年龄的对象直接晋升（不必等到15）

// 条件3：大对象直接进入老年代
byte[] big = new byte[1024 * 1024 * 10]; // 10MB，超过阈值直接老年代

// 条件4：空间分配担保
// Minor GC 前检查老年代最大连续空间 > 新生代所有对象总大小
// 如果不够，先做 Full GC
```

## 2.4 TLAB：解决多线程分配竞争

```
问题：多线程同时在 Eden 分配对象，如果不同步会覆盖彼此的内存

方案：TLAB（Thread Local Allocation Buffer）
  每个线程在 Eden 中预先申请一小块私有区域（默认 Eden 的 1%）
  线程在自己的 TLAB 内分配，无需加锁
  TLAB 用完后再申请新的（加锁，但频率很低）

效果：绝大多数对象分配是无锁的，极大提升并发分配性能
```

## 2.5 对象的内存布局

`ObjectLayoutDemo.java` 演示了对象在堆中的实际内存布局：

```
Java 对象在堆中的结构（64位JVM，开启压缩指针）：

┌──────────────────────┐
│   对象头（Header）     │
│  Mark Word（8字节）   │  存储 hashCode、GC分代年龄、锁标志
│  类型指针（4字节）     │  指向方法区中的类元数据（压缩后4字节）
├──────────────────────┤
│   实例数据（Instance） │  实际的字段值（按类型对齐排列）
├──────────────────────┤
│   对齐填充（Padding）  │  保证对象大小是8字节的整数倍
└──────────────────────┘

最小对象大小：16字节（8+4+4字节对齐）
```

## 2.6 本章总结

- **堆结构**：Eden + S0 + S1（新生代）+ 老年代，默认比例 8:1:1:（约20）
- **分配优先 Eden**：先尝试 TLAB（无锁），再 Eden，不够则 Minor GC，最后老年代
- **晋升老年代**：年龄达到阈值 / 动态年龄 / 大对象 / 担保失败
- **TLAB**：每线程独立小区域，避免分配时加锁，大幅提升并发性能
- **对象内存布局**：对象头（Mark Word + 类型指针）+ 实例数据 + 对齐填充
