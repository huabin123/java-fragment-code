# 第五章：内存溢出分析与排查

## 5.1 四种 OOM 的根本原因

### Heap OOM（最常见）
```bash
java.lang.OutOfMemoryError: Java heap space
```
```java
// OOMDemo.java → demonstrateHeapOOM()
// 根本原因：存活对象总大小 > 堆最大容量
// 典型场景：
// 1. 内存泄漏（对象无法回收，越积越多）
// 2. 数据量超出预期（一次性 select * 查出百万行）
// 3. 堆设置过小（-Xmx 太小）
List<byte[]> list = new ArrayList<>();
while (true) list.add(new byte[1024 * 1024]);  // 每次 1MB，直到 OOM
```

### Metaspace OOM
```bash
java.lang.OutOfMemoryError: Metaspace
```
```java
// 动态生成大量类，类元数据撑爆元空间
// 典型场景：CGLib 动态代理、频繁热部署、JSP 动态编译
// 解决：-XX:MaxMetaspaceSize=256m 加上限；检查是否有类加载泄漏
```

### Direct Buffer OOM
```bash
java.lang.OutOfMemoryError: Direct buffer memory
```
```java
// NIO DirectByteBuffer 申请堆外内存失败
// 默认直接内存上限 = -Xmx（可用 -XX:MaxDirectMemorySize 单独控制）
ByteBuffer.allocateDirect(1024 * 1024 * 1024);  // 申请 1GB 直接内存
```

### StackOverflowError
```java
// 递归过深，栈帧数量超过虚拟机栈深度
// 解决：检查递归是否有终止条件；增大 -Xss（但会减少可创建线程数）
```

## 5.2 OOM 排查流程

```
1. 分析日志：确认 OOM 类型
   grep "OutOfMemoryError" gc.log application.log

2. 获取 Heap Dump
   # 方案1：JVM 参数，OOM 时自动 dump
   -XX:+HeapDumpOnOutOfMemoryError
   -XX:HeapDumpPath=/tmp/heap-dump.hprof

   # 方案2：手动 dump（适合内存持续增长但未 OOM 时）
   jmap -dump:format=b,file=/tmp/heap.hprof <PID>

3. 分析 Heap Dump
   工具：MAT（Eclipse Memory Analyzer）、VisualVM、JProfiler
   关注：
   - Retained Heap 最大的对象
   - 占比最高的类
   - 是否有大量重复对象（内存泄漏特征）

4. 定位代码
   MAT → Dominator Tree → 找到持有大量内存的根对象 → 追踪引用链
   → 找到代码中是谁在持有这个引用没有释放
```

## 5.3 常见内存泄漏模式

```java
// 模式1：静态集合持有对象
static List<Object> cache = new ArrayList<>();
// 对象加入 cache 后永远不移除 → 内存泄漏

// 模式2：ThreadLocal 忘记 remove
ThreadLocal<byte[]> local = new ThreadLocal<>();
local.set(new byte[1024 * 1024]);
// 线程池中的线程不会消亡，ThreadLocal 引用不释放 → 泄漏

// 模式3：监听器/回调注册后未注销
eventBus.register(this);
// 持有 this 的引用，this 无法被 GC 回收

// 模式4：equals/hashCode 不一致的 key 放入 HashMap
// key 对象放入后 hashCode 变化，导致找不到 → 无法移除 → 泄漏

// 模式5：非静态内部类持有外部类引用
// OOMDemo.java 中有具体演示
```

## 5.4 实用 JVM 诊断命令

```bash
# 查看堆使用情况
jstat -gcutil <PID> 1000   # 每秒输出一次 GC 统计

# 查看堆中对象分布（排名前20）
jmap -histo:live <PID> | head -20

# 查看 JVM 启动参数
jinfo -flags <PID>

# 查看线程栈（排查 StackOverflow）
jstack <PID> > thread-dump.txt

# 实时监控 GC（JDK 11+）
jhsdb jmap --heap --pid <PID>
```

## 5.5 生产应急：保留现场脚本

OOM 发生时，**先保留现场再重启**，否则证据丢失：

```bash
#!/bin/bash
# emergency-dump.sh — OOM 应急一键保留现场

PID=$(pgrep -f "your-app")
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
DUMP_DIR="/data/dumps/${TIMESTAMP}"
mkdir -p ${DUMP_DIR}

# 1. 堆 dump（最重要，用于 MAT 分析）
echo "[1/5] Dumping heap..."
jmap -dump:live,format=b,file=${DUMP_DIR}/heap.hprof ${PID}

# 2. 线程栈（排查死锁/CPU 高）
echo "[2/5] Dumping threads..."
jstack -l ${PID} > ${DUMP_DIR}/threads.txt

# 3. GC 实时统计（连续 10 秒）
echo "[3/5] Collecting GC stats..."
jstat -gcutil ${PID} 1000 10 > ${DUMP_DIR}/gc_stat.txt

# 4. JVM 参数和系统属性
echo "[4/5] Collecting JVM info..."
jinfo ${PID} > ${DUMP_DIR}/jvm_info.txt

# 5. 系统资源快照
echo "[5/5] Collecting system info..."
top -l 1 > ${DUMP_DIR}/top.txt       # macOS（Linux 用 top -b -n 1）
vm_stat > ${DUMP_DIR}/memory.txt      # macOS（Linux 用 free -h）

echo "✅ Dump completed: ${DUMP_DIR}"
echo "Now safe to restart."
```

```
⚠️ 生产必备：启动时就加自动 dump 参数，不依赖手动脚本

-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/dumps/
-XX:OnOutOfMemoryError="sh /opt/scripts/emergency-dump.sh"
```

## 5.6 MAT 分析实战

### MAT 核心视图

```
打开 heap.hprof 后，MAT 提供以下核心分析视图：

1. Leak Suspects Report（泄漏嫌疑报告）
   → 自动分析，列出最可能的泄漏点
   → 第一步看这个，80% 的情况能直接定位

2. Dominator Tree（支配树）
   → 按对象"支配"关系排列（A 支配 B = 回收 A 必然回收 B）
   → 找到占内存最多的对象 → 顺着引用链找持有者

3. Histogram（对象直方图）
   → 按类统计实例数和占用空间
   → 哪个类的实例最多？占用最大？

4. Path to GC Roots（到 GC Root 的路径）
   → 对可疑对象右键 → Path to GC Roots → exclude weak references
   → 找到谁在持有它，为什么不释放
```

### 典型排查流程

```
步骤1：打开 heap.hprof → 看 Leak Suspects
        "Problem Suspect 1: Thread main keeps local variable
         java.util.ArrayList with 500,000 entries (800MB)"
                ↓
步骤2：打开 Dominator Tree → 按 Retained Heap 降序
        发现 ArrayList 占 800MB → 展开 → 里面是 50 万个 UserDTO
                ↓
步骤3：右键 ArrayList → Path to GC Roots → exclude weak refs
        GC Root → Thread "main" → CacheManager.userCache → ArrayList
                ↓
步骤4：定位代码
        CacheManager.userCache 只加不删 → 内存泄漏
                ↓
步骤5：修复
        改为 LRU 缓存（如 Caffeine），设置最大容量
```

## 5.7 本章总结

- **Heap OOM**：存活对象太多，内存泄漏或数据量过大；`-HeapDumpOnOutOfMemoryError` + MAT 分析
- **Metaspace OOM**：类加载泄漏（CGLib 无缓存、热部署）；`-XX:MaxMetaspaceSize` 加限制
- **Direct OOM**：NIO 堆外内存超限；`-XX:MaxDirectMemorySize` 控制
- **SOE**：无限递归；检查终止条件，或 `-Xss` 增大栈
- **排查三步**：看日志确类型 → dump 快照 → MAT 分析引用链
