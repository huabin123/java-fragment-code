# 第五章：内存溢出分析与排查

## 5.1 四种 OOM 的根本原因

### Heap OOM（最常见）
```bash
java.lang.OutOfMemoryError: Java heap space
```
```java
// OOMDemo.java → demonstrateHeapOOM()
// 根本原因：存活对象总大小 > 堆最大容量
// 典型场景：
// 1. 内存泄漏（对象无法回收，越积越多）
// 2. 数据量超出预期（一次性 select * 查出百万行）
// 3. 堆设置过小（-Xmx 太小）
List<byte[]> list = new ArrayList<>();
while (true) list.add(new byte[1024 * 1024]);  // 每次 1MB，直到 OOM
```

### Metaspace OOM
```bash
java.lang.OutOfMemoryError: Metaspace
```
```java
// 动态生成大量类，类元数据撑爆元空间
// 典型场景：CGLib 动态代理、频繁热部署、JSP 动态编译
// 解决：-XX:MaxMetaspaceSize=256m 加上限；检查是否有类加载泄漏
```

### Direct Buffer OOM
```bash
java.lang.OutOfMemoryError: Direct buffer memory
```
```java
// NIO DirectByteBuffer 申请堆外内存失败
// 默认直接内存上限 = -Xmx（可用 -XX:MaxDirectMemorySize 单独控制）
ByteBuffer.allocateDirect(1024 * 1024 * 1024);  // 申请 1GB 直接内存
```

### StackOverflowError
```java
// 递归过深，栈帧数量超过虚拟机栈深度
// 解决：检查递归是否有终止条件；增大 -Xss（但会减少可创建线程数）
```

## 5.2 OOM 排查流程

```
1. 分析日志：确认 OOM 类型
   grep "OutOfMemoryError" gc.log application.log

2. 获取 Heap Dump
   # 方案1：JVM 参数，OOM 时自动 dump
   -XX:+HeapDumpOnOutOfMemoryError
   -XX:HeapDumpPath=/tmp/heap-dump.hprof

   # 方案2：手动 dump（适合内存持续增长但未 OOM 时）
   jmap -dump:format=b,file=/tmp/heap.hprof <PID>

3. 分析 Heap Dump
   工具：MAT（Eclipse Memory Analyzer）、VisualVM、JProfiler
   关注：
   - Retained Heap 最大的对象
   - 占比最高的类
   - 是否有大量重复对象（内存泄漏特征）

4. 定位代码
   MAT → Dominator Tree → 找到持有大量内存的根对象 → 追踪引用链
   → 找到代码中是谁在持有这个引用没有释放
```

## 5.3 常见内存泄漏模式

```java
// 模式1：静态集合持有对象
static List<Object> cache = new ArrayList<>();
// 对象加入 cache 后永远不移除 → 内存泄漏

// 模式2：ThreadLocal 忘记 remove
ThreadLocal<byte[]> local = new ThreadLocal<>();
local.set(new byte[1024 * 1024]);
// 线程池中的线程不会消亡，ThreadLocal 引用不释放 → 泄漏

// 模式3：监听器/回调注册后未注销
eventBus.register(this);
// 持有 this 的引用，this 无法被 GC 回收

// 模式4：equals/hashCode 不一致的 key 放入 HashMap
// key 对象放入后 hashCode 变化，导致找不到 → 无法移除 → 泄漏

// 模式5：非静态内部类持有外部类引用
// OOMDemo.java 中有具体演示
```

## 5.4 实用 JVM 诊断命令

```bash
# 查看堆使用情况
jstat -gcutil <PID> 1000   # 每秒输出一次 GC 统计

# 查看堆中对象分布（排名前20）
jmap -histo:live <PID> | head -20

# 查看 JVM 启动参数
jinfo -flags <PID>

# 查看线程栈（排查 StackOverflow）
jstack <PID> > thread-dump.txt

# 实时监控 GC（JDK 11+）
jhsdb jmap --heap --pid <PID>
```

## 5.5 本章总结

- **Heap OOM**：存活对象太多，内存泄漏或数据量过大；`-HeapDumpOnOutOfMemoryError` + MAT 分析
- **Metaspace OOM**：类加载泄漏（CGLib 无缓存、热部署）；`-XX:MaxMetaspaceSize` 加限制
- **Direct OOM**：NIO 堆外内存超限；`-XX:MaxDirectMemorySize` 控制
- **SOE**：无限递归；检查终止条件，或 `-Xss` 增大栈
- **排查三步**：看日志确类型 → dump 快照 → MAT 分析引用链
