# 第六章：直接内存详解

> 直接内存（Direct Memory）也称堆外内存（Off-Heap Memory），两者是同一概念的不同叫法。

## 6.1 为什么需要直接内存？

传统 I/O 使用堆内存时，数据需要多一次拷贝：

```
传统 I/O 流程（堆内存 HeapByteBuffer）：

磁盘/网卡
    ↓ DMA 拷贝
内核缓冲区（Kernel Buffer）
    ↓ CPU 拷贝 ← 多出来的一次拷贝
JVM 堆内存（byte[]）
    ↓
应用程序处理

为什么需要额外拷贝？
  JVM 堆内存会被 GC 移动（复制算法/整理算法）
  → 对象地址随时可能变化
  → OS 做 DMA 时不能直接写入堆内存（地址不稳定）
  → 必须先写到内核缓冲区，再拷贝到堆内存

NIO 流程（直接内存 DirectByteBuffer）：

磁盘/网卡
    ↓ DMA 拷贝
直接内存（Native Memory）── 地址固定，不受 GC 移动
    ↓ 应用直接访问（零拷贝）
应用程序处理

优势：
  ✅ 减少一次 CPU 拷贝
  ✅ 地址稳定，OS 可直接 DMA 到此
  ✅ I/O 性能显著提升（大数据量时提升 30%-50%）
```

## 6.2 JVM 进程内存全景

```
┌──────────────────────────────────────────────────────┐
│                   JVM 进程内存                        │
│                                                      │
│  ┌────────────────────────────────────────┐          │
│  │           JVM 堆（Heap）               │          │
│  │  -Xms / -Xmx 控制                     │          │
│  │  受 GC 管理，对象会被移动              │          │
│  │  分配快（TLAB 指针碰撞，纳秒级）       │          │
│  └────────────────────────────────────────┘          │
│                                                      │
│  ┌────────────────────────────────────────┐          │
│  │       直接内存（Direct Memory）         │          │
│  │  -XX:MaxDirectMemorySize 控制          │          │
│  │  不受 GC 直接管理                      │          │
│  │  地址固定，适合 I/O                    │          │
│  │  分配慢（malloc，微秒级）              │          │
│  └────────────────────────────────────────┘          │
│                                                      │
│  ┌────────────────────────────────────────┐          │
│  │       元空间（Metaspace）              │          │
│  │  -XX:MaxMetaspaceSize 控制             │          │
│  │  存储类元数据                          │          │
│  └────────────────────────────────────────┘          │
│                                                      │
│  ┌────────────────────────────────────────┐          │
│  │       其他本地内存                     │          │
│  │  线程栈（-Xss）、JNI、JIT CodeCache   │          │
│  └────────────────────────────────────────┘          │
│                                                      │
└──────────────────────────────────────────────────────┘

⚠️ 常见误区：-Xmx 只限制堆大小，不限制直接内存
   一个 -Xmx=4g 的进程，实际占用的物理内存可能远超 4GB
   → 堆 4GB + 直接内存 + 元空间 + 线程栈 × N + CodeCache + ...
```

## 6.3 DirectByteBuffer 分配原理

```java
// 应用层调用
ByteBuffer buf = ByteBuffer.allocateDirect(1024);  // 分配 1KB 直接内存

// JDK 内部流程（DirectByteBuffer 构造函数简化）：
DirectByteBuffer(int cap) {
    // ① 记账：检查直接内存是否超过 MaxDirectMemorySize
    Bits.reserveMemory(size, cap);

    // ② 分配：调用 Unsafe.allocateMemory → 底层 malloc
    long base = unsafe.allocateMemory(size);

    // ③ 清零：初始化内存为 0
    unsafe.setMemory(base, size, (byte) 0);

    // ④ 注册 Cleaner：绑定释放回调（GC 时自动触发）
    cleaner = Cleaner.create(this, new Deallocator(base, size, cap));
}
```

```
堆内分配 vs 直接内存分配：

                    堆内（HeapByteBuffer）   直接（DirectByteBuffer）
─────────────────────────────────────────────────────────────────
底层实现             new byte[cap]            Unsafe.allocateMemory(cap)
                                              → malloc()
分配速度             快（纳秒级，TLAB）       慢（微秒级，系统调用）
I/O 性能             慢（多一次拷贝）         快（零拷贝）
GC 管理              是（会被移动）           否（地址固定）
对象头开销           有（16 字节）            无
适合场景             频繁创建的小 Buffer      长期复用的 I/O Buffer
```

## 6.4 直接内存的释放机制

直接内存不受 GC 直接回收，靠 **Cleaner** 机制间接释放：

```
释放流程：

DirectByteBuffer 对象（在堆上，很小，只有地址和 Cleaner 引用）
        │
        │ 没有引用指向它
        ↓
GC 发现 DirectByteBuffer 对象不可达
        │
        ↓
触发 Cleaner.clean()
        │
        ↓
执行 Deallocator.run()
        │
        ↓
unsafe.freeMemory(address)  → 底层 free()
        │
        ↓
直接内存释放，Bits 记账减少
```

```java
// Cleaner 回调（JDK 源码简化）
private static class Deallocator implements Runnable {
    private long address;
    private long size;
    private int capacity;

    public void run() {
        if (address == 0) return;
        unsafe.freeMemory(address);   // 释放本地内存
        address = 0;
        Bits.unreserveMemory(size, capacity);  // 更新记账
    }
}

// 手动释放（不等 GC，立即回收）
// 生产环境推荐在确定不再使用时手动清理
public static void cleanDirectBuffer(ByteBuffer buffer) {
    if (buffer.isDirect()) {
        ((DirectBuffer) buffer).cleaner().clean();
    }
}
```

```
⚠️ 隐患：Cleaner 依赖 GC 触发
  DirectByteBuffer 对象本身很小（几十字节，在堆上）
  → 可能很久不会触发 GC
  → 堆上对象没被回收 → Cleaner 不执行 → 直接内存不释放
  → 直接内存耗尽 → OOM: Direct buffer memory

  这就是为什么 Netty 不用 JDK 的 Cleaner，而是自己实现引用计数（ReferenceCounted）
```

## 6.5 直接内存泄漏排查

### 症状

```
Direct Memory OOM 的典型表现：

java.lang.OutOfMemoryError: Direct buffer memory

触发条件：
  已分配的直接内存 ≥ -XX:MaxDirectMemorySize
  （默认值 ≈ -Xmx 的值）
```

### 排查步骤

```bash
# ① 确认直接内存使用量
# 方式1：JMX（代码中）
java.nio.Bits.reservedMemory  # 已分配的直接内存总量

# 方式2：jcmd
jcmd <pid> VM.native_memory summary  # 需要启动时加 -XX:NativeMemoryTracking=summary

# 方式3：Arthas
memory  # 查看各区域内存使用

# ② 定位泄漏点
# 方式1：-verbose:gc 观察 DirectByteBuffer 回收情况
# 方式2：MAT 分析 heap dump
#   搜索 java.nio.DirectByteBuffer 实例
#   查看哪些对象持有 DirectByteBuffer 引用 → 为什么不释放

# ③ 常见泄漏原因
# 1. ByteBuffer 被集合/缓存持有，无法 GC
# 2. 忘记关闭 FileChannel（内部持有 MappedByteBuffer）
# 3. Netty 的 ByteBuf 忘记 release()
# 4. 线程池中的 ThreadLocal 持有 DirectByteBuffer
```

## 6.6 最佳实践

### 何时用直接内存

| 场景 | 用堆内存 | 用直接内存 |
|------|---------|-----------|
| 频繁创建/销毁的小 Buffer | ✅ | ❌（分配慢）|
| 长期复用的 I/O Buffer | ❌ | ✅（零拷贝）|
| 网络服务（Netty/NIO） | ❌ | ✅（减少拷贝）|
| 大文件传输 | ❌ | ✅（FileChannel + MappedByteBuffer）|
| 纯计算（无 I/O） | ✅ | ❌（无意义）|

### 关键参数

```bash
# 限制直接内存大小（必须设！否则默认等于 -Xmx，可能耗尽物理内存）
-XX:MaxDirectMemorySize=512m

# 启用 NMT（Native Memory Tracking）监控本地内存分配
-XX:NativeMemoryTracking=summary   # summary 或 detail
# 查看：jcmd <pid> VM.native_memory summary
```

### 编码建议

```java
// ✅ 分配一次，长期复用（池化）
private static final ByteBuffer DIRECT_BUF = ByteBuffer.allocateDirect(64 * 1024);

public void handleIO(SocketChannel channel) {
    DIRECT_BUF.clear();
    channel.read(DIRECT_BUF);
    DIRECT_BUF.flip();
    // 处理数据...
}

// ✅ 用完显式释放（不等 GC）
ByteBuffer tmp = ByteBuffer.allocateDirect(1024);
try {
    // 使用 tmp
} finally {
    ((DirectBuffer) tmp).cleaner().clean();  // 立即释放
}

// ✅ Netty 场景：用 ByteBufAllocator，严格 release
ByteBuf buf = ctx.alloc().directBuffer(1024);
try {
    // 使用 buf
} finally {
    buf.release();  // 引用计数 -1，归零时释放
}

// ❌ 不要在循环中反复 allocateDirect
for (int i = 0; i < 10000; i++) {
    ByteBuffer.allocateDirect(1024);  // 分配慢 + 可能泄漏
}
```

## 6.7 本章总结

- **为什么需要直接内存**：堆内存地址不稳定（GC 移动），I/O 需多一次拷贝；直接内存地址固定，支持零拷贝
- **分配原理**：`ByteBuffer.allocateDirect()` → `Unsafe.allocateMemory()` → `malloc()`
- **释放机制**：Cleaner 绑定到 DirectByteBuffer，GC 回收堆上对象时触发 `free()`；但 GC 不及时可能导致直接内存泄漏
- **核心参数**：`-XX:MaxDirectMemorySize` 必须设置，否则默认等于 `-Xmx`
- **使用原则**：分配一次长期复用（池化），不在循环中反复创建；用完显式释放或依赖框架引用计数（Netty）
- **泄漏排查**：`NativeMemoryTracking` + MAT 搜索 DirectByteBuffer 实例 + 检查引用链
