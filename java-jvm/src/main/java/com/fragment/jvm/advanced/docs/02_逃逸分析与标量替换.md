# 第二章：逃逸分析与标量替换

## 2.1 什么是逃逸分析？

逃逸分析（Escape Analysis）是 JIT 在编译时分析对象的**作用域**：

```
一个对象"逃逸"意味着它的引用被传递到方法外部（或被其他线程访问）

不逃逸（方法内逃逸分析）：
  对象只在方法内部使用，方法结束后不再被引用
  → JIT 可以做优化：栈上分配、标量替换

全局逃逸：
  对象赋给了静态变量，或返回给调用者，或传入其他方法且被持有
  → 只能分配在堆上，无法优化

参数逃逸：
  对象作为参数传给其他方法，但被调用方法不会保存它
  → 可以做部分优化
```

## 2.2 三种基于逃逸分析的优化

`EscapeAnalysisDemo.java`（memory 模块）演示了这三种优化：

### 优化1：栈上分配（Stack Allocation）

```java
// 分析：point 只在方法内使用，不逃逸
public static void process() {
    Point point = new Point(1, 2);  // 不逃逸
    System.out.println(point.x + point.y);
}  // 方法结束，point 不再使用

// JIT 优化后：point 的内存分配在栈上（而非堆）
// 方法返回时随栈帧一起回收，无需 GC 介入
// 效果：大量减少 GC 压力，提高分配速度

// 验证方式：关闭逃逸分析，对比 GC 频率
// -XX:-DoEscapeAnalysis  关闭逃逸分析（JDK 6u23+ 默认开启）
```

### 优化2：标量替换（Scalar Replacement）

```java
// 标量：不可再分解的数据（int, long, boolean 等基本类型）
// 聚合量：可以分解的数据（对象）

// 原始代码：
public static int compute() {
    Point p = new Point(3, 4);  // 不逃逸
    return p.x + p.y;
}

// 标量替换后（JIT 等价于）：
public static int compute() {
    int px = 3;  // 直接用基本类型存储，不创建 Point 对象
    int py = 4;
    return px + py;
}

// 效果：完全消除对象创建，字段直接存入寄存器或栈
// 这是比栈上分配更进一步的优化
```

### 优化3：同步消除（Lock Elimination）

```java
// 如果对象不逃逸，那对它加锁没有意义（只有一个线程能访问它）
// JIT 会直接消除这个锁

// OptimizationDemo.java → demonstrateLockElimination()
public String buildString() {
    StringBuffer sb = new StringBuffer();  // StringBuffer 是线程安全的（有 synchronized）
    sb.append("hello");
    sb.append(" world");
    return sb.toString();
    // sb 不逃逸 → JIT 消除所有 synchronized → 性能等同于 StringBuilder
}
```

## 2.3 逃逸分析的局限

```java
// 逃逸分析并非万能：

// 1. 大对象不做栈上分配（超过 JVM 阈值）
byte[] big = new byte[1024 * 1024];  // 即使不逃逸，也在堆上分配

// 2. 对象作为参数传出 → 逃逸，无法优化
public Point getPoint() {
    Point p = new Point(1, 2);
    return p;  // 逃逸！返回给调用者
}

// 3. 虚方法调用阻碍内联 → 间接阻碍逃逸分析
// 逃逸分析依赖方法内联，内联失败则无法正确判断逃逸状态

// 验证逃逸分析效果的 JVM 参数：
// -XX:+DoEscapeAnalysis      开启（默认）
// -XX:-DoEscapeAnalysis      关闭，用于对比测试
// -XX:+PrintEscapeAnalysis   打印逃逸分析结果（需 -XX:+UnlockDiagnosticVMOptions）
// -XX:+EliminateAllocations  开启标量替换（默认开启）
// -XX:+EliminateLocks        开启锁消除（默认开启）
```

## 2.4 对编码的指导意义

```java
// ✅ 局部对象尽量不"逃逸"，让 JIT 有机会优化
// ✅ 方法内的临时对象（Builder、DTO 组装）不要暴露引用
// ✅ 小对象、不逃逸的对象不需要刻意池化（JIT 会优化分配）

// ❌ 不要为了"减少对象创建"手动复用局部对象（反而阻碍 JIT 优化）
// 例如：在循环里重复使用同一个 StringBuilder 清空后复用
// → 不如每次 new StringBuilder，让 JIT 通过标量替换优化

// 实测：OptimizationDemo.java → compareObjectCreation() 验证了这一点
```

## 2.5 本章总结

- **逃逸分析**：判断对象作用域，不逃逸的对象可以做进一步优化
- **三种优化**：
  - **栈上分配**：对象分配在栈而非堆，方法结束自动回收，减少 GC 压力
  - **标量替换**：对象字段拆散为基本类型，完全消除对象创建
  - **锁消除**：不逃逸对象的同步块被去除（如局部 StringBuffer）
- **局限**：大对象、逃逸对象、虚方法内联失败时无法优化
- **编码建议**：局部临时对象让 JIT 自由优化，不要过早手动"优化"对象创建
