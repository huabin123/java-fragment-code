## 三、核心因果链：知识点之间的依赖关系

以下展示知识点之间"**因为 A → 所以需要 B**"的因果关系：

```
因为 Java 是跨平台的
  → 所以需要字节码作为中间表示 ─────────────────────── [advanced/03_字节码]
  → 所以需要类加载器把字节码加载进 JVM ─────────────── [classloader/01_类加载时机]

因为不同类库需要隔离
  → 所以需要双亲委派模型 ───────────────────────────── [classloader/02_双亲委派]
  → 打破双亲委派 → SPI / 热部署 / 模块化 ─────────── [classloader/03-04]

因为类被加载后需要存储
  → 类信息存入方法区（元空间）──────────────────────── [memory/04_方法区]
  → 对象实例存入堆 ────────────────────────────────── [memory/02_堆]
  → 方法调用信息存入虚拟机栈 ──────────────────────── [memory/03_栈]

因为堆是共享的，多线程分配对象有竞争
  → 所以需要 TLAB（线程私有分配缓冲区）───────────── [advanced/04_TLAB]

因为字节码解释执行太慢
  → 所以需要 JIT 把热点代码编译为机器码 ──────────── [advanced/01_JIT]
  → JIT 编译时做逃逸分析 → 栈上分配/标量替换 ────── [advanced/02_逃逸分析]
  → 栈上分配的对象不进堆 → 减少 GC 压力 ─────────── [gc/01 ← advanced/02]

因为堆内存有限，对象不断创建
  → 所以需要 GC 自动回收不可达对象 ────────────────── [gc/01_存活判定]
  → 不同算法适合不同场景（复制/整理/清除）────────── [gc/02_GC算法]
  → 不同收集器适合不同服务（Serial→G1→ZGC）─────── [gc/03_收集器详解]
  → GC 有触发时机（Eden满/老年代满/担保失败等）──── [gc/03_§3.2]

因为 GC 会 Stop-The-World
  → 所以需要调优减少 STW 时间 ─────────────────────── [gc/04_GC调优]
  → 需要 GC 日志分析 + 参数调整 + 压测验证 ─────── [gc/05_最佳实践]

因为线上会出问题
  → OOM → 堆转储分析 → MAT → 定位泄漏 ───────────── [memory/05_OOM排查]
  → CPU 飙高 → jstack → 线程分析 ──────────────────── [tuning/03_CPU排查]
  → 内存泄漏 → jstat + jmap → 堆分析 ─────────────── [tuning/04_内存泄漏]
  → 死锁 → jstack → 锁链分析 ─────────────────────── [tuning/05_死锁排查]
```

## 四、对象的一生（贯穿所有模块的主线）

```
                    对象从创建到回收的完整旅程
                    ═══════════════════════

  new MyObject()
       │
       │ ① 类是否已加载？
       │    否 → 触发类加载 ─────────────────── classloader/01_时机与过程
       │         加载→验证→准备→解析→初始化       classloader/02_双亲委派
       │         类信息存入方法区                  memory/04_方法区
       │    是 ↓
       │
       │ ② 计算对象大小（对象头 + 实例数据 + 对齐填充）
       │    ─────────────────────────────────── memory/02_堆与对象分配
       │
       │ ③ 分配内存（选择分配路径）
       │    ├─ 逃逸分析：不逃逸？ ──────────── advanced/02_逃逸分析
       │    │    是 → 标量替换（栈/寄存器）      不进堆，方法结束即回收
       │    │    否 ↓
       │    ├─ TLAB 有空间？ ───────────────── advanced/04_TLAB
       │    │    是 → 指针碰撞，O(1)，无锁
       │    │    否 ↓
       │    ├─ 大对象？ ────────────────────── gc/03_§3.2（GC 触发时机）
       │    │    是 → 直接进老年代
       │    │    否 → Eden 区 CAS 分配
       │    │         Eden 满 → 触发 Minor GC
       │    │
       │ ④ 初始化
       │    设置对象头（MarkWord + 类型指针）
       │    执行构造方法 <init>
       │    ─────────────────────────────────── memory/02_堆与对象分配
       │
       │ ⑤ 使用阶段
       │    方法调用 → 虚拟机栈压栈帧 ─────────  memory/03_栈与栈帧
       │    热点方法 → JIT 编译优化 ───────────  advanced/01_JIT
       │    锁操作  → 偏向锁/轻量级锁 ─────────  （对象头 MarkWord 状态变化）
       │
       │ ⑥ 不可达（无引用指向该对象）
       │    可达性分析：从 GC Roots 遍历 ──────  gc/01_存活判定
       │    引用类型：强→软→弱→虚 ──────────────  gc/01_§引用类型
       │
       │ ⑦ 垃圾回收
       │    ├─ 在新生代（存活 < 15 次 GC）
       │    │    标记-复制算法 → Eden→S0→S1 ──  gc/02_GC算法
       │    │    Minor GC（STW 短）
       │    │
       │    ├─ 晋升到老年代（存活 ≥ 15 次 or 大对象）
       │    │    标记-清除/整理 ────────────────  gc/02_GC算法
       │    │    Major/Full GC（STW 长）
       │    │
       │    └─ 收集器执行回收 ─────────────────  gc/03_收集器详解
       │         Serial / CMS / G1 / ZGC        （不同收集器不同策略）
       │
       │ ⑧ 内存释放，空间可再次分配
       │
       ↓
     对象生命结束

  ❌ 如果始终不回收？→ OOM ────────────────────  memory/05_OOM排查
  ⚠️ GC 太频繁？→ 调优 ──────────────────────  gc/04_GC调优 + gc/05_最佳实践
  🔧 线上问题？→ 排查 ──────────────────────── tuning/ 模块
```

## 七、一句话串联

> **Java 代码** 经 `javac` 编译为 **字节码**[advanced/03]，由 **类加载器**[classloader] 通过双亲委派加载进 JVM，类信息存入 **方法区**[memory/04]，对象实例在 **堆** 的 **TLAB**[advanced/04] 中通过指针碰撞快速分配[memory/02]，方法调用压入 **虚拟机栈**[memory/03]；**JIT 编译器**[advanced/01] 对热点代码做 **逃逸分析**[advanced/02] 实现栈上分配和标量替换以减少堆压力；当对象不可达时，**GC**[gc/01] 使用标记-复制/整理算法[gc/02]，由 G1/ZGC 等收集器[gc/03] 在 Eden 满或老年代满时触发回收；出现 OOM 或性能问题时，通过 jstack/jmap/MAT 等工具排查[memory/05 + tuning]。
