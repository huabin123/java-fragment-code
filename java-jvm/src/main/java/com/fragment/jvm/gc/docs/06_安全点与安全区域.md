# 第六章：安全点与安全区域

## 6.1 为什么需要安全点？

GC 需要 Stop-The-World（STW），但**不能在任意位置停线程**：

```
为什么不能随意停止线程？

场景1：引用赋值执行到一半
  线程A：obj.field = newObj;
         ↑ 如果在此刻停线程 → 旧引用已断开，新引用未写入
         → GC 扫描时引用关系不一致 → newObj 可能被误回收

场景2：栈帧构建不完整
  线程B：正在调用方法，参数已压栈，返回地址未压栈
         → GC 无法正确遍历栈上的引用

场景3：寄存器中有对象引用
  线程C：mov eax, [obj]  // 引用在 CPU 寄存器中
         → GC 不知道 eax 里是对象引用还是普通数值

结论：
  必须在**引用关系明确、栈帧完整**的特定位置停线程
  这些位置就是 → 安全点（Safepoint）
```

安全点不仅服务于 GC，以下操作都需要所有线程到达安全点：

| STW 场景 | 说明 |
|---------|------|
| 垃圾回收（GC） | 标记/移动对象、更新引用 |
| 偏向锁撤销 | 撤销偏向锁，升级为轻量级锁 |
| 去优化（Deoptimization） | 回退 JIT 编译的代码，恢复解释执行 |
| Thread Dump | `jstack` 需要所有线程栈帧一致 |
| 类重定义（Class Redefinition） | Java Agent 热替换类 |

## 6.2 安全点的位置选择

```
安全点设置在哪里？

原则：选择"程序长时间执行"的位置，避免线程长时间到不了安全点

✅ 会设置安全点的位置：
  1. 方法调用（call 指令前）
     public void a() {
         doSomething();   // ← 安全点（调用前检查）
         doOther();       // ← 安全点
     }

  2. 循环的回边（backward branch）
     for (int i = 0; i < 1000000; i++) {
         // 循环体
     }   // ← 回边处设安全点（防止长循环不到安全点）

  3. 异常抛出/捕获点
     try { ... }
     catch (Exception e) { ... }  // ← 安全点

❌ 不设置安全点的位置：
  - 方法内的普通赋值/运算语句（太频繁，开销大）
  - JNI 代码内部（JVM 无法控制 native 代码）

⚠️ 经典陷阱：可数循环（counted loop）
  JIT 编译的可数循环（int 循环变量 + 固定上界）默认不插入安全点！

  // 这个循环不会被插入安全点（JIT 认为可数循环会很快结束）
  for (int i = 0; i < hugeNumber; i++) {
      // 纯计算，无方法调用
      sum += array[i];
  }
  // → 可能导致其他线程等这个线程到达安全点，STW 时间异常长

  // JDK 解决方案：
  // JDK 17+：-XX:+UseCountedLoopSafepoints（默认关闭，可手动开启）
  // 或者改用 long 循环变量（long 不被视为可数循环）
```

## 6.3 线程如何到达安全点？

HotSpot 使用**主动式中断**（Voluntary Suspension），不强制停线程：

```
主动式中断流程：

1. JVM 设置全局安全点标志（polling page 设为不可读）

2. 每个线程在安全点位置执行轮询（poll）：
   test [polling_page]   // 读取 polling page
   ├─ 正常（页可读）→ 继续执行
   └─ 触发 SIGSEGV  → 信号处理器 → 线程挂起

3. 所有线程到达安全点 → JVM 执行 STW 操作（GC / 偏向锁撤销等）

4. 操作完成 → 恢复 polling page → 唤醒所有线程

为什么用内存页保护而不是 if 判断？
  if (safepoint) { suspend(); }   // 每次检查需要一次条件跳转
  test [polling_page]              // 只是一次内存读，无分支，CPU 更快
  → 正常情况下 polling page 可读，test 指令几乎零开销
  → 需要 STW 时，把页标记为不可读，触发信号 → 线程挂起
```

```
不同线程状态到达安全点的方式：

┌────────────────────┬──────────────────────────────────────┐
│ 线程状态           │ 如何到达安全点                        │
├────────────────────┼──────────────────────────────────────┤
│ 解释执行中         │ 每条字节码之间检查安全点标志           │
│ JIT 编译代码执行中 │ 在编译代码的安全点位置插入 poll 指令    │
│ JNI 调用中         │ 返回 Java 代码时检查                  │
│ 阻塞中（sleep/wait）│ 不需要到安全点 → 进入安全区域（见 6.4）│
│ 已在安全点         │ 无需操作                              │
└────────────────────┴──────────────────────────────────────┘
```

## 6.4 安全区域（Safe Region）

安全点有一个假设：线程在**运行中**，会主动轮询到安全点。但 sleep / blocked / waiting 的线程无法主动走到安全点：

```
安全区域（Safe Region）= 一段引用关系不会变化的代码区间

线程进入 sleep/wait/blocked 状态时：
  1. 线程标记自己"已进入安全区域"
  2. JVM 发起 STW 时，看到该线程在安全区域 → 不需要等它
  3. GC 正常执行

线程从 sleep/wait/blocked 恢复时：
  1. 检查 JVM 是否在 STW 中
  2. 如果是 → 等待 STW 结束
  3. 如果否 → 正常恢复执行

时序图：

线程状态：  [运行] ──→ [sleep] ──────────────────────→ [唤醒]
安全区域：           ├── 标记进入 Safe Region ──┤
                     │                          │
JVM 发起 STW：       │     ← 看到线程在 SR →    │
                     │        不用等它           │
                     │                          ├── 检查 STW 是否结束？
                     │                          │   是 → 恢复
                     │                          │   否 → 继续等待
```

## 6.5 安全点导致的性能问题

### TTSP（Time To SafePoint）

```
TTSP = 从 JVM 发起 STW 请求到所有线程到达安全点的时间

正常情况：TTSP < 1ms
异常情况：某个线程迟迟到不了安全点 → TTSP 飙高 → 所有线程都在等它

常见导致 TTSP 过长的原因：

1. 可数循环无安全点（最常见）
   for (int i = 0; i < 10_000_000; i++) {
       sum += data[i];   // JIT 编译后无安全点，其他线程等到死
   }

2. 大内存块拷贝
   System.arraycopy(src, 0, dst, 0, hugeLength);
   // native 方法内部无安全点

3. 大量线程同时 JNI 调用
   JNI 返回 Java 时才检查安全点
```

### 监控安全点

```bash
# JDK 8：打印安全点统计
java -XX:+PrintSafepointStatistics -XX:PrintSafepointStatisticsCount=1 -jar app.jar

# 输出示例：
#          vmop                    [threads: total initially_running wait_to_block]
# 4.020: G1IncCollectionPause     [     20          1              0    ]
# ↑ 时间   ↑ STW 操作              ↑ 总线程数  ↑ 运行中   ↑ 等待到达安全点的线程

# JDK 9+：使用统一日志
java -Xlog:safepoint=info -jar app.jar

# 关注指标：
#   spin + block 时间 = TTSP（应 < 1ms）
#   如果 TTSP 经常 > 10ms → 排查是否有可数循环或大 arraycopy

# JDK 17+：开启可数循环安全点
java -XX:+UseCountedLoopSafepoints -jar app.jar
```

## 6.6 安全点总结

```
核心概念串联：

  GC 需要 STW
      ↓
  不能随意停线程（引用不一致/栈帧不完整）
      ↓
  必须在安全点停（引用关系明确的特定位置）
      ↓
  安全点位置：方法调用、循环回边、异常处理
      ↓
  线程主动轮询（polling page，零开销）
      ↓
  sleep/blocked 线程 → 安全区域（不用等它）
      ↓
  所有线程就绪 → 执行 STW 操作 → 恢复
```

| 概念 | 一句话 |
|------|--------|
| **安全点** | 程序中引用关系明确、可安全暂停线程的特定位置 |
| **安全区域** | 引用关系不变的代码区间，sleep/blocked 线程通过它参与 STW |
| **主动式中断** | JVM 不强停线程，通过 polling page 让线程自行挂起 |
| **TTSP** | 所有线程到达安全点的耗时，> 10ms 需排查 |
| **可数循环陷阱** | JIT 编译的 int 可数循环不插入安全点，可能导致 TTSP 飙高 |
