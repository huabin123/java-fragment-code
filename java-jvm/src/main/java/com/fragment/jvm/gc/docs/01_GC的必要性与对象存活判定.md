# 第一章：GC 的必要性与对象存活判定

## 1.1 为什么需要 GC？

Java 与 C++ 的根本差别之一：
```
C++：程序员手动 malloc/free，不释放 → 内存泄漏，重复释放 → 崩溃
Java：JVM 自动 GC，但代价是 Stop-The-World 暂停

GC 要回答三个问题：
1. 哪些对象可以回收？（存活判定）
2. 何时回收？（触发时机）
3. 怎么回收？（GC 算法）
```

---

## 1.2 对象存活判定：可达性分析

Java 使用**可达性分析**（而非引用计数法）判断对象是否存活：

```
从 GC Roots 出发，沿引用链遍历
  → 能遍历到的对象：存活
  → 遍历不到的对象：可回收

GC Roots 包括：
  1. 虚拟机栈中引用的对象（当前执行方法的局部变量）
  2. 方法区中静态变量引用的对象
  3. 方法区中常量引用的对象（如字符串常量池中的引用）
  4. 本地方法栈中 JNI 引用的对象
  5. JVM 内部引用（基本数据类型对应的 Class 对象、常驻异常对象）
  6. 被 synchronized 锁持有的对象
```

`GCAlgorithmDemo.java → demonstrateReachabilityAnalysis()` 可观察 GC Roots 的实际表现。

## 1.3 为什么不用引用计数法？

```java
// 引用计数法的致命缺陷：循环引用
class Node {
    Node next;
}

Node a = new Node();
Node b = new Node();
a.next = b;   // b 的引用计数 = 1
b.next = a;   // a 的引用计数 = 1

a = null;     // a 的计数 -1 = 0？不，因为 b.next 还指向它，计数还是 1
b = null;     // b 的计数同理

// 结果：a 和 b 互相持有引用，计数永不为 0 → 永远不被回收！

// GCAlgorithmDemo.java → demonstrateCircularReference() 演示此问题
// Java 的可达性分析正确处理循环引用：
// 从 GC Roots 出发，a 和 b 都不可达，正确判定为可回收
```

## 1.4 四种引用类型

`GCAlgorithmDemo.java → demonstrateReferenceTypes()` 演示了四种引用：

| 引用类型 | 类 | 回收时机 | 典型用途 |
|---------|-----|---------|---------|
| 强引用 | 普通 `Object obj = new Object()` | 永不回收（内存不足也不回收）| 日常使用 |
| 软引用 | `SoftReference<T>` | 内存不足时回收 | 图片缓存、可重建的缓存 |
| 弱引用 | `WeakReference<T>` | 下次 GC 时回收 | WeakHashMap、ThreadLocal 的 key |
| 虚引用 | `PhantomReference<T>` | 随时可回收，get() 永远返回 null | 跟踪对象被回收的通知 |

```java
// 软引用缓存示例
SoftReference<byte[]> cache = new SoftReference<>(new byte[10 * 1024 * 1024]);
byte[] data = cache.get();  // 内存充足时不为 null；内存紧张时被回收，get() 返回 null
if (data == null) {
    data = reloadData();    // 缓存被回收，重新加载
    cache = new SoftReference<>(data);
}
```

## 1.5 对象的 finalize()：为什么不要用？

```java
// finalize() 给对象最后一次"自救"机会
// 被判定为不可达 → 放入 F-Queue → finalizer 线程调用 finalize()
// 如果 finalize() 中重新建立引用链 → 对象复活（只能自救一次）

// 为什么不要用：
// 1. 不保证何时执行（GC 不确定触发时机）
// 2. 不保证一定执行（JVM 可能提前退出）
// 3. 执行代价高（需要放入 F-Queue，再调度 finalizer 线程）
// 4. 已在 JDK 9 标注为 @Deprecated

// 替代方案：
// 1. try-finally / try-with-resources（IO 资源释放）
// 2. java.lang.ref.Cleaner（JDK 9+，PhantomReference 的封装）
```

---

## 1.6 本章总结

- **可达性分析**：从 GC Roots 出发，不可达 = 可回收；正确处理循环引用
- **引用计数法弃用**：无法处理循环引用，Java 不使用
- **四种引用**：强（不回收）、软（内存不足时）、弱（下次GC）、虚（跟踪通知）
- **finalize() 不要用**：不可预测、代价高、已 Deprecated，改用 try-with-resources
