# 第三章：垃圾收集器详解

## 3.1 收集器全景图

```
新生代收集器：Serial、ParNew、Parallel Scavenge
老年代收集器：Serial Old、Parallel Old、CMS
全堆收集器：G1、ZGC、Shenandoah

常见组合：
  Serial + Serial Old        → 单线程，Client 模式，小内存
  ParNew + CMS               → 服务端历史首选（JDK 8 主流）
  Parallel Scavenge + Parallel Old → 吞吐量优先（批处理、数据计算）
  G1                         → JDK 9+ 默认，延迟与吞吐平衡
  ZGC                        → JDK 15+ 生产可用，超低延迟（< 10ms）
```

---

## 3.2 GC 触发时机

### 3.2.1 Minor GC（新生代 GC）

**触发条件**：Eden 区空间不足，无法分配新对象时触发。

```
对象分配路径（触发 Minor GC 的完整链路）：

new Object()
  │
  ├─ TLAB 有剩余空间？
  │    是 → 直接在 TLAB 中分配（无锁，最快）
  │    否 ↓
  ├─ Eden 区有空间？
  │    是 → 在 Eden 区分配
  │    否 → 触发 Minor GC ← ★ 触发点
  │          GC 后 Eden 仍不够 → 尝试老年代分配（大对象直接进老年代）
  │          老年代也不够   → 触发 Full GC
  └─ 大对象（> -XX:PretenureSizeThreshold）→ 直接进老年代
```

**特点**：
- 只回收新生代（Eden + 两个 Survivor）
- STW，但停顿时间短（新生代空间小，存活对象少）
- 频率高（几秒到几十秒一次，取决于分配速率）

---

### 3.2.2 Full GC（全堆 GC）

Full GC 的触发条件较多，任意一条满足即触发：

```
触发 Full GC 的条件：

① 老年代空间不足
   - Minor GC 后，Survivor 放不下的存活对象要晋升老年代，但老年代空间不足
   - 大对象（超过 PretenureSizeThreshold）直接分配到老年代，但空间不足

② 空间分配担保失败（Promotion Failure）
   - Minor GC 前，JVM 检查：老年代剩余空间 < 历次晋升对象的平均大小
   - 判断本次 Minor GC 晋升的对象老年代"大概率放不下"
   - → 直接触发 Full GC，放弃本次 Minor GC

③ 方法区（元空间）空间不足
   - 类加载过多，元空间达到 -XX:MaxMetaspaceSize 上限
   - JVM 先尝试卸载无用类，若仍不足则触发 Full GC

④ 显式调用 System.gc()
   - 应用代码或第三方库调用 System.gc()
   - JVM 不保证立即执行，但通常会触发 Full GC
   - 可用 -XX:+DisableExplicitGC 禁止（RPC 框架等场景必须禁止）

⑤ CMS 并发模式失败（Concurrent Mode Failure）
   - CMS 并发回收期间，老年代空间耗尽，新对象无法分配
   - CMS 降级为 Serial Old → 单线程 Full GC，STW 时间极长

⑥ G1 Evacuation Failure（Region 复制失败）
   - G1 在 GC 时，目标 Region 空间不足，无法完成对象转移
   - 降级为 Full GC
```

---

### 3.2.3 各收集器的 GC 触发时机对比

| 收集器 | Young GC 触发 | Old GC / Full GC 触发 |
|-------|-------------|----------------------|
| Serial / ParNew + Serial Old | Eden 满 | 老年代满 |
| ParNew + CMS | Eden 满 | 老年代占用率达 `-XX:CMSInitiatingOccupancyFraction`（默认 68%）|
| G1 | Eden Region 用完 | 堆占用率达 `-XX:InitiatingHeapOccupancyPercent`（默认 45%）触发并发标记，进而触发 Mixed GC |
| ZGC | 分配速率 × 预测回收时间 > 剩余空间（自适应触发）| 几乎无 Full GC；极端情况下触发 |

### 3.2.4 如何观察 GC 触发时机

```bash
# 打印 GC 日志（JDK 9+ 统一日志格式）
java -Xlog:gc*:file=gc.log:time,uptime,level,tags -jar app.jar

# GC 日志关键字段：
# [0.500s][info][gc] GC(0) Pause Young (Normal) (G1 Evacuation Pause) ...
#                              ↑ GC 类型    ↑ 触发原因

# 常见触发原因标注：
#   G1 Evacuation Pause  → G1 正常 Young GC（Eden 满）
#   G1 Humongous Allocation → 大对象分配导致 GC
#   Metadata GC Threshold   → 元空间不足触发
#   System.gc()             → 显式调用
#   Promotion Failed        → 担保失败，触发 Full GC
#   Concurrent Mode Failure → CMS 并发失败，降级 Full GC

# JDK 8 旧格式（-XX:+PrintGCDetails -XX:+PrintGCDateStamps）：
# 2024-01-01T10:00:00.000+0800: [GC (Allocation Failure) ...]
#                                          ↑ Allocation Failure = Eden 满
```

---

### 3.2.5 减少 Full GC 的关键手段

```
① 避免显式 System.gc()
   -XX:+DisableExplicitGC

② 避免大对象频繁创建（直接进老年代，跳过新生代晋升）
   合理设置 -XX:PretenureSizeThreshold（Serial/ParNew 生效，G1 不适用）

③ 降低对象晋升速率（Minor GC 后尽量留在新生代）
   - 增大新生代比例（-XX:NewRatio）
   - 适当调大 Survivor 区（-XX:SurvivorRatio）
   - 增大晋升年龄阈值（-XX:MaxTenuringThreshold，默认 15）

④ CMS：提前触发，避免并发模式失败
   -XX:CMSInitiatingOccupancyFraction=70  # 70% 时触发，留足并发回收空间
   -XX:+UseCMSInitiatingOccupancyOnly     # 只按此比例触发，不自适应

⑤ G1：调整触发并发标记的堆占用阈值
   -XX:InitiatingHeapOccupancyPercent=40  # 降低阈值，更早触发并发标记
```

## 3.3 Serial / Serial Old

```
Serial：单线程，新生代，标记-复制
  GC 时：暂停所有线程 → 单线程收集 → 恢复

Serial Old：单线程，老年代，标记-整理

适用：
  - Client 模式（桌面应用）
  - 内存较小（< 100MB）
  - 单核 CPU

参数：-XX:+UseSerialGC
```

---

## 3.4 ParNew + CMS（历史最广泛的服务端组合）

**ParNew**：Serial 的多线程版，新生代，标记-复制。

**CMS（Concurrent Mark Sweep）**：并发标记清除，老年代：

```
CMS 的四个阶段：
  1. 初始标记（STW）  → 标记 GC Roots 直接关联的对象（快，STW 短）
  2. 并发标记          → 与用户线程并发，遍历整个引用链（慢，但不 STW）
  3. 重新标记（STW）  → 修正并发标记期间变化的引用（比初始标记慢，比并发标记快）
  4. 并发清除          → 与用户线程并发，清除不可达对象（不 STW）

优点：STW 时间短，响应时间好
缺点：
  1. 浮动垃圾（并发清除期间产生的新垃圾，下次 GC 才能回收）
  2. 内存碎片（标记-清除算法）
  3. 并发模式失败（CMS 期间老年代满了 → 降级为 Serial Old，STW 极长）

参数：-XX:+UseConcMarkSweepGC
     -XX:CMSInitiatingOccupancyFraction=75  # 老年代 75% 时触发 CMS
```

## 3.5 G1（Garbage First）

G1 是 JDK 9+ 的默认收集器，核心思想是**化整为零**：

```
G1 把堆划分为若干等大的 Region（默认 1~32MB）
每个 Region 可以是 Eden、Survivor、Old、Humongous（大对象）的任意一种

GC 时优先收集垃圾最多的 Region（Garbage First 名称由来）
→ 在可控时间内最大化回收效率

G1 的四种 GC：
  1. Young GC   → 只收集新生代 Region，STW
  2. Concurrent Marking → 并发标记老年代（与用户线程并发）
  3. Mixed GC   → 收集全部新生代 + 部分老年代 Region
  4. Full GC    → 兜底，单线程，STW（尽量避免）

参数：
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=200    # 目标最大停顿时间（默认 200ms）
  -XX:G1HeapRegionSize=16m    # Region 大小
```

### G1 大对象（Humongous Object）处理

```
大对象判定标准：对象大小 ≥ Region 大小的 50%

Region 大小计算规则：
  默认：堆大小 / 2048（结果向上取整为 2 的幂，范围 1MB~32MB）
  手动：-XX:G1HeapRegionSize=<size>

  ┌──────────┬─────────────┬──────────────────┐
  │ 堆大小   │ Region 大小 │ 大对象阈值       │
  ├──────────┼─────────────┼──────────────────┤
  │ 2GB      │ 1MB         │ ≥ 512KB          │
  │ 4GB      │ 2MB         │ ≥ 1MB            │
  │ 8GB      │ 4MB         │ ≥ 2MB            │
  │ 16GB     │ 8MB         │ ≥ 4MB            │
  │ 32GB     │ 16MB        │ ≥ 8MB            │
  └──────────┴─────────────┴──────────────────┘

Humongous Region 分配方式：
  对象 < Region → 占用 1 个 Humongous Region
  对象 > Region → 占用多个连续的 Humongous Region

  示例（Region = 8MB）：
    new byte[3MB]  → 1 个 Humongous Region（3MB ≥ 4MB 阈值？否，走普通分配）
    new byte[5MB]  → 1 个 Humongous Region（5MB ≥ 4MB）
    new byte[20MB] → 3 个连续 Humongous Region（20MB / 8MB = 向上取整 3 个）

大对象的特殊之处：
  1. 跳过新生代，直接在老年代的 Humongous Region 分配
  2. 不参与 Young GC 的复制（太大，复制代价高）
  3. JDK 8u40+：Young GC 和 Mixed GC 时会回收无引用的 Humongous 对象
     （之前只有 Full GC 才回收，造成内存浪费）
  4. 连续 Region 要求可能导致碎片（找不到足够连续的 Region → 触发 Full GC）

⚠️ 大对象的坑：
  List<byte[]> cache = new ArrayList<>();
  for (...) {
      cache.add(new byte[5 * 1024 * 1024]);  // 每次 5MB → Humongous
  }
  // 30 分钟后 cache 占用大量 Humongous Region
  // 即使 cache.clear()，也要等 GC 回收 Humongous Region
  // 如果老年代占用率未达 IHOP 阈值 → 迟迟不触发并发标记 → 内存不释放

优化手段：
  1. 增大 Region（-XX:G1HeapRegionSize=16m），提高大对象阈值，减少 Humongous
  2. 避免频繁创建大数组（复用 / 对象池）
  3. 降低 IHOP（-XX:InitiatingHeapOccupancyPercent=35），更早触发并发标记
```

## 3.6 ZGC（JDK 15+ 生产可用）

```
目标：停顿时间 < 10ms，且不随堆大小增长（支持 TB 级堆）

核心技术：
  1. 着色指针（Colored Pointer）：在指针的高位存储 GC 状态，不需要 Mark Word
  2. 读屏障（Load Barrier）：每次读对象引用时，检查/修正指针（代替 STW 重标记）
  3. 几乎全程并发：只有根扫描和 ZGC Pause 两个极短 STW

STW 时间：通常 < 1ms，与堆大小无关

参数：
  -XX:+UseZGC
  -XX:ZAllocationSpikeTolerance=5   # 分配速率容忍度

适用：对延迟极度敏感的服务（API 响应时间要求 < 50ms）
```

## 3.7 收集器选型指南

```
服务类型                      推荐收集器
─────────────────────────────────────────────────────
API 服务（延迟敏感，< 200ms）  G1（JDK 9+）
超低延迟服务（< 50ms）          ZGC（JDK 15+）
批处理/数据计算（吞吐量优先）    Parallel GC（-XX:+UseParallelGC）
历史遗留系统（JDK 6/7）         ParNew + CMS
小内存应用（< 256MB）            Serial GC
```

---

## 3.8 本章总结

- **Serial**：单线程，简单，STW 最长，适合小内存
- **CMS**：并发标记清除，低延迟，但有碎片和浮动垃圾；JDK 14 已移除
- **G1**：分 Region，可预测停顿，JDK 9+ 默认
- **ZGC**：着色指针 + 读屏障，STW < 10ms，JDK 15+ 生产可用
- **选型**：延迟敏感选 G1/ZGC，吞吐量优先选 Parallel，历史系统保持 CMS
