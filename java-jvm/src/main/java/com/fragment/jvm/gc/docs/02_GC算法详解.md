# 第二章：GC 算法详解

## 2.1 标记-清除（Mark-Sweep）

```
阶段1：标记（Mark）
  从 GC Roots 出发，标记所有可达对象

阶段2：清除（Sweep）
  遍历堆，回收未标记对象

优点：实现简单，不需要移动对象
缺点：产生大量内存碎片（回收后内存不连续）
     碎片过多导致大对象分配失败，触发 Full GC
```

```
回收前：[A][B][  ][C][D][  ][E][  ][F]
回收后：[A][  ][  ][C][  ][  ][E][  ][F]
           ↑         ↑         ↑
        碎片空间，无法分配连续的大对象
```

---

## 2.2 标记-复制（Mark-Copy）

```
将内存分为两半，每次只用一半

阶段1：标记存活对象
阶段2：将存活对象复制到另一半，然后清空当前半区

优点：无碎片，分配速度快（只需移动指针）
缺点：内存利用率只有 50%

改进版（新生代实际用法）：Eden + S0 + S1（8:1:1）
  每次 Minor GC：Eden + 使用中的 Survivor → 另一个 Survivor
  内存利用率提升到 90%（只浪费一个 Survivor）
```

`GCAlgorithmDemo.java → demonstrateGCAlgorithms()` 模拟了标记-复制的过程。

---

## 2.3 标记-整理（Mark-Compact）

```
阶段1：标记存活对象
阶段2：将存活对象移动到内存一端（整理）
阶段3：清理边界以外的内存

优点：无碎片，内存利用率高（100%）
缺点：移动对象代价高（需要更新所有引用），暂停时间长

适用：老年代（对象存活率高，用标记-复制浪费太多）
```

```
整理前：[A][  ][  ][C][  ][  ][E][  ][F]
整理后：[A][C][E][F][                  ]
                    ↑
              连续空闲空间，可分配大对象
```

---

## 2.4 三种算法的对比与适用场景

| 算法 | 碎片 | 内存效率 | 停顿时间 | 适用区域 |
|------|------|---------|---------|---------|
| 标记-清除 | ❌ 多碎片 | 高（不移动）| 短 | CMS 初始阶段 |
| 标记-复制 | ✅ 无碎片 | 低（50%）| 最短 | 新生代（Eden+Survivor）|
| 标记-整理 | ✅ 无碎片 | 高（100%）| 最长 | 老年代 |

**实际分代策略**：
- **新生代**：标记-复制（大多数对象朝生夕死，存活少，复制代价低）
- **老年代**：标记-整理 或 标记-清除（存活率高，复制代价太高）

---

## 2.5 Stop-The-World（STW）

```
GC 执行期间，必须暂停所有 Java 线程（STW），否则引用关系在 GC 过程中被修改，标记结果失效。

不同 GC 算法的 STW 时长差异很大：
  Serial GC     → 单线程 GC，STW 最长
  Parallel GC   → 多线程 GC，缩短 STW
  CMS           → 并发标记，减少 STW，但不能完全消除
  G1            → 可预测停顿，目标 < 200ms
  ZGC / Shenandoah → 几乎全程并发，STW < 10ms

STW 的影响：
  Young GC（Minor GC）：通常几毫秒到几十毫秒
  Full GC：可能几百毫秒到几秒，对服务可用性影响大
```

---

## 2.6 本章总结

- **三种基本算法**：标记-清除（有碎片）、标记-复制（内存50%，无碎片）、标记-整理（无碎片，代价高）
- **新生代用复制**：对象存活率低，复制代价小；Eden+S0+S1 提升到 90% 利用率
- **老年代用整理**：对象存活率高，复制代价太高；需要整理消除碎片
- **STW 是核心矛盾**：GC 越彻底、停顿越长；现代 GC（G1/ZGC）核心目标就是降低 STW
