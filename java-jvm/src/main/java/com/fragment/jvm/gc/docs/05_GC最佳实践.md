## 5.1 生产环境必开的 JVM 参数

```bash
# 完整的生产环境推荐配置（以 4GB 堆 + G1 为例）
java \
  -Xms4g -Xmx4g \                                    # 堆大小，初始=最大
  -XX:+UseG1GC \                                      # G1 收集器
  -XX:MaxGCPauseMillis=200 \                          # 目标停顿时间
  -XX:G1HeapRegionSize=16m \                          # Region 大小
  -XX:InitiatingHeapOccupancyPercent=45 \             # 触发并发标记阈值
  -XX:+HeapDumpOnOutOfMemoryError \                   # OOM 时自动 dump
  -XX:HeapDumpPath=/var/log/app/heap.hprof \
  -Xlog:gc*:file=/var/log/app/gc.log:time,uptime:filecount=10,filesize=50m \
  -jar app.jar
```

## 5.2 减少 GC 压力的编码实践

```java
// ✅ 1. 避免创建大量短生命周期对象
// ❌ 每次请求都 new 大数组
byte[] buffer = new byte[8192];

// ✅ 使用对象池或 ThreadLocal 复用
private static final ThreadLocal<byte[]> BUFFER =
    ThreadLocal.withInitial(() -> new byte[8192]);
byte[] buffer = BUFFER.get();

// ✅ 2. 使用 StringBuilder 代替 String 拼接
// ❌ 每次 + 都创建新 String 对象
String result = "";
for (int i = 0; i < 1000; i++) result += i;  // 创建 1000 个临时 String

// ✅
StringBuilder sb = new StringBuilder(4096);  // 预估容量
for (int i = 0; i < 1000; i++) sb.append(i);
String result = sb.toString();
```

## 5.3 软引用缓存的正确用法

```java
// 使用软引用实现内存敏感缓存
// 内存充足时缓存存活；内存不足时 GC 自动清除
public class SoftCache<K, V> {
    private final Map<K, SoftReference<V>> cache = new ConcurrentHashMap<>();

    public V get(K key) {
        SoftReference<V> ref = cache.get(key);
        if (ref == null) return null;
        V value = ref.get();
        if (value == null) cache.remove(key);  // 引用已被 GC 清除，清理 key
        return value;
    }

    public void put(K key, V value) {
        cache.put(key, new SoftReference<>(value));
    }
}

// 注意：不要滥用软引用代替合理的缓存淘汰策略
// 软引用的清除不可预测，更好的方案是 Caffeine/Guava Cache（基于大小/时间淘汰）
```

## 5.5 GC 与业务代码的正确边界

```java
// ❌ 不要手动调用 System.gc()
System.gc();  // 只是"建议"JVM GC，不保证立即执行，且可能触发 Full GC

// ❌ 不要依赖 finalize() 做资源清理（已 Deprecated）
@Override
protected void finalize() {
    closeResource();  // 不可靠！
}

// ✅ 正确的资源管理
try (InputStream in = new FileInputStream(file)) {
    // 自动关闭
}  // try-with-resources，无论正常/异常都会 close()

// ✅ 大数组用完后显式置 null（辅助 GC，非必须）
byte[] largeBuffer = new byte[100 * 1024 * 1024];
process(largeBuffer);
largeBuffer = null;   // 提示 GC 可以回收
```

## 5.6 检测当前使用的垃圾收集器

生产环境中经常需要确认应用实际使用的 GC 类型：

```bash
# 方式1：jinfo（最直接，推荐）
jinfo -flag UseG1GC <PID>          # -XX:+UseG1GC 表示启用
jinfo -flag UseParallelGC <PID>    # -XX:-UseParallelGC 表示未启用
jinfo -flag UseZGC <PID>
jinfo -flag UseConcMarkSweepGC <PID>
jinfo -flag UseSerialGC <PID>

# 方式2：jcmd
jcmd <PID> VM.flags | grep -i gc

# 方式3：启动时打印
java -XX:+PrintFlagsFinal -jar app.jar 2>&1 | grep -i "Use.*GC"
# 输出示例：
#   bool UseG1GC           = true   {product} {ergonomic}
#   bool UseParallelGC     = false  {product} {default}
#   ← ergonomic 表示 JVM 自动选择，command line 表示手动指定

# 方式4：GC 日志关键字判断
# G1:       "G1 Evacuation Pause" / "G1 Humongous Allocation"
# Parallel: "PSYoungGen" / "ParOldGen"
# CMS:      "ParNew" / "CMS-concurrent"
# Serial:   "DefNew" / "Tenured"
# ZGC:      "ZGC" / "Z:"
```

```java
// 方式5：代码中通过 MXBean 检测
import java.lang.management.GarbageCollectorMXBean;
import java.lang.management.ManagementFactory;

for (GarbageCollectorMXBean gc : ManagementFactory.getGarbageCollectorMXBeans()) {
    System.out.println(gc.getName() + " → " + gc.getMemoryPoolNames());
}
// G1 输出：    "G1 Young Generation" + "G1 Old Generation"
// Parallel：  "PS Scavenge" + "PS MarkSweep"
// CMS：       "ParNew" + "ConcurrentMarkSweep"
// ZGC：       "ZGC"

// 快速判断
boolean isG1 = ManagementFactory.getGarbageCollectorMXBeans().stream()
    .anyMatch(gc -> gc.getName().contains("G1"));
```

```
各 JDK 版本的默认 GC：
  JDK 7   → Parallel GC（-XX:+UseParallelGC）
  JDK 8   → Parallel GC
  JDK 9+  → G1 GC（-XX:+UseG1GC）
  JDK 15+ → ZGC 可用（-XX:+UseZGC，非默认）
  JDK 21+ → Generational ZGC（-XX:+UseZGC -XX:+ZGenerational）
```

## 5.7 本章总结

**六条核心实践**：
1. **生产必开**：GC 日志 + HeapDumpOnOutOfMemoryError + G1（JDK 9+）
2. **减少对象创建**：ThreadLocal 复用、StringBuilder、预设集合容量、流式处理大数据
3. **软引用缓存**：内存敏感场景使用，但更复杂需求用 Caffeine
4. **诊断工具**：jstat 实时监控，GCEasy 分析日志，MAT 分析 heap dump
5. **不要乱用**：不调 `System.gc()`，不用 `finalize()`，用 try-with-resources 管理资源
6. **确认 GC 类型**：`jinfo -flag UseG1GC`、GC 日志关键字、MXBean 代码检测
