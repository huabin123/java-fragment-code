# 第四章：GC 调优实战

## 4.1 GC 调优的目标

调优不是无脑加大堆或换收集器，而是：

```
1. 降低 Full GC 频率（Full GC 是最影响服务可用性的）
2. 缩短 GC 停顿时间（STW）
3. 提高吞吐量（应用运行时间 / 总运行时间）

三者不能同时最优，需要根据业务取舍：
  - 低延迟优先（API 服务）→ 接受低吞吐，选 G1/ZGC
  - 高吞吐优先（批处理）  → 接受长停顿，选 Parallel GC
```

## 4.2 GC 日志分析

首先要会看 GC 日志：

```bash
# 开启 GC 日志（JDK 9+ 统一日志格式）
-Xlog:gc*:file=/tmp/gc.log:time,uptime,level,tags:filecount=5,filesize=20m

# JDK 8 旧格式
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/tmp/gc.log

# 典型 G1 GC 日志解读：
[2026-04-10T10:00:01.234+0800] GC(5) Pause Young (Normal) (G1 Evacuation Pause)
  Eden: 512M(512M)->0B(512M) Survivors: 32M->32M Heap: 1024M(2048M)->520M(2048M)
  [Eden: 512MB → 0]      # 新生代被清空
  [Survivors: 32MB 未变]  # Survivor 区
  [Heap: 1024→520MB]     # 堆使用量减少（回收了约 504MB）
  Pause 45ms             # STW 45毫秒
```

`GCTuningDemo.java → analyzGCLog()` 演示了 GC 日志的解析。

## 4.3 常见 GC 问题与解法

### 问题1：频繁 Young GC

```
现象：Young GC 每秒 3 次以上，CPU 使用率高
原因：新生代太小，对象频繁填满 Eden
解法：增大新生代 -Xmn 或 -XX:NewRatio
     或优化对象分配，减少短生命周期大对象
```

### 问题2：频繁 Full GC

```
现象：Full GC 每分钟多次，服务响应时间出现尖刺
原因：
  A. 老年代内存泄漏（对象持续积累）
  B. 新生代太小，对象过早晋升老年代
  C. 大对象直接进老年代过多
解法：
  A → 分析 Heap Dump，找内存泄漏
  B → 增大新生代，增大 -XX:MaxTenuringThreshold
  C → 避免大对象，或增大老年代
```

### 问题3：CMS 并发模式失败（Concurrent Mode Failure）

```
现象：GC 日志出现 "concurrent mode failure"，Full GC 频繁
原因：CMS 并发清理期间老年代空间不足（浮动垃圾 + 新晋升对象撑满）
解法：降低触发阈值 -XX:CMSInitiatingOccupancyFraction=65
     增大老年代（增大 -Xmx）
     迁移到 G1
```

### 问题4：G1 Full GC

```
现象：G1 日志出现 "Pause Full"，停顿时间极长
原因：Evacuation Failure（对象复制时空间不足）
解法：增大堆大小 -Xmx
     增大 -XX:G1ReservePercent（保留更多空间作为缓冲）
     降低 -XX:InitiatingHeapOccupancyPercent（更早触发并发标记）
```

## 4.4 调优参数速查

```bash
# 堆大小（生产环境 -Xms == -Xmx，避免动态扩缩）
-Xms4g -Xmx4g

# G1 调优参数
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200            # 目标停顿时间
-XX:G1HeapRegionSize=16m            # Region 大小（建议 1~32m，2的幂次）
-XX:InitiatingHeapOccupancyPercent=45 # 堆占用多少比例触发并发标记（默认45）
-XX:G1ReservePercent=15             # 保留多少比例作为 Evacuation 缓冲（默认10）

# GC 日志（必须开启，生产排障必需）
-Xlog:gc*:file=/var/log/app/gc.log:time,uptime:filecount=10,filesize=50m

# OOM 时自动 dump（必须开启）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/app/heap-dump.hprof
```

## 4.5 GC 调优的正确姿势

```
Step 1：收集基线数据
  开启 GC 日志，运行 1~7 天，用 GCEasy / GCViewer 分析

Step 2：确认问题类型
  频繁 Young GC → 新生代太小
  频繁 Full GC  → 内存泄漏 or 老年代太小
  停顿时间过长  → 换收集器（G1/ZGC）

Step 3：单次只改一个参数
  修改后观察至少 24 小时效果，避免多参数同时修改

Step 4：验证改善
  对比改前改后：GC 频率、平均停顿、P99 停顿、吞吐量
```

## 4.6 本章总结

- **调优三目标**：降低 Full GC 频率 > 缩短 STW > 提高吞吐，三者需取舍
- **先开 GC 日志**：没有日志就没有调优依据；生产环境 GC 日志 + HeapDump 必须开启
- **常见问题**：频繁 YGC（新生代小）、频繁 FGC（泄漏/老年代小/CMS 阈值）
- **调优原则**：单参数逐步调整，观察 24h+，用数据说话
