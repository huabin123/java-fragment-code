# 第一章：为什么需要注解？

## 1.1 注解解决的核心问题：配置与代码的耦合

在注解出现之前（JDK 1.4 之前），框架配置必须依赖外部 XML 文件。以一个简单的事务配置为例：

```xml
<!-- Spring 早期风格：applicationContext.xml -->
<bean id="userService" class="com.example.UserService">
    <property name="transactional" value="true"/>
    <property name="timeout" value="30"/>
</bean>
```

这种方式的核心问题：

1. **配置与代码分离**：修改类名或方法名后，必须同步修改 XML，否则运行时才会报错
2. **可读性差**：看代码时无法直接知道这个类/方法有哪些行为修饰
3. **维护成本高**：大型项目 XML 文件可达数千行，几乎无法维护

**注解的本质**：把元数据（"关于代码的数据"）直接嵌入源代码，让配置与代码同步演进，编译器和工具链可以直接读取和验证。

## 1.2 注解的三种作用层次

注解本身不包含任何逻辑，它的作用取决于**谁在处理它**：

### 层次一：给编译器看（RetentionPolicy.SOURCE）

```java
@Override          // 让编译器验证方法确实覆盖了父类方法
@SuppressWarnings  // 告诉编译器忽略特定警告
@Deprecated        // 标记已废弃，编译器会产生警告
```

这类注解在编译后就消失了，不存在于 .class 文件中。

### 层次二：给字节码工具看（RetentionPolicy.CLASS，默认）

```java
// 字节码增强工具（如 ASM、ByteBuddy）在这一层工作
// Lombok 的 @Data、@Getter 等注解在编译时生成 get/set 方法字节码
```

### 层次三：给运行时看（RetentionPolicy.RUNTIME）

```java
// Spring 的 @Autowired, @Transactional
// JPA 的 @Entity, @Column
// Hibernate Validator 的 @NotNull, @Email
```

**这是最强大、使用最广泛的层次**。运行时通过反射读取注解，动态改变程序行为。`AnnotationBasicDemo.java` 中所有的 `clazz.getAnnotation()`、`field.isAnnotationPresent()` 都属于这一层。

---

## 1.3 注解如何改变程序行为：三种机制

注解本身是"惰性的"，它不会主动做任何事。改变行为有三种触发机制：

**机制一：运行时反射（最常见）**

```java
// AnnotationProcessorDemo.java → SimpleIoCContainer
// 扫描类上的 @Component 注解，决定是否创建 Bean
if (!clazz.isAnnotationPresent(Component.class)) {
    throw new IllegalArgumentException(clazz + " 没有@Component注解");
}
// 扫描字段上的 @Autowired 注解，决定是否注入依赖
for (Field field : clazz.getDeclaredFields()) {
    if (field.isAnnotationPresent(Autowired.class)) {
        // 注入对应类型的 Bean
    }
}
```

**机制二：动态代理拦截（AOP 的基础）**

```java
// AnnotationProcessorDemo.java → SimpleCacheManager.createProxy()
// 代理拦截方法调用，读取 @Cacheable 注解决定是否走缓存
if (method.isAnnotationPresent(Cacheable.class)) {
    String cacheKey = cacheable.value() + ":" + Arrays.toString(args);
    if (cache.containsKey(cacheKey)) return cache.get(cacheKey);
    Object result = method.invoke(target, args);
    cache.put(cacheKey, result);
    return result;
}
```

**机制三：编译时注解处理器（APT）**

Lombok 的 `@Data`、MapStruct 的 `@Mapper` 在编译阶段生成额外的 Java 代码，这是性能最高的方式（零运行时开销）。

## 1.4 标准注解 vs 框架注解

Java 内置的标准注解（`java.lang.annotation.*`）非常少，工程价值主要体现在它们定义了"注解的规则"（元注解）：

| 元注解 | 作用 |
|--------|------|
| `@Retention` | 声明注解的存活时期（SOURCE/CLASS/RUNTIME）|
| `@Target` | 声明注解可以修饰的元素类型 |
| `@Inherited` | 子类是否继承父类的注解 |
| `@Documented` | 是否包含在 Javadoc 中 |
| `@Repeatable` | 是否允许在同一位置重复使用（JDK 8+）|

`AnnotationBasicDemo.java → demonstrateAnnotationInheritance()` 演示了 `@Inherited` 的精确行为：只有加了 `@Inherited` 的注解才会被子类继承，否则子类读不到父类的注解。

## 1.5 注解驱动开发的设计哲学

以 `ValidationFramework.java` 为例，它用注解替代了大量的手写验证逻辑：

```java
// 旧写法：验证逻辑硬编码，散落在业务代码中
public void register(UserRegisterForm form) {
    if (form.getUsername() == null || form.getUsername().length() < 3) {
        throw new ValidationException("用户名长度不符合要求");
    }
    if (!form.getEmail().matches("^[A-Za-z0-9+_.-]+@(.+)$")) {
        throw new ValidationException("邮箱格式不正确");
    }
    // ... 更多验证
}

// 新写法：验证规则声明式地附着在数据类上
@NotBlank
@Size(min = 3, max = 20, message = "用户名长度必须在3-20之间")
private String username;

@Email(message = "邮箱格式不正确")
private String email;
```

这体现了**声明式编程**的核心思想：描述"是什么"，而不是"怎么做"。注解让框架承担了"怎么做"的职责，业务代码只需声明规则。

---

## 1.6 本章总结

- **注解的本质**：附着在代码元素上的结构化元数据，不含业务逻辑
- **三种保留策略**：SOURCE（编译器检查）、CLASS（字节码工具）、RUNTIME（反射驱动）
- **三种触发机制**：运行时反射、动态代理、编译时 APT
- **设计价值**：把配置从 XML 拉回到代码中，实现声明式编程，消除配置与代码的耦合
- **注解自身无能力**：改变行为的是读取注解的框架代码，不是注解本身
