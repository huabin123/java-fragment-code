# 第三章：注解的工作原理

## 3.1 注解的底层实现：动态代理

通过反射获取的注解对象，实际上是 JVM 生成的**动态代理对象**，这个代理实现了注解接口并持有属性值：

```java
Column column = field.getAnnotation(Column.class);
// column 是一个实现了 Column 接口的代理对象
// 调用 column.name() 实际上是通过代理读取注解属性值

System.out.println(column.getClass());
// 输出类似：class com.sun.proxy.$Proxy1
```

这解释了为什么注解属性只能是有限的几种类型——它们都必须能被存储在 .class 文件的常量池中，并被代理对象在运行时返回。

## 3.2 反射读取注解的完整 API

`AnnotationBasicDemo.java` 展示了四个维度的注解读取：

### 类级别

```java
// demonstrateClassAnnotation()
Class<User> clazz = User.class;

// 检查是否存在某注解
boolean hasEntity = clazz.isAnnotationPresent(Entity.class);

// 获取特定注解
Entity entity = clazz.getAnnotation(Entity.class);

// 获取所有注解（不含继承的）
Annotation[] declared = clazz.getDeclaredAnnotations();

// 获取所有注解（含 @Inherited 继承的）
Annotation[] all = clazz.getAnnotations();
```

### 字段级别

```java
// demonstrateFieldAnnotation()
for (Field field : clazz.getDeclaredFields()) {
    if (field.isAnnotationPresent(Column.class)) {
        Column column = field.getAnnotation(Column.class);
        System.out.println("列名: " + column.name());
        System.out.println("可空: " + column.nullable());
        System.out.println("长度: " + column.length());
    }
}
```

### 方法级别

```java
// demonstrateMethodAnnotation()
for (Method method : clazz.getDeclaredMethods()) {
    if (method.isAnnotationPresent(Transactional.class)) {
        Transactional tx = method.getAnnotation(Transactional.class);
        System.out.println("传播: " + tx.propagation());
        System.out.println("超时: " + tx.timeout());
    }
}
```

### 可重复注解（JDK 8+）

```java
// demonstrateRepeatableAnnotation()
// 方式1：通过容器注解（兼容旧写法）
Schedules container = method.getAnnotation(Schedules.class);
for (Schedule s : container.value()) { ... }

// 方式2：getAnnotationsByType（推荐，自动处理容器）
Schedule[] schedules = method.getAnnotationsByType(Schedule.class);
```

## 3.3 ORM 注解处理器的实现原理

`AnnotationProcessorDemo.java → SimpleSQLGenerator` 演示了最经典的注解处理模式——读取注解生成 SQL：

```java
// generateInsertSQL() 的核心逻辑
public String generateInsertSQL(Object entity) throws Exception {
    Class<?> clazz = entity.getClass();

    // 1. 读取类上的 @Table 注解获取表名
    String tableName = clazz.getAnnotation(Table.class).name();  // "t_product"

    // 2. 遍历字段，读取 @Column 注解获取列名
    for (Field field : clazz.getDeclaredFields()) {
        if (field.isAnnotationPresent(Column.class)) {
            Column column = field.getAnnotation(Column.class);
            columns.add(column.name());           // 注解里的列名
            field.setAccessible(true);
            values.add(formatValue(field.get(entity)));  // 字段实际值
        }
    }

    // 3. 组装 SQL
    return "INSERT INTO " + tableName + " (" + join(columns) + ") VALUES (" + join(values) + ")";
}
```

**这个模式的本质**：注解是"字段元数据"的载体，处理器在运行时读取这些元数据，动态生成行为（SQL）。JPA、MyBatis 的注解模式与此完全一致。

## 3.4 依赖注入的注解处理流程

`AnnotationProcessorDemo.java → SimpleIoCContainer` 演示了 Spring `@Autowired` 的核心原理：

```java
// register() → injectDependencies()
private void injectDependencies(Object bean) throws Exception {
    for (Field field : bean.getClass().getDeclaredFields()) {
        if (field.isAnnotationPresent(Autowired.class)) {
            Class<?> fieldType = field.getType();

            // 从容器中查找对应类型的 Bean
            Object dependency = beans.get(fieldType);

            // 强制访问私有字段
            field.setAccessible(true);

            // 注入：本质上是反射赋值
            field.set(bean, dependency);
        }
    }
}
```

**完整流程**：
```
register(UserService.class)
  ↓
创建实例：UserService.class.newInstance()
  ↓
扫描字段，发现 @Autowired UserDao userDao
  ↓
从容器找到 UserDao 实例
  ↓
field.setAccessible(true); field.set(bean, dao);
  ↓
注入完成
```

## 3.5 注解 + 动态代理 = 轻量级 AOP

`AnnotationProcessorDemo.java → SimpleCacheManager` 展示了注解与动态代理的协同：

```java
// createProxy() 的完整逻辑
Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),
    (proxy, method, args) -> {
        // ①  读取方法上的 @Cacheable 注解
        if (method.isAnnotationPresent(Cacheable.class)) {
            String cacheKey = method.getAnnotation(Cacheable.class).value()
                              + ":" + Arrays.toString(args);
            // ② 缓存命中则直接返回，不执行原方法
            if (cache.containsKey(cacheKey)) return cache.get(cacheKey);
            // ③ 未命中：执行原方法并缓存结果
            Object result = method.invoke(target, args);
            cache.put(cacheKey, result);
            return result;
        }
        // ④ 读取 @CacheEvict，清除缓存
        if (method.isAnnotationPresent(CacheEvict.class)) {
            String prefix = method.getAnnotation(CacheEvict.class).value() + ":";
            cache.keySet().removeIf(k -> k.startsWith(prefix));
        }
        return method.invoke(target, args);
    }
);
```

Spring AOP 的 `@Transactional`、`@Cacheable` 的实现原理与此完全一致，区别只在于代理创建方式（JDK Proxy vs CGLIB）和切面逻辑的复杂程度。

## 3.6 本章总结

- **注解对象是代理**：`getAnnotation()` 返回的是 JVM 生成的动态代理，调用方法读取注解属性
- **四个读取入口**：类（`clazz.getAnnotation`）、字段（`field.getAnnotation`）、方法（`method.getAnnotation`）、参数（`param.getAnnotation`）
- **ORM 模式**：注解存储字段元数据，处理器遍历字段读取注解，动态生成 SQL
- **IoC 模式**：注解标记注入点，容器扫描注入点并通过 `field.set()` 完成依赖注入
- **AOP 模式**：代理拦截方法调用，根据注解决定是否执行增强逻辑（缓存/事务/日志）
