# 第四章：注解的实际应用与陷阱

## 4.1 ValidationFramework：注解驱动验证框架解析

`ValidationFramework.java` 是本模块最完整的实战案例，实现了类似 Hibernate Validator 的验证框架。

### 注解体系设计

```
字段级注解：@NotNull, @NotBlank, @Size, @Email, @Phone, @Min, @Max, @Range, @Pattern, @In, @Valid
类级注解：@FieldEquals（跨字段校验，如两次密码一致性）
```

### 核心验证器实现

```java
// ValidationFramework.java → Validator.validate()
public ValidationResult validate(Object obj) {
    // 1. 类级别注解（@FieldEquals 跨字段校验）
    validateClassAnnotations(obj, clazz, result, groups);

    // 2. 字段级别注解（逐字段逐注解校验）
    for (Field field : clazz.getDeclaredFields()) {
        field.setAccessible(true);
        Object value = field.get(obj);

        if (field.isAnnotationPresent(NotNull.class) && value == null) {
            result.addError(field.getName(), annotation.message());
            continue;  // null 时跳过后续长度/格式校验
        }

        if (field.isAnnotationPresent(Size.class)) {
            int len = value.toString().length();
            if (len < min || len > max) result.addError(...);
        }

        // @Valid 级联验证子对象
        if (field.isAnnotationPresent(Valid.class)) {
            ValidationResult sub = validate(value);
            sub.getErrors().forEach(e ->
                result.addError(fieldName + "." + e.getField(), e.getMessage()));
        }
    }
}
```

### 级联验证（@Valid）

`testCascadeValidation()` 演示了嵌套对象的验证：`User` 包含 `Address`，`address` 字段加了 `@Valid`，验证 `User` 时会自动递归验证 `Address`，错误路径为 `address.city`、`address.zipCode` 等。

## 4.2 常见陷阱

### 陷阱一：忘记 @Retention(RUNTIME)

```java
// ❌ 忘写 @Retention，默认是 CLASS，运行时读不到
@Target(ElementType.METHOD)
public @interface MyCache {
    String value();
}

// 运行时：
method.getAnnotation(MyCache.class);  // 返回 null！不报错！
method.isAnnotationPresent(MyCache.class);  // 返回 false！

// ✅ 正确：
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface MyCache { ... }
```

这是最常见的注解失效原因，且错误信息极其隐蔽（静默返回 null）。

### 陷阱二：@Inherited 对接口和方法无效

```java
@Inherited
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface ServiceAnnotation {}

// 父类的注解：子类能继承
@ServiceAnnotation
class ParentClass {}
class ChildClass extends ParentClass {}
ChildClass.class.isAnnotationPresent(ServiceAnnotation.class); // true ✅

// 接口的注解：实现类不继承
@ServiceAnnotation
interface MyInterface {}
class MyImpl implements MyInterface {}
MyImpl.class.isAnnotationPresent(ServiceAnnotation.class); // false ❌（接口不传递）

// 父类方法的注解：子类覆盖后读不到
class Parent {
    @Transactional
    public void save() {}
}
class Child extends Parent {
    @Override
    public void save() {}  // 覆盖后，这个方法上的 @Transactional 消失了
}
// Child.class.getMethod("save").getAnnotation(Transactional.class) == null ❌
```

Spring AOP 处理这个问题的方式：向上查找父类和接口的方法注解（`AnnotationUtils.findAnnotation()`）。

### 陷阱三：在接口方法上标注注解，代理可能读不到

```java
// CGLIB 代理读取注解时的坑
interface UserService {
    @Cacheable("users")  // 注解在接口方法上
    User getUser(Long id);
}

class UserServiceImpl implements UserService {
    public User getUser(Long id) { ... }  // 实现类方法上没有注解
}

// CGLIB 代理：基于子类，method.getAnnotation() 查的是实现类方法 → null
// JDK 代理：基于接口，method.getAnnotation() 查的是接口方法 → 能读到
```

Spring 的解决方案：`AnnotationUtils.findAnnotation()` 会同时查找方法本身、接口方法和父类方法。

### 陷阱四：注解处理的性能开销

反射读取注解有一定开销，高频调用时需要缓存：

```java
// ❌ 每次方法调用都反射读取注解
public Object invoke(Object proxy, Method method, Object[] args) {
    if (method.isAnnotationPresent(Cacheable.class)) { ... }  // 每次都反射
}

// ✅ 启动时缓存注解信息
private final Map<Method, Cacheable> cacheableCache = new ConcurrentHashMap<>();

public Object invoke(...) {
    Cacheable cacheable = cacheableCache.computeIfAbsent(method,
        m -> m.getAnnotation(Cacheable.class));
    if (cacheable != null) { ... }
}
```

### 陷阱五：注解不能继承注解

```java
// ❌ 编译错误：注解不能继承其他注解
@interface MyAnnotation extends BaseAnnotation { }

// ✅ 替代方案：注解组合（元注解）
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
@Component  // 把 @Component 作为元注解附加到自定义注解上
public @interface Service {
    String value() default "";
}
// Spring 的 @Service、@Repository、@Controller 都是这样组合 @Component 的
```

## 4.3 注解与枚举的结合

`ValidationFramework.java → @In` 注解提供了字符串枚举约束，但更类型安全的做法是与 Java 枚举结合：

```java
// 当前实现：字符串数组（易出错）
@In(value = {"male", "female"}, message = "性别只能是male或female")
private String gender;

// 更好的实践：使用枚举（编译时约束）
enum Gender { MALE, FEMALE }

@NotNull
private Gender gender;  // 类型本身就是约束，无需额外注解
```

## 4.4 本章总结

- **最大陷阱**：忘写 `@Retention(RUNTIME)`，导致注解运行时静默消失
- **@Inherited 范围**：只传递类级注解，不传递接口实现类注解，不传递方法覆盖
- **接口注解 vs CGLIB**：CGLIB 代理读接口方法注解需特殊处理，Spring 用 `AnnotationUtils` 解决
- **性能**：高频调用时缓存 `Method → Annotation` 映射，避免重复反射
- **注解不能继承**：用注解组合（元注解叠加）实现注解的层次化
