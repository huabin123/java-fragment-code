# 第五章：注解处理器与编译时代码生成

## 5.1 运行时注解 vs 编译时注解处理

目前已讨论的注解处理都是**运行时反射**模式（RUNTIME Retention）。编译时注解处理器（APT, Annotation Processing Tool）是另一条路：

| 维度 | 运行时反射 | 编译时 APT |
|------|-----------|----------|
| 触发时机 | 程序运行时 | `javac` 编译阶段 |
| 性能 | 有反射开销 | 零运行时开销（已生成代码）|
| 典型框架 | Spring IoC/AOP | Lombok, MapStruct, Dagger2 |
| 调试难度 | 直接调试 | 需要查看生成的代码 |
| 灵活性 | 极高（运行时决策）| 较低（编译时确定）|

---

## 5.2 编译时注解处理器的工作机制

APT 处理器实现 `javax.annotation.processing.AbstractProcessor`，在 `javac` 编译时被自动调用：

```
源码(.java) → javac 解析 → 触发 APT 处理器 → 生成新 .java 文件 → 再次编译 → .class 文件
```

**Lombok 的工作原理**：

```java
// 你写的代码
@Data
public class User {
    private String name;
    private int age;
}

// Lombok APT 在编译时生成（不可见，直接进入字节码）：
public class User {
    private String name;
    private int age;

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public int getAge() { return age; }
    public void setAge(int age) { this.age = age; }

    @Override public boolean equals(Object o) { ... }
    @Override public int hashCode() { ... }
    @Override public String toString() { ... }
}
```

## 5.3 ValidationFramework 与 Hibernate Validator 的对比

`ValidationFramework.java` 是运行时注解处理的典型。对比工业级的 Hibernate Validator：

| 特性 | ValidationFramework（示例）| Hibernate Validator |
|------|--------------------------|---------------------|
| 实现层次 | 运行时反射 | 运行时反射 |
| 分组验证 | 支持（`groups` 属性）| 支持 |
| 级联验证 | 支持（`@Valid`）| 支持 |
| 自定义验证器 | 不支持 | 支持 `ConstraintValidator<A, T>` |
| 国际化消息 | 不支持 | 支持 MessageSource |
| 方法参数验证 | 不支持 | 支持 |

在 Spring Boot 项目中引入 `spring-boot-starter-validation`，就获得了完整的 Hibernate Validator 支持：

```java
// Spring Boot 中的使用（与 ValidationFramework 注解名完全一致！）
@NotBlank(message = "用户名不能为空")
@Size(min = 3, max = 20)
private String username;

@Email
private String email;

// Controller 中启用验证
@PostMapping("/register")
public Result register(@Valid @RequestBody UserRegisterForm form) {
    // 验证失败自动抛出 MethodArgumentNotValidException
}
```

## 5.4 自定义注解的完整开发模式

基于 `ValidationFramework.java` 的模式，总结一个完整的自定义注解开发流程：

```java
// Step 1: 定义注解
@Retention(RetentionPolicy.RUNTIME)  // 必须是 RUNTIME！
@Target(ElementType.FIELD)
@Documented
public @interface PhoneNumber {
    String message() default "手机号格式不正确";
    String regex() default "^1[3-9]\\d{9}$";
}

// Step 2: 实现处理器（验证器/处理逻辑）
public class PhoneNumberValidator {
    public boolean validate(String value, PhoneNumber annotation) {
        if (value == null) return true;  // null 由 @NotNull 处理
        return value.matches(annotation.regex());
    }
}

// Step 3: 集成到框架（注册到验证器）
if (field.isAnnotationPresent(PhoneNumber.class)) {
    PhoneNumber ann = field.getAnnotation(PhoneNumber.class);
    if (!new PhoneNumberValidator().validate((String) value, ann)) {
        result.addError(fieldName, ann.message());
    }
}

// Step 4: 使用
public class UserForm {
    @PhoneNumber(message = "请输入有效的手机号")
    private String mobile;
}
```

## 5.5 工业级注解框架的设计规律

研究 Spring、JPA、Hibernate Validator 等框架，可以总结出注解体系的设计规律：

**规律一：注解只声明意图，处理器决定行为**

```java
@Transactional  // 只是一个标记
void createUser() { ... }

// 行为完全由 TransactionInterceptor 决定：
// 开事务 → 执行方法 → 提交/回滚
```

**规律二：组合注解（元注解叠加）减少重复**

```java
// Spring 的 @Service 就是 @Component 的特化版
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Component  // 关键：把 @Component 作为元注解
public @interface Service {
    @AliasFor(annotation = Component.class)
    String value() default "";
}
```

**规律三：注解属性提供配置，默认值覆盖 80% 场景**

```java
@Transactional(
    propagation = Propagation.REQUIRED,  // 默认值，80% 场景
    isolation = Isolation.DEFAULT,       // 默认值
    timeout = -1,                        // 默认值（无超时）
    readOnly = false                     // 默认值
)
```

大多数使用只需要 `@Transactional`，无需任何参数。

## 5.6 本章总结

- **两种处理时机**：运行时反射（灵活，有开销）vs 编译时 APT（零开销，生成代码）
- **Lombok 原理**：APT 在编译阶段直接修改/生成字节码，`@Data` 等注解运行时不存在
- **Hibernate Validator**：工业级运行时验证框架，`ValidationFramework.java` 是其简化版
- **自定义注解三步**：定义注解（RUNTIME + Target）→ 实现处理器 → 集成到框架
- **设计规律**：注解声明意图，处理器实现行为；组合注解减少重复；属性提供配置默认值覆盖常用场景
