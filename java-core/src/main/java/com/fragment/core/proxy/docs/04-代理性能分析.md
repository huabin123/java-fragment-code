# 第四章：代理性能分析

## 4.1 性能测试的基准

`ProxyPerformanceDemo.java` 对四种调用方式进行了基准对比：

```
测试条件：百万次方法调用，JVM 充分预热后的稳定数据

直接调用（无代理）：    约  1ms（基准）
JDK 动态代理：         约  5ms（慢 ~5x）
CGLIB 代理（早期版本）：约  3ms（慢 ~3x）
CGLIB FastClass 优化：  约  2ms（慢 ~2x）
```

---

## 4.2 JDK 代理的性能开销来源

```java
// 每次调用 proxy.findById(1L) 的实际执行路径：

// 1. 代理类方法入口
public User findById(Long id) {
    // 2. 调用 InvocationHandler.invoke()
    return (User) h.invoke(this, m1, new Object[]{id});
    //                           ↑          ↑
    //                   Method 对象    装箱 Long→Object[]
}

// 3. InvocationHandler.invoke()
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    // 4. 反射调用目标方法
    return method.invoke(target, args);
    //     ↑ 反射开销：安全检查、参数类型检查、基本类型拆装箱
}
```

**主要开销**：
1. **参数装箱**：基本类型参数（int, long, boolean）在包装成 `Object[]` 时需要装箱
2. **反射调用**：`Method.invoke()` 有安全检查、参数类型验证开销
3. **对象创建**：`new Object[]{id}` 每次调用都创建新数组

---

## 4.3 CGLIB 为什么比 JDK 代理快？

CGLIB 使用 **FastClass 机制**避免了反射：

```java
// JDK 代理：每次都通过反射调用
method.invoke(target, args);  // 反射，有开销

// CGLIB：FastClass 直接索引方法，无反射
proxy.invokeSuper(obj, args);
// 内部：根据方法的 index 直接调用，类似于：
// switch(methodIndex) {
//     case 0: return ((Calculator)obj).add(args);
//     case 1: return ((Calculator)obj).multiply(args);
// }
```

FastClass 把方法调用转换成带 switch 的直接调用，完全绕过了 Java 反射机制。

---

## 4.4 JIT 优化对代理性能的影响

实际生产环境中，JVM 的 JIT 编译器会对热点代理调用进行**内联优化**，性能差距大幅缩小：

```
JIT 充分优化后（百万次调用，JVM 运行数分钟后）：

直接调用：     ~1ms
JDK 代理：     ~2ms（慢约 2x，JIT 优化了反射）
CGLIB：        ~1.5ms（慢约 1.5x）
```

**结论**：在 JVM 充分预热的生产环境中，代理开销通常可以忽略不计。性能问题往往来自业务逻辑（数据库查询、网络请求），而非代理本身。

## 4.5 实际场景中的性能建议

`ProxyPerformanceDemo.java` 的测试揭示了以下实践原则：

### 建议一：避免在代理 Handler 中做重量级操作

```java
// ❌ Handler 里做了太多事，抵消了代理的价值
@Override
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    // 每次调用都去数据库查权限，性能灾难
    Permission perm = database.loadPermission(currentUser, method.getName());
    if (!perm.isAllowed()) throw new AccessDeniedException();

    return method.invoke(target, args);
}

// ✅ 权限信息缓存起来，Handler 只做内存查询
@Override
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    String key = currentUser + ":" + method.getName();
    Permission perm = permissionCache.computeIfAbsent(key, k -> database.loadPermission(...));
    if (!perm.isAllowed()) throw new AccessDeniedException();

    return method.invoke(target, args);
}
```

### 建议二：缓存 Method 对象，避免重复反射查找

```java
// Handler 中缓存 Method → 决策结果，避免每次都 method.getName() + 判断
private final Map<Method, Boolean> needsCacheMap = new ConcurrentHashMap<>();

@Override
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    boolean needsCache = needsCacheMap.computeIfAbsent(method,
        m -> m.isAnnotationPresent(Cacheable.class));

    if (needsCache) { /* 走缓存逻辑 */ }
    return method.invoke(target, args);
}
```

### 建议三：优先 CGLIB 用于高频调用

对于调用频率极高的核心服务，在 Spring 中可以强制使用 CGLIB：

```java
@SpringBootApplication
@EnableAspectJAutoProxy(proxyTargetClass = true)  // 全局强制 CGLIB
public class App {}

// 或者单个类级别
@Component
@Scope(proxyMode = ScopedProxyMode.TARGET_CLASS)  // 该类强制 CGLIB
public class HotService {}
```

## 4.6 代理创建的开销 vs 调用的开销

两个完全不同的性能维度：

| 阶段 | JDK 代理 | CGLIB |
|------|---------|-------|
| 创建代理对象 | 较快（读缓存或简单生成）| 较慢（需要 ASM 生成字节码）|
| 每次方法调用 | 较慢（反射）| 较快（FastClass）|

**实践**：代理对象应该是单例的（Spring Bean 默认就是），创建开销只有一次；方法调用开销才是关键指标。

---

## 4.7 本章总结

- **JDK 代理开销**：参数装箱 + 反射调用，百万次调用约慢 5 倍（未优化）
- **CGLIB 更快的原因**：FastClass 机制，switch 直接索引，无反射
- **JIT 影响**：生产环境 JIT 充分优化后，两者差距缩小到 1.5-2x，通常可忽略
- **真正的性能陷阱**：Handler 中做重量级操作（DB查询/网络请求）
- **缓存 Method 信息**：避免每次调用都重复反射查找注解/方法名
