# 第四章：内存泄漏问题与最佳实践

## 引言

本章将深入分析ThreadLocal内存泄漏的完整案例，总结最佳实践，帮助你在实际项目中正确使用ThreadLocal，避免内存泄漏陷阱。

## 1. 内存泄漏完整案例分析

### 1.1 问题1：典型的内存泄漏场景是什么？

**案例1：Web应用中的用户上下文**

```java
public class UserContextHolder {
    private static final ThreadLocal<User> userHolder = new ThreadLocal<>();
    
    public static void setUser(User user) {
        userHolder.set(user);
    }
    
    public static User getUser() {
        return userHolder.get();
    }
    
    public static void clear() {
        userHolder.remove();
    }
}

// Servlet容器使用线程池处理请求
public class UserServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        try {
            // 1. 设置用户上下文
            User user = getUserFromSession(req);
            UserContextHolder.setUser(user);
            
            // 2. 处理业务逻辑
            processRequest();
            
        } finally {
            // ❌ 如果忘记清理，导致内存泄漏
            // UserContextHolder.clear();
        }
    }
}
```

**内存泄漏分析**：

```
Tomcat线程池（默认200个线程）：

第1个请求（Thread-1）：
UserContextHolder.set(user1)
→ Thread-1.threadLocals.put(UserContextHolder, user1)

第2个请求（Thread-2）：
UserContextHolder.set(user2)
→ Thread-2.threadLocals.put(UserContextHolder, user2)

...

第201个请求（复用Thread-1）：
UserContextHolder.set(user201)
→ Thread-1.threadLocals.put(UserContextHolder, user201)
→ user1仍然在内存中（未被清理）

问题：
- 200个线程，每个线程持有一个User对象
- 如果User对象很大（包含大量数据），内存占用严重
- 线程池的线程永远不结束，User对象永远无法被GC
```

**案例2：数据库连接泄漏**

```java
public class ConnectionManager {
    private static final ThreadLocal<Connection> connHolder = new ThreadLocal<>();
    
    public static Connection getConnection() throws SQLException {
        Connection conn = connHolder.get();
        if (conn == null) {
            conn = DriverManager.getConnection("jdbc:mysql://localhost:3306/db");
            connHolder.set(conn);
        }
        return conn;
    }
    
    public static void closeConnection() {
        Connection conn = connHolder.get();
        if (conn != null) {
            try {
                conn.close();
            } catch (SQLException e) {
                e.printStackTrace();
            }
            // ❌ 忘记remove
            // connHolder.remove();
        }
    }
}

// 使用示例
public class UserService {
    public void updateUser(User user) {
        try {
            Connection conn = ConnectionManager.getConnection();
            // 执行SQL
            PreparedStatement ps = conn.prepareStatement("UPDATE users SET name = ?");
            ps.setString(1, user.getName());
            ps.executeUpdate();
        } catch (SQLException e) {
            e.printStackTrace();
        } finally {
            ConnectionManager.closeConnection();
        }
    }
}
```

**内存泄漏分析**：

```
问题：
1. Connection对象被close()，但仍然被ThreadLocal持有
2. 数据库连接池无法回收这些连接
3. 最终导致连接池耗尽

正确做法：
finally {
    ConnectionManager.closeConnection();
    connHolder.remove(); // 必须remove
}
```

```java
public class ThreadLocalMonitor {
    /**
     * 获取当前线程的ThreadLocal数量
     */
    public static int getThreadLocalCount() {
        try {
            Thread thread = Thread.currentThread();
            Field field = Thread.class.getDeclaredField("threadLocals");
            field.setAccessible(true);
            
            Object threadLocalMap = field.get(thread);
            if (threadLocalMap == null) {
                return 0;
            }
            
            Field tableField = threadLocalMap.getClass().getDeclaredField("table");
            tableField.setAccessible(true);
            Object[] table = (Object[]) tableField.get(threadLocalMap);
            
            int count = 0;
            for (Object entry : table) {
                if (entry != null) {
                    count++;
                }
            }
            return count;
        } catch (Exception e) {
            return -1;
        }
    }
    
    /**
     * 打印ThreadLocal信息
     */
    public static void printThreadLocalInfo() {
        int count = getThreadLocalCount();
        System.out.println("当前线程ThreadLocal数量: " + count);
        System.out.println("线程名称: " + Thread.currentThread().getName());
    }
}

// 使用示例
public class MonitorDemo {

    public static void main(String[] args) {
        ThreadLocalMonitor.printThreadLocalInfo(); // 0
        
        holder.set(new byte[1024 * 1024]); // 1MB
        ThreadLocalMonitor.printThreadLocalInfo(); // 1
        
        holder.remove();
        ThreadLocalMonitor.printThreadLocalInfo(); // 0
    }
}
```

## 2. ThreadLocal最佳实践

### 2.1 问题3：如何正确使用ThreadLocal？

**最佳实践1：使用try-finally保证remove**

```java
// ✅ 正确示例
public class BestPractice1 {
    private static final ThreadLocal<User> userHolder = new ThreadLocal<>();

        try {
            // 设置值
            userHolder.set(user);
            
            // 执行业务逻辑
            doSomething();

            // 必须在finally中remove

        }
```

**重要说明：withInitial不能自动清理内存**

很多开发者误以为使用`withInitial`可以自动清理ThreadLocal，**这是错误的**！

```java
public class WithInitialMisconception {
    private static final ThreadLocal<User> userHolder = 
        ThreadLocal.withInitial(() -> new User());
    
    public void handleRequest() {
        // ❌ 错误认知：以为withInitial会自动清理
        User user = userHolder.get();
        user.setName("张三");
        // 没有调用remove()，仍然会内存泄漏！
    }
}
```

**原理分析：withInitial只是延迟初始化，不负责清理**

让我们看看`withInitial`的源码实现：

```java
// ThreadLocal.java
public static <S> ThreadLocal<S> withInitial(Supplier<? extends S> supplier) {
    return new SuppliedThreadLocal<>(supplier);
}

// SuppliedThreadLocal内部类
static final class SuppliedThreadLocal<T> extends ThreadLocal<T> {
    private final Supplier<? extends T> supplier;

    SuppliedThreadLocal(Supplier<? extends T> supplier) {
        this.supplier = Objects.requireNonNull(supplier);
    }

    @Override
    protected T initialValue() {
        return supplier.get();
    }
}
```

**工作原理**：

```
1. 第一次调用get()时：
   userHolder.get()
   → ThreadLocalMap中没有值
   → 调用initialValue()
   → 执行supplier.get()创建对象
   → 将对象存入ThreadLocalMap
   → 返回对象

2. 后续调用get()时：
   userHolder.get()
   → ThreadLocalMap中已有值
   → 直接返回（不会再次创建）

3. 关键点：
   - withInitial只在首次get()时创建对象
   - 创建后对象一直存在ThreadLocalMap中
   - 不会自动清理，必须手动调用remove()
```

**内存泄漏演示**：

```java
public class WithInitialMemoryLeakDemo {
    private static final ThreadLocal<byte[]> dataHolder = 
        ThreadLocal.withInitial(() -> new byte[1024 * 1024]); // 1MB
    
    public static void main(String[] args) throws Exception {
        ExecutorService executor = Executors.newFixedThreadPool(10);
        
        // 提交100个任务
        for (int i = 0; i < 100; i++) {
            executor.execute(() -> {
                // 获取数据（首次会创建1MB数组）
                byte[] data = dataHolder.get();
                System.out.println("处理数据: " + data.length);
                
                // ❌ 没有调用remove()
                // 即使使用了withInitial，仍然会内存泄漏
            });
        }
        
        executor.shutdown();
        executor.awaitTermination(1, TimeUnit.MINUTES);
        
        // 此时10个线程各持有1MB数据，共10MB无法释放
        System.gc();
        Thread.sleep(1000);
        
        // 使用VisualVM查看，会发现10MB内存无法回收
    }
}
```

**正确使用方式：withInitial + remove()**

```java
public class CorrectUsage {
    private static final ThreadLocal<SimpleDateFormat> sdfHolder = 
        ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
    
    // ✅ 方式1：每次使用后立即清理
    public String formatDate1(Date date) {
        try {
            return sdfHolder.get().format(date);
        } finally {
            sdfHolder.remove(); // 必须清理
        }
    }
    
    // ✅ 方式2：在请求结束时统一清理（适用于Web应用）
    public String formatDate2(Date date) {
        // 只使用，不清理
        return sdfHolder.get().format(date);
        // 在Filter中统一清理所有ThreadLocal
    }
}

// Web应用中的统一清理
@WebFilter("/*")
public class ThreadLocalCleanupFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        try {
            chain.doFilter(request, response);
        } finally {
            // 统一清理（即使使用了withInitial也需要清理）
            CorrectUsage.sdfHolder.remove();
        }
    }
}
```

**withInitial的真正优势**：

```java

// 优势3：延迟创建，节省资源
```

**总结**：

| 特性 | withInitial | 手动set | 说明 |
|------|------------|---------|------|
| 自动初始化 | ✅ | ❌ | 首次get()时自动创建 |
| 代码简洁性 | ✅ | ❌ | Lambda表达式更简洁 |
| 避免空指针 | ✅ | ❌ | 不需要判空 |
| 延迟创建 | ✅ | ❌ | 按需创建，节省资源 |
| **自动清理** | **❌** | **❌** | **都需要手动remove()** |
| 内存泄漏风险 | ⚠️ | ⚠️ | 不调用remove()都会泄漏 |

**关键要点**：
1. ❌ **withInitial不能自动清理内存**
2. ✅ **withInitial只是提供了便捷的初始化方式**
3. ⚠️ **无论是否使用withInitial，都必须调用remove()**
4. 💡 **withInitial适合需要默认值的场景，但清理责任不变**

### 2.2 问题4：线程池场景下的特殊注意事项？

**问题：线程复用导致的数据污染**

```java
public class ThreadPoolProblem {
    private static ThreadLocal<String> holder = new ThreadLocal<>();
    private static ExecutorService executor = Executors.newFixedThreadPool(2);
    
    public static void main(String[] args) throws InterruptedException {
        // 任务1：设置值
        executor.execute(() -> {
            holder.set("task1 value");
            System.out.println("Task1: " + holder.get());
            // 忘记remove
        });
        
        Thread.sleep(100);
        
        // 任务2：期望是null，但可能获取到task1的值
        executor.execute(() -> {
            String value = holder.get();
            System.out.println("Task2: " + value); // 可能输出"task1 value"
        });
    }
}
```

**输出**：

```
Task1: task1 value
Task2: task1 value  ← 错误！获取到了上一个任务的值
```

---

**解决方案1：每个任务开始时重置**

```java
public class Solution1 {
    private static ThreadLocal<String> holder = new ThreadLocal<>();
    
    public void executeTask() {
        try {
            // 开始时重置
            holder.remove();
            
            // 设置新值
            holder.set("new value");
            
            // 执行业务逻辑
            doSomething();
            
        } finally {
            // 结束时清理
            holder.remove();
        }
    }
}
```

### 3.1 问题5：ThreadLocal的常见陷阱有哪些？

## 4. ThreadLocal使用检查清单

### 4.1 问题6：如何确保正确使用ThreadLocal？

**使用前检查清单**：

```
□ 1. 是否真的需要ThreadLocal？
   - 能否通过参数传递解决？
   - 能否使用其他方式（如Request Scope）？

□ 2. 是否使用了try-finally？
   - set()后是否在finally中remove()？
   - 即使发生异常也能保证remove()？

□ 3. 是否避免了大对象？
   - 存储的对象是否足够小？
   - 能否只存储ID而非完整对象？

□ 4. 是否考虑了线程池场景？
   - 线程复用时是否会数据污染？
   - 是否在任务开始时重置？

□ 5. 是否考虑了父子线程传递？
   - 是否需要InheritableThreadLocal？
   - 是否需要TransmittableThreadLocal？

□ 6. 是否添加了监控？
   - 是否能检测内存泄漏？
   - 是否能统计ThreadLocal数量？

□ 7. 是否编写了单元测试？
   - 是否测试了remove()逻辑？
   - 是否测试了异常情况？
```

## 5. 性能优化建议

### 5.1 问题7：如何优化ThreadLocal的性能？

**优化1：减少ThreadLocal数量**

```java
// ❌ 不推荐：多个ThreadLocal
public class MultipleThreadLocal {
    private static ThreadLocal<String> userId = new ThreadLocal<>();
    private static ThreadLocal<String> userName = new ThreadLocal<>();
    private static ThreadLocal<String> userRole = new ThreadLocal<>();
}

// ✅ 推荐：使用一个ThreadLocal存储上下文对象
public class SingleThreadLocal {
    private static ThreadLocal<UserContext> context = new ThreadLocal<>();
    
    public static class UserContext {
        private String userId;
        private String userName;
        private String userRole;
        // getters and setters
    }
}
```

**优化2：延迟初始化**

```java
// ❌ 不推荐：立即初始化
public class EagerInit {
    private static ThreadLocal<Connection> connHolder = 
        ThreadLocal.withInitial(() -> {
            // 即使不使用，也会创建连接
            return createConnection();
        });
}

// ✅ 推荐：按需初始化
public class LazyInit {
    private static ThreadLocal<Connection> connHolder = new ThreadLocal<>();
    
    public static Connection getConnection() {
        Connection conn = connHolder.get();
        if (conn == null) {
            conn = createConnection();
            connHolder.set(conn);
        }
        return conn;
    }
}
```

```java
// 如果对象创建开销大，考虑使用对象池
public class ObjectPoolThreadLocal {
    private static ThreadLocal<ObjectPool<ExpensiveObject>> poolHolder = 
        ThreadLocal.withInitial(() -> new ObjectPool<>(ExpensiveObject::new));
    
    public static ExpensiveObject borrowObject() {

    public static void returnObject(ExpensiveObject obj) {
        poolHolder.get().returnObject(obj);
    }
```

## 6. 核心问题总结

### Q1: ThreadLocal内存泄漏的根本原因是什么？
**A**: 使用后不调用remove()，导致value的强引用链路无法断开，特别是在线程池场景下。

### Q2: 如何检测ThreadLocal内存泄漏？
**A**: 使用JProfiler/VisualVM查看Heap Dump，或使用代码监控ThreadLocal数量。

### Q3: ThreadLocal的最佳实践是什么？
**A**: 
1. 使用try-finally保证remove()
2. 使用withInitial提供默认值
3. 避免存储大对象
4. 线程池场景下注意数据污染

### Q4: 线程池场景下有什么特殊注意事项？
**A**: 线程复用可能导致数据污染，需要在任务开始时重置，结束时清理。

### Q5: ThreadLocal的常见陷阱有哪些？
**A**: 
1. 父子线程无法传递值
2. 在static代码块中初始化
3. 在异步回调中使用
4. 在单例Bean中使用

### Q6: 如何优化ThreadLocal的性能？
**A**: 
1. 减少ThreadLocal数量
2. 延迟初始化
3. 使用对象池
