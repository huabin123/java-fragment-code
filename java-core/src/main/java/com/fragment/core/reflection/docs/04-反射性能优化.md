# 第四章：反射性能优化

## 4.1 反射的性能开销来源

```java
// ReflectionPerformanceDemo.java 的基准测试揭示了各阶段开销

// 开销1：Class.forName() / getDeclaredField() / getDeclaredMethod()
// 需要在类元数据中查找，有哈希查找和字符串比较开销
// 建议：启动时查找一次，缓存 Field/Method 对象

// 开销2：setAccessible(true)
// 每次调用都有安全检查（SecurityManager）
// 建议：查找时只调用一次，缓存已设置 accessible 的对象

// 开销3：method.invoke() / field.get()
// 参数装箱（基本类型包装为 Object）
// 方法查找的 native 调用（JIT 优化前）
// 安全访问检查（已 setAccessible 则跳过）
```

## 4.2 缓存是最重要的优化手段

`ReflectionPerformanceDemo.java` 对比了有缓存和无缓存的性能差距：

```java
// ❌ 无缓存：每次调用都重新查找 Field/Method
public void setField(Object obj, String fieldName, Object value) throws Exception {
    Field field = obj.getClass().getDeclaredField(fieldName);  // 每次查找！
    field.setAccessible(true);  // 每次设置！
    field.set(obj, value);
}

// ✅ 有缓存：Field/Method 对象只查找一次
private final Map<String, Field> fieldCache = new ConcurrentHashMap<>();

public void setField(Object obj, String fieldName, Object value) throws Exception {
    String cacheKey = obj.getClass().getName() + "#" + fieldName;
    Field field = fieldCache.computeIfAbsent(cacheKey, k -> {
        try {
            Field f = obj.getClass().getDeclaredField(fieldName);
            f.setAccessible(true);  // 只调用一次！
            return f;
        } catch (NoSuchFieldException e) {
            throw new RuntimeException(e);
        }
    });
    field.set(obj, value);
}
```

**性能对比（百万次调用）**：

```
无缓存（每次 getDeclaredField + setAccessible）：~800ms
有缓存（直接用缓存的 Field）：                  ~120ms
直接字段访问（无反射）：                         ~10ms
```

## 4.3 MethodHandle：比反射更快的替代方案（JDK 7+）

`ReflectionPerformanceDemo.java` 对比了 `MethodHandle` 与传统反射：

```java
// 传统反射
Method method = String.class.getDeclaredMethod("length");
int len = (int) method.invoke("hello");

// MethodHandle：编译器更友好，JIT 可以内联优化
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle mh = lookup.findVirtual(String.class, "length",
    MethodType.methodType(int.class));
int len = (int) mh.invoke("hello");

// MethodHandle 的优势：
// 1. JIT 可以内联（method.invoke 通常不能被内联）
// 2. 更接近直接调用的性能（充分预热后）
// 3. 更好的类型安全（MethodType 在构造时验证）
```

**性能对比（百万次，JIT 充分预热后）**：

```
直接调用：          ~5ms（基准）
MethodHandle：      ~8ms（1.6x，JIT 内联后接近直接调用）
Method.invoke()：   ~30ms（6x，反射开销）
```

## 4.4 实战缓存框架设计

```java
// ReflectionPerformanceDemo.java → ReflectionCache（框架级缓存设计）
public class ReflectionCache {
    // 按类缓存其所有字段（含父类）
    private static final Map<Class<?>, List<Field>> fieldCache = new ConcurrentHashMap<>();
    // 按类+方法签名缓存方法对象
    private static final Map<String, Method> methodCache = new ConcurrentHashMap<>();

    public static List<Field> getAllFields(Class<?> clazz) {
        return fieldCache.computeIfAbsent(clazz, c -> {
            List<Field> fields = new ArrayList<>();
            while (c != null && c != Object.class) {
                for (Field f : c.getDeclaredFields()) {
                    f.setAccessible(true);
                    fields.add(f);
                }
                c = c.getSuperclass();
            }
            return Collections.unmodifiableList(fields);
        });
    }

    public static Method getMethod(Class<?> clazz, String name, Class<?>... paramTypes) {
        String key = clazz.getName() + "#" + name + Arrays.toString(paramTypes);
        return methodCache.computeIfAbsent(key, k -> {
            try {
                Method m = clazz.getDeclaredMethod(name, paramTypes);
                m.setAccessible(true);
                return m;
            } catch (NoSuchMethodException e) {
                throw new RuntimeException(e);
            }
        });
    }
}
```

## 4.5 Lambda 方法引用替代反射（JDK 8+）

对于已知固定方法签名的场景，可以用函数式接口替代反射：

```java
// ReflectionPerformanceDemo.java → 函数式替代方案
// 场景：需要频繁调用某个 getter 方法

// ❌ 反射方式（每次有开销）
Method getName = User.class.getMethod("getName");
String name = (String) getName.invoke(user);

// ✅ 方法引用方式（编译成直接调用）
Function<User, String> getName = User::getName;
String name = getName.apply(user);

// ✅ 用 LambdaMetafactory 动态生成函数式接口（最快的动态方式）
// 可以在运行时把 Method 转换为无反射开销的函数式调用
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle handle = lookup.findVirtual(User.class, "getName",
    MethodType.methodType(String.class));
CallSite callSite = LambdaMetafactory.metafactory(
    lookup, "apply",
    MethodType.methodType(Function.class),
    MethodType.methodType(Object.class, Object.class),
    handle,
    MethodType.methodType(String.class, User.class)
);
Function<User, String> fastGetter = (Function<User, String>) callSite.getTarget().invoke();
// 之后调用 fastGetter.apply(user) 的性能接近直接调用
```

## 4.6 本章总结

- **最重要的优化**：缓存 `Field`/`Method` 对象（`computeIfAbsent`），同时缓存 `setAccessible(true)` 的调用
- **性能量级**：无缓存反射 ~800ms/百万次，有缓存 ~120ms，直接调用 ~10ms
- **MethodHandle**：JDK 7+ 的更快替代，JIT 可内联，充分预热后接近直接调用
- **框架缓存设计**：按 `Class` 或 `Class#methodName` 作为 key，`ConcurrentHashMap` 存储
- **LambdaMetafactory**：把 `MethodHandle` 转换为函数式接口，完全消除反射开销（复杂但极致性能）
