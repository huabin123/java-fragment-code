# 第一章：访问修饰符详解

## 1.1 为什么需要访问控制？

访问修饰符的本质是**封装**——面向对象三大特性之一。封装的目的不是藏代码，而是：

1. **隐藏实现细节**：调用方只依赖接口，实现可以随时重构
2. **防止误用**：内部辅助方法不应被外部调用
3. **建立边界**：模块之间通过有限的公共接口交互，降低耦合

---

## 1.2 四个访问级别：精确的范围界定

```
private < (default/package) < protected < public
  ↑最窄                                    ↑最宽
```

| 修饰符 | 本类 | 同包 | 子类（不同包）| 其他 |
|--------|------|------|-------------|------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| `(default)` | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

---

## 1.3 各级别的设计意图

### private：类内部实现细节

```java
// AccessModifierDemo.java 中的典型用法
public class BankAccount {
    private double balance;        // 余额不能被外部直接修改
    private String accountNumber;  // 账号内部管理

    // 公开业务方法，内部逻辑用 private 辅助
    public void deposit(double amount) {
        validateAmount(amount);    // 私有方法：内部验证
        balance += amount;
        logTransaction(amount);    // 私有方法：内部日志
    }

    private void validateAmount(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("金额必须为正数");
    }
}
```

**原则**：类的内部实现细节、辅助方法都应该是 `private`，这是最安全的默认选择。

### default（包级别）：包内部协作

```java
// 同包的类可以访问，跨包不行
// SamePackageAccess.java 演示了这种场景

class InternalHelper {     // 包级可见，不对外暴露
    void assist() { ... }
}

// 同包的类可以用，外部包无法访问
class Service {
    void doWork() {
        new InternalHelper().assist();  // ✅ 同包
    }
}
```

适用于：框架内部多个类之间的协作工具类，不想对用户暴露。

### protected：给子类的扩展点

```java
// protected 的核心用途：模板方法模式
public abstract class DataProcessor {
    // 公开的处理入口
    public final void process(Data data) {
        Data validated = validate(data);    // protected：子类可重写验证规则
        Data transformed = transform(validated);  // protected：子类可重写转换逻辑
        persist(transformed);
    }

    protected Data validate(Data data) {
        // 默认实现
        return data;
    }

    protected abstract Data transform(Data data);  // 强制子类实现
}
```

**注意**：`protected` 对同包也可见，这有时会造成意外——本意是给子类的扩展点，却被同包其他类随意调用。

### public：对外公开的 API

```java
// 设计原则：只暴露必要的公共接口
public class UserService {
    public User createUser(String username) { ... }  // 公开 API
    public User getUserById(Long id) { ... }          // 公开 API

    // 以下都应该是 private，不暴露
    private void validateUsername(String username) { ... }
    private void sendWelcomeEmail(User user) { ... }
    private User buildUserEntity(String username) { ... }
}
```

## 1.4 访问修饰符的几个反直觉细节

### 细节一：private 对同类的其他实例也可见

```java
public class Node {
    private int value;

    public int compareTo(Node other) {
        return this.value - other.value;  // ✅ 可以访问 other 的 private 字段！
        // 原因：访问控制是类级别的，不是对象级别的
    }
}
```

### 细节二：接口方法默认是 public abstract

```java
public interface Repository {
    User findById(Long id);         // 等价于 public abstract User findById(Long id)
    default void close() { ... }    // JDK 8+：接口可以有 default 实现
    static Repository noOp() { ... } // JDK 8+：接口可以有 static 方法
    // private 方法 JDK 9+ 才支持
}
```

### 细节三：子类 override 只能扩大访问权限

```java
class Parent {
    protected void doWork() { }
}

class Child extends Parent {
    @Override
    public void doWork() { }    // ✅ 扩大到 public
    // protected void doWork() { }  // ✅ 保持不变
    // private void doWork() { }    // ❌ 编译错误：缩小了访问权限
}
```

### 细节四：顶级类只能是 public 或 default

```java
public class Outer { }     // ✅ 顶级类
class PackageLevel { }     // ✅ 顶级类（default）
// private class Hidden { }  // ❌ 编译错误：顶级类不能是 private
// protected class Semi { }  // ❌ 编译错误：顶级类不能是 protected

// 内部类则不同，可以是任意访问级别
public class Outer {
    private class Inner { }   // ✅ 内部类可以是 private
}
```

---

## 1.5 最小权限原则

**设计原则**：赋予最小的必要访问权限，而非最大权限。

实践准则：
1. 字段始终 `private`，通过方法暴露（getter/setter 按需）
2. 类的辅助方法默认 `private`
3. 只有需要被子类重写的方法才用 `protected`
4. 只有需要对外暴露的方法才用 `public`
5. `default` 适合包内协作的工具类/方法

---

## 1.6 本章总结

- **四级访问**：private（类内）→ default（包内）→ protected（子类+包内）→ public（所有）
- **设计意图**：private 隐藏实现，default 包内协作，protected 提供扩展点，public 对外 API
- **最小权限原则**：始终从最小权限开始，需要时再放开
- **反直觉细节**：private 对同类其他实例可见，子类 override 只能扩权不能缩权
