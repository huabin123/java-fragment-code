# 第五章：线程池与最佳实践

## 5.1 为什么需要线程池？

直接 `new Thread()` 的三个问题：
1. **创建开销大**：每次 `new Thread()` 都需要 OS 创建内核线程，涉及内存分配（默认 512KB 栈）
2. **无法控制并发数**：100 个请求同时来，创建 100 个线程，内存耗尽
3. **资源无法复用**：任务完成后线程销毁，下次又要重新创建

线程池维护一组工作线程，任务通过队列提交，线程复用。

## 5.2 ThreadPoolExecutor 七个参数

```java
// ThreadPoolDemo.java → createCustomPool()
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    4,                                   // 1. corePoolSize：核心线程数（常驻）
    8,                                   // 2. maximumPoolSize：最大线程数
    60L,                                 // 3. keepAliveTime：空闲线程存活时间
    TimeUnit.SECONDS,                    // 4. unit：存活时间单位
    new ArrayBlockingQueue<>(100),       // 5. workQueue：任务队列
    new ThreadFactory() {                // 6. threadFactory：线程工厂（可命名）
        private int count = 0;
        public Thread newThread(Runnable r) {
            Thread t = new Thread(r, "worker-" + count++);
            t.setUncaughtExceptionHandler((thread, e) -> log.error("线程异常", e));
            return t;
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy()  // 7. rejectedHandler：拒绝策略
);
```

### 线程池的工作逻辑

```
提交任务
  ↓
当前线程数 < corePoolSize？
  → 是：创建新核心线程执行
  ↓
队列未满？
  → 是：加入队列等待
  ↓
当前线程数 < maximumPoolSize？
  → 是：创建临时线程执行
  ↓
执行拒绝策略
```

### 四种拒绝策略

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `AbortPolicy`（默认）| 抛出 `RejectedExecutionException` | 需要感知拒绝的场景 |
| `CallerRunsPolicy` | 调用者线程直接执行任务 | 不允许丢失任务 |
| `DiscardPolicy` | 静默丢弃新任务 | 允许丢失，如日志任务 |
| `DiscardOldestPolicy` | 丢弃队列中最旧的任务 | 实时性要求高 |

## 5.3 Executors 工厂方法及其陷阱

```java
// ThreadPoolDemo.java → demonstrateBuiltinPools()

// FixedThreadPool：固定大小，无界队列（LinkedBlockingQueue）
ExecutorService fixed = Executors.newFixedThreadPool(4);
// ❌ 风险：无界队列，任务积压可能导致 OOM

// CachedThreadPool：无核心线程，最大线程数 Integer.MAX_VALUE，60s 空闲销毁
ExecutorService cached = Executors.newCachedThreadPool();
// ❌ 风险：并发量大时创建大量线程，可能导致 OOM

// SingleThreadExecutor：单线程，无界队列
ExecutorService single = Executors.newSingleThreadExecutor();
// ❌ 风险：同 FixedThreadPool，无界队列 OOM

// ScheduledThreadPool：定时/周期任务
ScheduledExecutorService scheduled = Executors.newScheduledThreadPool(2);
// ❌ 风险：最大线程数 Integer.MAX_VALUE，同 CachedThreadPool
```

**阿里巴巴 Java 开发手册明确禁止使用 `Executors` 工厂方法**，原因就是无界队列和无限制线程数的 OOM 风险。

```java
// ✅ 始终使用 ThreadPoolExecutor 显式指定参数
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),  // 有界队列！
    namedThreadFactory,
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

## 5.4 线程池大小如何设定？

`ThreadPoolDemo.java` 提供了理论公式和实践建议：

### 理论公式

```
CPU 密集型任务：线程数 = CPU 核心数 + 1
  （+1 是为了防止偶发的 Page Fault 导致线程空闲时有备用线程）

I/O 密集型任务：线程数 = CPU 核心数 × (1 + 等待时间/计算时间)
  典型 Web 应用（等待时间/计算时间 ≈ 10）：线程数 = CPU 核心数 × 10
```

### 实践建议

```java
int cpuCores = Runtime.getRuntime().availableProcessors();

// CPU 密集：图像处理、数据计算
int cpuBound = cpuCores + 1;

// I/O 密集：数据库查询、HTTP 请求（假设等待比例 90%，即 9:1）
int ioBound = cpuCores * (1 + 9);  // = cpuCores * 10

// 混合任务：需要根据实际监控调整
// 核心指标：CPU 利用率、队列长度、任务等待时间、拒绝数量
```

## 5.5 线程池的优雅关闭

```java
// ThreadPoolDemo.java → gracefulShutdown()
public void shutdownGracefully(ExecutorService pool) {
    // 第一步：停止接受新任务，等待已提交任务完成
    pool.shutdown();

    try {
        // 第二步：等待最多 30 秒让现有任务完成
        if (!pool.awaitTermination(30, TimeUnit.SECONDS)) {
            // 第三步：超时后强制停止
            pool.shutdownNow();

            // 第四步：再等待 5 秒让中断响应完成
            if (!pool.awaitTermination(5, TimeUnit.SECONDS)) {
                System.err.println("线程池未能完全关闭！");
            }
        }
    } catch (InterruptedException e) {
        pool.shutdownNow();
        Thread.currentThread().interrupt();
    }
}
```

**shutdown vs shutdownNow**：
- `shutdown()`：不再接受新任务，等待现有任务完成（优雅关闭）
- `shutdownNow()`：向所有工作线程发送 `interrupt()`，尝试立即停止（强制关闭）

## 5.6 线程池监控

```java
// ThreadPoolDemo.java → monitorPool()
if (pool instanceof ThreadPoolExecutor) {
    ThreadPoolExecutor tpe = (ThreadPoolExecutor) pool;
    System.out.printf("活跃线程数: %d%n", tpe.getActiveCount());
    System.out.printf("当前线程数: %d%n", tpe.getPoolSize());
    System.out.printf("队列大小:   %d%n", tpe.getQueue().size());
    System.out.printf("已完成任务: %d%n", tpe.getCompletedTaskCount());
    System.out.printf("拒绝计数:   %d%n", rejectedCount.get());  // 自定义拒绝策略计数
}
```

**告警阈值建议**：
- 队列长度 > 80% 容量 → 告警，考虑扩容
- 活跃线程数 = 最大线程数持续超过 1 分钟 → 告警
- 拒绝数 > 0 → 告警（已开始丢弃任务）

## 5.7 本章总结

- **线程池七参数**：corePoolSize、maximumPoolSize、keepAlive、unit、workQueue、threadFactory、rejectedHandler
- **工作逻辑**：核心满 → 入队 → 临时线程 → 拒绝策略
- **禁用 Executors**：无界队列（FixedThreadPool/SingleThread）或无限线程（CachedThread/Scheduled）都有 OOM 风险
- **大小公式**：CPU 密集 = 核心数+1；I/O 密集 = 核心数×(1+等待比)
- **优雅关闭**：`shutdown()` → `awaitTermination(30s)` → `shutdownNow()` → `awaitTermination(5s)`
- **监控指标**：活跃线程数、队列长度、完成任务数、拒绝数
