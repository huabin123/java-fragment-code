# 第二章：线程同步与锁

## 2.1 为什么需要同步？

多个线程共享可变数据时，会发生**竞态条件（Race Condition）**：

```java
// ThreadSynchronizationDemo.java → 竞态条件演示
public class Counter {
    private int count = 0;  // 共享可变状态

    public void increment() {
        count++;  // 看起来是一步，实际是三步：读取 → 加一 → 写回
    }
}

// 两个线程同时执行 increment()，各执行 10000 次
// 期望结果：20000
// 实际结果：< 20000（每次不同）
// 原因：线程1读取 count=5，线程2读取 count=5，两个都写回 6，丢失一次更新
```

## 2.2 synchronized：Java 的基础同步机制

```java
// ThreadSynchronizationDemo.java → SynchronizedCounter
public class SynchronizedCounter {
    private int count = 0;

    // 方式1：同步方法（锁是 this 对象）
    public synchronized void increment() {
        count++;  // 同一时刻只有一个线程能执行
    }

    // 方式2：同步代码块（可以指定锁对象，粒度更细）
    public void incrementBlock() {
        synchronized (this) {
            count++;
        }
    }

    // 方式3：静态同步方法（锁是 Class 对象）
    public static synchronized void staticMethod() { ... }
}
```

### synchronized 的锁升级（JDK 6+）

```
无锁 → 偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（OS 互斥量）
         ↑              ↑                        ↑
    只有一个线程   多线程竞争但时间短        竞争激烈，线程挂起
```

- **偏向锁**：大多数情况下锁被同一线程持有，用线程ID标记，无额外开销
- **轻量级锁**：多线程轻度竞争，CAS 自旋等待，避免线程挂起
- **重量级锁**：竞争激烈，线程进入 OS 等待队列，涉及用户态/内核态切换

## 2.3 volatile：可见性保证

```java
// ThreadSynchronizationDemo.java → volatile 演示
public class VisibilityDemo {
    private volatile boolean running = true;  // volatile 保证可见性

    public void stop() {
        running = false;  // 写操作立即刷新到主内存
    }

    public void run() {
        while (running) {  // 每次读取都从主内存读，而不是 CPU 缓存
            doWork();
        }
    }
}

// 没有 volatile 时：
// 线程1修改了 running = false
// 但线程2的 CPU 缓存中 running 仍然是 true
// 线程2可能永远不会看到 running 变为 false！
```

**volatile 的两个保证**：
1. **可见性**：写操作立即对所有线程可见（强制刷新主内存）
2. **有序性**：禁止指令重排（happens-before 保证）

**volatile 不能保证**：原子性（`count++` 即使加了 volatile 也不是原子的）

### volatile vs synchronized

| 维度 | volatile | synchronized |
|------|---------|-------------|
| 可见性 | ✅ | ✅ |
| 有序性 | ✅ | ✅ |
| 原子性 | ❌ | ✅ |
| 性能 | 轻量（无锁）| 重量（有锁开销）|
| 适用场景 | 单写多读的标志位 | 复合操作、临界区 |

## 2.4 原子类：无锁的线程安全

```java
// ThreadSynchronizationDemo.java → AtomicCounter
import java.util.concurrent.atomic.*;

public class AtomicCounter {
    private AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();  // CAS 原子操作，无需加锁
    }

    public int get() { return count.get(); }
}

// 常用原子类
AtomicInteger  ai = new AtomicInteger(0);
AtomicLong     al = new AtomicLong(0L);
AtomicBoolean  ab = new AtomicBoolean(false);
AtomicReference<String> ar = new AtomicReference<>("initial");

// 原子更新操作
ai.incrementAndGet();          // ++i（先加后返回）
ai.getAndIncrement();          // i++（先返回后加）
ai.addAndGet(5);               // i += 5，返回新值
ai.compareAndSet(expected, updated);  // CAS：期望值匹配则更新
```

**CAS 原理**：Compare-And-Swap，底层是 CPU 指令 `cmpxchg`，无需 OS 级别锁，自旋重试直到成功。

## 2.5 ReentrantLock：更灵活的锁

```java
// ThreadSynchronizationDemo.java → ReentrantLockDemo
import java.util.concurrent.locks.*;

public class BankAccount {
    private double balance;
    private final ReentrantLock lock = new ReentrantLock();

    public void deposit(double amount) {
        lock.lock();  // 获取锁（不可中断）
        try {
            balance += amount;
        } finally {
            lock.unlock();  // 必须在 finally 中释放！
        }
    }

    // 可以尝试获取锁（不会无限等待）
    public boolean tryDeposit(double amount) {
        if (lock.tryLock()) {  // 尝试获取，立即返回
            try {
                balance += amount;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;  // 锁被占用，直接返回 false
    }

    // 可中断的锁等待
    public void interruptibleDeposit(double amount) throws InterruptedException {
        lock.lockInterruptibly();  // 等待过程中可以被中断
        try {
            balance += amount;
        } finally {
            lock.unlock();
        }
    }
}
```

**ReentrantLock vs synchronized**：

| 维度 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 语法 | 关键字，简洁 | 显式 lock/unlock，需要 finally |
| 可中断 | ❌ | ✅ `lockInterruptibly()` |
| 尝试获取 | ❌ | ✅ `tryLock()` |
| 公平锁 | ❌ | ✅ `new ReentrantLock(true)` |
| Condition | ❌ | ✅ `lock.newCondition()` |
| 性能 | JDK 6+ 已优化，差别小 | 高竞争下略好 |

## 2.6 本章总结

- **竞态条件**：多线程读-改-写共享状态，`count++` 实际是三步操作
- **synchronized**：方法锁（this 或 Class）/ 代码块锁，JDK 6+ 有锁升级优化
- **volatile**：保证可见性和有序性，不保证原子性；适合状态标志位
- **原子类**：基于 CAS，无锁原子操作，适合单变量的线程安全更新
- **ReentrantLock**：synchronized 的增强版，支持可中断、超时、公平、Condition
