# 第三章：线程协作与通信

## 3.1 wait/notify：经典的线程协作机制

`ThreadCooperationDemo.java` 用生产者-消费者模式演示了 `wait/notify` 的完整用法：

```java
// ThreadCooperationDemo.java → ProducerConsumer
public class BoundedBuffer<T> {
    private final Queue<T> buffer;
    private final int capacity;
    private final Object lock = new Object();

    public void put(T item) throws InterruptedException {
        synchronized (lock) {
            // ✅ 必须用 while 而不是 if！
            while (buffer.size() == capacity) {
                lock.wait();  // 缓冲区满，释放锁并等待
                // wait 返回后 buffer 可能再次满（spurious wakeup，虚假唤醒）
            }
            buffer.offer(item);
            lock.notifyAll();  // 唤醒所有等待的线程
        }
    }

    public T take() throws InterruptedException {
        synchronized (lock) {
            while (buffer.isEmpty()) {
                lock.wait();  // 缓冲区空，释放锁并等待
            }
            T item = buffer.poll();
            lock.notifyAll();  // 唤醒生产者
            return item;
        }
    }
}
```

**关键规则**：
1. **必须在 synchronized 块中调用** `wait/notify`，否则 `IllegalMonitorStateException`
2. **用 `while` 而不是 `if`**：防止虚假唤醒（spurious wakeup）——JVM 规范允许线程在没有被 notify 的情况下从 wait 返回
3. **优先用 `notifyAll`**：`notify` 只唤醒一个线程，可能唤醒了不该唤醒的线程；`notifyAll` 更安全

## 3.2 Condition：比 wait/notify 更强大

`Condition` 是 `ReentrantLock` 配套的条件变量，允许为同一个锁创建多个等待队列：

```java
// ThreadCooperationDemo.java → ConditionBoundedBuffer
public class ConditionBuffer<T> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();  // 队列未满时唤醒生产者
    private final Condition notEmpty = lock.newCondition();  // 队列非空时唤醒消费者

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (buffer.size() == capacity) {
                notFull.await();  // 等待"未满"信号
            }
            buffer.offer(item);
            notEmpty.signal();  // 只唤醒消费者（精确）
        } finally {
            lock.unlock();
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (buffer.isEmpty()) {
                notEmpty.await();  // 等待"非空"信号
            }
            T item = buffer.poll();
            notFull.signal();   // 只唤醒生产者（精确）
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

**Condition vs wait/notify**：

| 维度 | wait/notify | Condition |
|------|-------------|-----------|
| 等待队列 | 每个对象一个 | 每个锁可以多个 |
| 唤醒精度 | notifyAll 全部唤醒 | signal 精确唤醒特定队列 |
| 超时等待 | `wait(timeout)` | `await(timeout, unit)` |
| 可中断 | 可中断 | `awaitUninterruptibly()` 不可中断 |

## 3.3 CountDownLatch：等待多个线程完成

```java
// ThreadCooperationDemo.java → ParallelTaskDemo

// 场景：主线程等待 5 个子任务全部完成后汇总结果
CountDownLatch latch = new CountDownLatch(5);  // 计数器初始为 5

for (int i = 0; i < 5; i++) {
    final int taskId = i;
    executor.submit(() -> {
        try {
            doWork(taskId);  // 执行子任务
        } finally {
            latch.countDown();  // 任务完成，计数器 -1
        }
    });
}

latch.await();  // 主线程阻塞，直到计数器变为 0
System.out.println("所有任务完成！");

// 带超时的等待
boolean completed = latch.await(30, TimeUnit.SECONDS);
if (!completed) System.out.println("超时！部分任务未完成");
```

**特点**：一次性，计数到 0 后无法重置（想重用用 `CyclicBarrier`）。

## 3.4 CyclicBarrier：多个线程相互等待

```java
// ThreadCooperationDemo.java → PhaseBarrierDemo

// 场景：3 个线程必须同时到达某个同步点，才能一起继续执行下一阶段
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    // 所有线程都到达屏障时执行（在继续之前，可以做阶段汇总）
    System.out.println("=== 阶段 " + currentPhase + " 完成 ===");
});

Runnable worker = () -> {
    for (int phase = 0; phase < 3; phase++) {
        doPhaseWork(phase);  // 执行当前阶段工作
        try {
            barrier.await();  // 等待其他线程到达屏障
        } catch (BrokenBarrierException e) {
            // 其他线程被中断导致屏障破坏
        }
        // 所有线程都到达后，继续下一阶段
    }
};
```

**CountDownLatch vs CyclicBarrier**：

| 维度 | CountDownLatch | CyclicBarrier |
|------|---------------|---------------|
| 语义 | 等待 N 个事件完成 | N 个线程相互等待 |
| 可重用 | ❌ 一次性 | ✅ 可重置（cyclic）|
| 主线程参与 | 主线程等待，子线程 countDown | 所有线程地位平等 |
| 到达时触发 | 无 | 可设置 barrierAction |

## 3.5 Semaphore：控制并发数量

```java
// ThreadCooperationDemo.java → ConnectionPoolDemo

// 场景：数据库连接池，最多允许 10 个并发连接
Semaphore semaphore = new Semaphore(10);  // 10 个许可证

public Connection getConnection() throws InterruptedException {
    semaphore.acquire();  // 获取许可证，无许可证时阻塞
    try {
        return pool.borrowConnection();
    } catch (Exception e) {
        semaphore.release();  // 出错时归还许可证！
        throw e;
    }
}

public void returnConnection(Connection conn) {
    pool.returnConnection(conn);
    semaphore.release();  // 归还许可证，唤醒等待的线程
}

// 尝试获取（不阻塞）
if (semaphore.tryAcquire(100, TimeUnit.MILLISECONDS)) {
    // 100ms 内获取到许可证
} else {
    // 超时，服务降级或返回错误
}
```

## 3.6 本章总结

- **wait/notify 三要素**：在 synchronized 中调用；用 while 防虚假唤醒；优先 notifyAll
- **Condition**：每个锁多个等待队列，精确唤醒，`notFull/notEmpty` 分离生产者消费者等待
- **CountDownLatch**：主线程等待 N 个子任务完成，一次性
- **CyclicBarrier**：N 个线程相互等待到达同步点，可重用，支持到达时回调
- **Semaphore**：限制并发访问数量，连接池/限流的基础
