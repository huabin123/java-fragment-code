# 第四章：序列化与安全

## 4.1 Java 序列化的安全漏洞

Java 原生序列化是历史上最著名的安全问题之一。2015 年曝光的 Apache Commons Collections 漏洞让整个 Java 生态陷入恐慌：**任何接受反序列化数据的 Java 应用都可能被远程执行任意代码（RCE）**。

### 漏洞原理

```
攻击者构造恶意字节流
    → 服务端调用 ObjectInputStream.readObject()
    → 反序列化触发特定类的 readObject() 方法
    → 链式调用（Gadget Chain）最终执行 Runtime.exec("rm -rf /")
```

不需要目标系统有代码漏洞，只需要：
1. 系统接受反序列化输入
2. classpath 中有可利用的"gadget 类"（如 Apache Commons Collections）

```java
// 高风险场景1：接受网络输入并反序列化
ServerSocket server = new ServerSocket(9999);
Socket client = server.accept();
ObjectInputStream ois = new ObjectInputStream(client.getInputStream());
Object obj = ois.readObject();  // ❌ 危险！可能执行恶意代码

// 高风险场景3：Java RMI（Remote Method Invocation）
// RMI 传输的对象使用 Java 序列化，历史上多次被利用

// 高风险场景4：某些缓存框架
// 早期版本的 Spring Session + Redis，以及部分 Shiro 版本
```

## 4.3 安全防御：ObjectInputFilter（JDK 9+）

```java
// SerializationSecurityDemo.java → 使用 ObjectInputFilter 白名单

// 方式1：在创建 ObjectInputStream 时设置过滤器
ObjectInputStream ois = new ObjectInputStream(inputStream);
ois.setObjectInputFilter(filterInfo -> {
    Class<?> clazz = filterInfo.serialClass();
    if (clazz == null) return ObjectInputFilter.Status.UNDECIDED;

    // 白名单：只允许反序列化特定的类
    if (clazz == User.class || clazz == Order.class) {
        return ObjectInputFilter.Status.ALLOWED;
    }

    // 拒绝所有其他类
    return ObjectInputFilter.Status.REJECTED;
});

// 方式2：设置全局过滤器（JVM 级别）
ObjectInputFilter.Config.setSerialFilter(filterInfo -> {
    Class<?> clazz = filterInfo.serialClass();
    if (clazz == null) return ObjectInputFilter.Status.UNDECIDED;

    // 拒绝高危类
    String name = clazz.getName();
    if (name.startsWith("org.apache.commons.collections")
            || name.startsWith("com.sun.org.apache.xalan")) {
        return ObjectInputFilter.Status.REJECTED;
    }

    return ObjectInputFilter.Status.UNDECIDED;
});
```

## 4.4 根本解决方案：不用 Java 原生序列化

**最安全的方案**：用 JSON、Protobuf、Avro 等格式替代 Java 原生序列化。

```java
// SerializationSecurityDemo.java → 替代方案演示

// 方案1：Jackson JSON（最常用）
ObjectMapper mapper = new ObjectMapper();
// 序列化
String json = mapper.writeValueAsString(user);
// 反序列化（只能还原已知类型）
User user2 = mapper.readValue(json, User.class);

// 方案2：Gson（轻量）
Gson gson = new Gson();
String json = gson.toJson(user);
User user2 = gson.fromJson(json, User.class);

// 方案3：Protobuf（高性能，需要 .proto 文件）
// 二进制格式，比 JSON 更紧凑，性能更好
// 适合高性能 RPC（如 gRPC）

// 方案4：Kryo（高性能，保留类型信息）
// 比 Java 原生序列化快 10x+，字节流更小
// Netflix 大规模使用 Kryo 做缓存序列化
```

### 各方案对比

| 格式 | 可读性 | 性能 | 安全性 | 跨语言 |
|------|--------|------|--------|--------|
| Java 原生 | ❌ 二进制 | 中 | ❌ 有漏洞 | ❌ 仅 Java |
| JSON (Jackson) | ✅ 可读 | 中 | ✅ 安全 | ✅ |
| Protobuf | ❌ 二进制 | ✅ 最快 | ✅ 安全 | ✅ |
| Kryo | ❌ 二进制 | ✅ 很快 | ✅ 安全 | ❌ 仅 Java |
| Avro | ❌ 二进制 | ✅ 快 | ✅ 安全 | ✅ |

## 4.5 序列化数据的完整性保护

如果必须使用 Java 序列化，可以添加签名验证：

```java
// SerializationSecurityDemo.java → SignedObjectDemo
import java.security.*;

// 生成签名（序列化时）
KeyPairGenerator kpg = KeyPairGenerator.getInstance("DSA");
kpg.initialize(2048);
KeyPair kp = kpg.generateKeyPair();

// SignedObject 会序列化对象并用私钥签名
SignedObject signedUser = new SignedObject(user, kp.getPrivate(),
    Signature.getInstance("SHA256withDSA"));

// 验证签名（反序列化时）
boolean valid = signedUser.verify(kp.getPublic(),
    Signature.getInstance("SHA256withDSA"));
if (!valid) throw new SecurityException("数据被篡改！");

// 获取原始对象（只有签名验证通过才能获取）
User restoredUser = (User) signedUser.getObject();
```

## 4.6 本章总结

- **核心漏洞**：Java 反序列化可触发 Gadget Chain，实现 RCE（远程代码执行）
- **高风险场景**：接受网络/HTTP 输入并反序列化、Java RMI、某些缓存框架
- **JDK 9+ 防御**：`ObjectInputFilter` 白名单过滤，只允许反序列化可信类
- **根本方案**：用 JSON（Jackson/Gson）或 Protobuf 替代 Java 原生序列化
- **完整性保护**：`SignedObject` 对序列化数据签名，防篡改
