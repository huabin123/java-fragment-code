# 第三章：自定义序列化

## 3.1 为什么需要自定义序列化？

默认序列化把所有非 transient 字段都序列化，有时不满足需求：

1. **数据压缩**：字段冗余，需要更紧凑的格式
2. **安全加密**：某些字段需要加密后再序列化
3. **兼容旧格式**：历史遗留的序列化格式无法更改
4. **性能优化**：跳过不必要的字段，减少字节流大小
5. **对象状态恢复**：反序列化后需要重建某些运行时状态

---

## 3.2 writeObject / readObject：自定义序列化逻辑

通过在类中声明 `writeObject` 和 `readObject` 方法（必须是 `private`），可以完全控制序列化过程：

```java
// CustomSerializationDemo.java → SecureUser
public class SecureUser implements Serializable {
    private static final long serialVersionUID = 1L;

    private String username;
    private transient String password;  // 明文密码不序列化

    // 自定义序列化：密码加密后写入
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();  // 先写默认字段（username）
        // 加密密码后序列化
        String encrypted = encrypt(password);
        oos.writeObject(encrypted);
    }

    // 自定义反序列化：读出后解密
    private void readObject(ObjectInputStream ois)
            throws IOException, ClassNotFoundException {
        ois.defaultReadObject();  // 先读默认字段（username）
        // 读取并解密密码
        String encrypted = (String) ois.readObject();
        this.password = decrypt(encrypted);
    }

    private String encrypt(String s) { return "ENC:" + s; }  // 示例：实际用 AES
    private String decrypt(String s) { return s.substring(4); }
}
```

**关键点**：
- 方法必须是 `private`（JVM 通过反射调用，不需要 public）
- 必须先调用 `defaultWriteObject()`/`defaultReadObject()` 处理默认字段
- `writeObject` 写的顺序必须与 `readObject` 读的顺序完全一致

## 3.3 writeReplace / readResolve：控制序列化的对象替换

```java
// CustomSerializationDemo.java → Singleton 序列化安全

// 问题：单例类实现 Serializable 后，反序列化会创建新实例，破坏单例
public class MySingleton implements Serializable {
    private static final MySingleton INSTANCE = new MySingleton();
    private MySingleton() {}

    // ✅ readResolve：反序列化时替换为真正的单例对象
    private Object readResolve() {
        return INSTANCE;  // 反序列化返回的对象会被替换为 INSTANCE
    }
}

// 验证：
MySingleton s1 = MySingleton.INSTANCE;
// 序列化 s1
// 反序列化得到 s2
// s1 == s2 → true（readResolve 保证了单例）
```

```java
// writeReplace：序列化时用代理对象替换（序列化 "代理" 而非原对象）
public class ComplexObject implements Serializable {

    // 序列化时，用轻量的 SerializationProxy 替换复杂对象
    private Object writeReplace() {
        return new SerializationProxy(this);  // 序列化代理
    }

    // 静态内部类：序列化代理
    private static class SerializationProxy implements Serializable {
        private final String data;  // 只保存必要数据

        SerializationProxy(ComplexObject obj) {
            this.data = obj.toCompactString();
        }

        // 反序列化代理后，重建原始对象
        private Object readResolve() {
            return ComplexObject.fromCompactString(data);
        }
    }
}
```

**序列化代理模式**的价值：把序列化逻辑集中在代理类中，原始类不需要暴露内部构造细节，是 Effective Java 中推荐的序列化最佳实践。

## 3.4 Externalizable：完全自定义序列化

`Externalizable` 比 `Serializable` 更底层：不会自动序列化任何字段，必须完全手动实现：

```java
// CustomSerializationDemo.java → OptimizedPoint
public class OptimizedPoint implements Externalizable {
    private int x;
    private int y;
    private String label;

    // 必须有无参构造函数（Externalizable 要求）
    public OptimizedPoint() {}

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeInt(x);         // 只序列化必要的字段
        out.writeInt(y);
        out.writeUTF(label);    // 比 writeObject(String) 更紧凑
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException {
        this.x = in.readInt();
        this.y = in.readInt();
        this.label = in.readUTF();
    }
}
```

**Externalizable vs Serializable**：

| 维度 | Serializable | Externalizable |
|------|-------------|----------------|
| 默认字段处理 | 自动 | 无，完全手动 |
| 无参构造函数 | 不需要（反序列化不调用）| **必须** |
| 控制粒度 | 通过 transient 排除 | 完全自定义 |
| 性能 | 默认，有开销 | 可以更优化 |
| 实现复杂度 | 低 | 高 |

---

## 3.5 本章总结

- **writeObject/readObject**：`private` 方法，先调 `defaultWriteObject/ReadObject` 处理默认字段，再自定义额外操作
- **readResolve**：反序列化后替换对象，用于保护单例（枚举天然安全）
- **writeReplace**：序列化前替换对象，序列化代理模式的基础
- **Externalizable**：完全手动控制，必须有无参构造，性能可优化，代价是实现复杂
