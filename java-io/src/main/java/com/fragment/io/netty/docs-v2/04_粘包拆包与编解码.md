# 第四章：粘包拆包与编解码

## 4.1 粘包拆包的根本原因

TCP 是**流式协议**，没有消息边界。发送端写入两条消息，接收端可能读到任意切割：

```
发送端写入：[Hello][World][Netty]

接收端可能读到：
  正常：  [Hello] | [World] | [Netty]
  粘包：  [HelloWorld] | [Netty]
  粘包：  [HelloWorldNetty]
  拆包：  [Hel] | [loWor] | [ldNetty]
  混合：  [HelloWor] | [ldNet] | [ty]
```

**UDP 没有这个问题**（面向消息，有边界），但 TCP 的可靠传输是代价。

## 4.2 四种解决方案

### 方案1：固定长度（FixedLengthFrameDecoder）

```java
// 每条消息固定 10 字节，不足补空格
pipeline.addLast(new FixedLengthFrameDecoder(10));

// 适用：消息长度固定的协议（如某些金融报文）
// 缺点：消息长度不固定时浪费带宽
```

### 方案2：分隔符（LineBasedFrameDecoder / DelimiterBasedFrameDecoder）

```java
// 以换行符 \n 作为消息结束符（HTTP 报头、日志协议等）
pipeline.addLast(new LineBasedFrameDecoder(1024));  // 最大帧长 1024 字节

// 自定义分隔符（如 | 竖线）
ByteBuf delimiter = Unpooled.copiedBuffer("|", CharsetUtil.UTF_8);
pipeline.addLast(new DelimiterBasedFrameDecoder(1024, delimiter));

// 适用：文本协议（HTTP、SMTP、Redis RESP）
// 缺点：消息内容中不能包含分隔符（或需要转义）
```

### 方案3：长度字段（LengthFieldBasedFrameDecoder）—— 最常用

```java
// 协议格式：[长度（4字节）][消息体（N字节）]
// CodecDemo.java → demonstrateLengthFieldCodec()

pipeline.addLast(new LengthFieldBasedFrameDecoder(
    65535,    // maxFrameLength：最大帧长度
    0,        // lengthFieldOffset：长度字段偏移量（从帧头开始算）
    4,        // lengthFieldLength：长度字段本身占几个字节
    0,        // lengthAdjustment：长度值的修正量（如长度字段包含了自身长度时为 -4）
    4         // initialBytesToStrip：解码后跳过开头几个字节（跳过长度字段本身）
));
pipeline.addLast(new LengthFieldPrepender(4));  // 编码时自动加 4 字节长度头

// 适用：几乎所有自定义二进制协议（RPC、IM、游戏服务器）
// 优点：高效，支持任意长度消息；适合与 Protobuf 等搭配
```

### 方案4：自定义协议帧解码器

```java
// CodecDemo.java → CustomProtocolDecoder
// 协议：[魔数2字节][版本1字节][消息类型1字节][序列号8字节][长度4字节][数据N字节]

public class CustomProtocolDecoder extends ByteToMessageDecoder {
    private static final int HEADER_LENGTH = 16;  // 固定头部大小

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        // 可读字节不足头部长度，等待更多数据
        if (in.readableBytes() < HEADER_LENGTH) return;

        // 标记读指针（读失败可以回退）
        in.markReaderIndex();

        // 读取魔数校验
        short magic = in.readShort();
        if (magic != 0xCAFE) {
            ctx.close();   // 非法协议，断开连接
            return;
        }

        byte version   = in.readByte();
        byte msgType   = in.readByte();
        long requestId = in.readLong();
        int  bodyLen   = in.readInt();

        // 消息体还没到齐，回退等待
        if (in.readableBytes() < bodyLen) {
            in.resetReaderIndex();
            return;
        }

        byte[] body = new byte[bodyLen];
        in.readBytes(body);

        out.add(new RpcMessage(version, msgType, requestId, body));
    }
}
```

## 4.3 Netty 内置编解码器速查

| 编解码器 | 说明 | 典型用途 |
|---------|------|---------|
| `StringDecoder` / `StringEncoder` | 字符串与 ByteBuf 互转 | 文本协议、调试 |
| `LineBasedFrameDecoder` | 按换行符切帧 | HTTP 报头、日志 |
| `LengthFieldBasedFrameDecoder` | 按长度字段切帧 | 自定义二进制协议 |
| `ProtobufDecoder` / `ProtobufEncoder` | Protobuf 序列化 | 高性能 RPC |
| `HttpServerCodec` | HTTP 请求/响应编解码 | HTTP 服务 |
| `WebSocketServerProtocolHandler` | WebSocket 握手+帧处理 | 实时通信 |
| `SslHandler` | TLS 加解密 | 安全传输 |
| `IdleStateHandler` | 空闲检测（心跳基础）| 心跳/超时断连 |

## 4.4 MessageToByteEncoder / ByteToMessageDecoder 自定义

```java
// 自定义编码器（出站）：将对象转为字节
public class OrderEncoder extends MessageToByteEncoder<Order> {
    @Override
    protected void encode(ChannelHandlerContext ctx, Order order, ByteBuf out) {
        out.writeInt(order.getOrderId());
        byte[] nameBytes = order.getName().getBytes(StandardCharsets.UTF_8);
        out.writeInt(nameBytes.length);
        out.writeBytes(nameBytes);
        out.writeDouble(order.getPrice());
    }
}

// 自定义解码器（入站）：将字节转为对象
public class OrderDecoder extends ByteToMessageDecoder {
    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < 12) return;  // 至少 int + int + double 的固定部分
        in.markReaderIndex();

        int orderId    = in.readInt();
        int nameLength = in.readInt();
        if (in.readableBytes() < nameLength + 8) {
            in.resetReaderIndex();
            return;
        }
        byte[] nameBytes = new byte[nameLength];
        in.readBytes(nameBytes);
        double price = in.readDouble();

        out.add(new Order(orderId, new String(nameBytes, StandardCharsets.UTF_8), price));
    }
}
```

## 4.5 本章总结

- **粘包拆包的本质**：TCP 流式协议无消息边界，接收端需要自行还原消息
- **四种解决方案**：固定长度（简单）、分隔符（文本）、**长度字段（最推荐）**、自定义帧解码
- **`LengthFieldBasedFrameDecoder`**：生产中最常用，配合 `LengthFieldPrepender` 自动加长度头
- **自定义协议**：继承 `ByteToMessageDecoder`，用 `markReaderIndex/resetReaderIndex` 处理数据不足的情况
- **编码器**：继承 `MessageToByteEncoder<T>`，Netty 自动处理 ByteBuf 的分配和释放
