# 第五章：协议最佳实践

## 5.1 协议设计的七个原则

```
① 向前兼容：版本号字段必须有，新版本服务端能处理旧客户端的消息
② 幂等性：网络重传时，重复收到同一请求不会造成副作用（用序列号去重）
③ 快速失败：非法魔数立即断连，不要继续解析（安全 + 性能）
④ 限制消息大小：设置最大帧长度，防止内存溢出攻击
⑤ 心跳检测：内置 PING/PONG 消息类型，感知死连接
⑥ 错误消息类型：内置 ERROR 类型，区分业务错误和协议错误
⑦ 扩展预留：在协议头中预留 flags 字段，供未来扩展（如压缩、加密标志）
```

## 5.2 安全防护

```java
// CustomProtocolDemo.java → demonstrateSecurity()

// 防护1：魔数校验（立即拒绝非法连接）
short magic = in.readShort();
if (magic != MAGIC) {
    ctx.close();   // 立即断连，不读后续内容
    return;
}

// 防护2：消息大小限制（防止内存耗尽攻击）
int bodyLen = in.readInt();
if (bodyLen < 0 || bodyLen > MAX_FRAME_SIZE) {  // MAX_FRAME_SIZE = 10MB
    ctx.close();
    return;
}

// 在 Netty 中，用 LengthFieldBasedFrameDecoder 的 maxFrameLength 参数
pipeline.addLast(new LengthFieldBasedFrameDecoder(
    10 * 1024 * 1024,  // maxFrameLength = 10MB，超过则关闭连接
    0, 4, 0, 4
));

// 防护3：版本兼容
byte version = in.readByte();
if (version > CURRENT_VERSION) {
    // 收到更高版本的消息，回复 VERSION_NOT_SUPPORTED 错误
    sendError(ctx, requestId, "不支持的协议版本: " + version);
    return;
}
if (version < MIN_SUPPORTED_VERSION) {
    // 客户端版本太旧，拒绝服务
    ctx.close();
    return;
}
```

## 5.3 版本兼容策略

```
兼容性原则（扩展时）：
  ✅ 可以添加新字段（新版本服务端发，旧版本客户端忽略）
  ✅ 可以添加新消息类型（旧版本收到不认识的类型，返回 UNKNOWN_TYPE 错误）
  ❌ 不能修改已有字段的语义（破坏性变更）
  ❌ 不能修改固定头的字段顺序或长度（会破坏所有客户端）

Protobuf 的兼容性保证（推荐使用）：
  - 添加新字段（新 field number）：旧客户端忽略，向前兼容
  - 删除字段：保留 field number（不复用），旧数据读取返回默认值
  - 修改字段类型：不允许（除了 wire type 兼容的情况）

// 协议头中预留扩展位
┌──────┬──────┬──────┬──────────┬────────┬────────┬──────┐
│ 魔数 │ 版本 │ 类型 │  序列号  │ 标志位 │  长度  │ 数据 │
│ 2字节│ 1字节│ 1字节│  8字节  │  2字节 │  4字节 │ N字节│
└──────┴──────┴──────┴──────────┴────────┴────────┴──────┘
  标志位（2字节 = 16个bit）：
    bit0: 是否压缩（0=无压缩，1=gzip）
    bit1: 是否加密（0=明文，1=AES）
    bit2-15: 预留，全部置 0
```

## 5.4 常见问题排查

### 问题1：协议解析出错，消息内容混乱

```
可能原因：
  1. 粘包/拆包未正确处理（最常见）
     → 检查 LengthFieldBasedFrameDecoder 配置参数是否正确
  2. 编码器和解码器的字段顺序不一致
     → 仔细对比 encode() 和 decode() 的字段写入/读取顺序
  3. 字节序不一致（大端/小端）
     → Java 默认大端（Big-Endian），确保客户端一致
  4. 长度字段包含/不包含自身
     → lengthAdjustment 参数是否正确
```

### 问题2：高并发下序列号重复

```java
// ❌ 错误：int 溢出后变为负数，可能与其他请求的序列号冲突
private int requestId = 0;
public synchronized int nextId() { return requestId++; }

// ✅ 正确：用 AtomicLong（8字节，用尽概率接近 0）
private final AtomicLong requestId = new AtomicLong(0);
public long nextId() { return requestId.incrementAndGet(); }
// 即使每秒 100 万请求，2^63 / 1000000 ≈ 2900 万年才会溢出
```

### 问题3：pending 请求 Map 内存泄漏

```java
// 场景：RPC 请求发出，但服务端宕机，响应永远不来
// pending Map 中的 Future 永远不会被移除 → 内存泄漏

// ✅ 必须设置请求超时
channel.eventLoop().schedule(() -> {
    CompletableFuture<RpcMessage> f = pending.remove(requestId);
    if (f != null) {
        f.completeExceptionally(new TimeoutException("RPC 超时: " + requestId));
    }
}, 5, TimeUnit.SECONDS);

// ✅ 连接断开时清理所有 pending 请求
@Override
public void channelInactive(ChannelHandlerContext ctx) {
    pending.forEach((id, future) ->
        future.completeExceptionally(new IOException("连接已断开")));
    pending.clear();
}
```

## 5.5 协议 vs 框架选型

```
场景                        推荐方案
────────────────────────────────────────────────────────────────
微服务 RPC（Java 主导）     gRPC（Protobuf + HTTP/2，跨语言）
微服务 RPC（纯 Java）       Dubbo（成熟，治理能力强）
高性能内部 RPC              自定义协议 + Netty + Protobuf
实时消息推送                Netty + 自定义协议 or WebSocket
IoT 设备通信                MQTT over Netty（专为物联网设计）
API 网关                    HTTP/2（对外标准，内部转发）
```

## 5.6 本章总结

**协议设计七原则**：向前兼容、幂等、快速失败、限制消息大小、内置心跳、错误类型分离、预留扩展位

**三类安全防护**：魔数快速校验 + 最大帧长度限制（防 OOM 攻击）+ 版本范围校验

**三类常见 Bug**：
1. 粘包拆包 → 检查 `LengthFieldBasedFrameDecoder` 参数
2. 序列号溢出 → 用 `AtomicLong`
3. pending Map 泄漏 → 必须设超时 + `channelInactive` 时清理
