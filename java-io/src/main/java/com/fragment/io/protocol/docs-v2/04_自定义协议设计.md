# 第四章：自定义协议设计

## 4.1 为什么需要自定义协议？

```
HTTP 的局限性（某些场景）：
  - 报头开销大（HTTP/1.1 报头通常 200~800 字节，小消息占比高）
  - 请求-响应强制对应（不支持服务端主动推送，需 WebSocket）
  - 强制文本格式（HTTP/1.1），序列化效率低
  - 无法精细控制序列化格式

自定义协议的优势：
  - 极致压缩报头（如 16 字节固定头）
  - 支持任意消息模式（请求-响应、单向推送、双向流）
  - 自由选择序列化格式（Protobuf、Thrift、自定义二进制）
  - 适合：金融交易系统、游戏服务器、物联网设备通信、高性能 RPC
```

---

## 4.2 协议设计的五个要素

```
一个完整的自定义协议需要解决：

① 消息边界（防止粘包拆包）
  方案：固定长度头 + 长度字段（最常用）
  
② 协议标识（魔数）
  作用：快速识别协议合法性，拒绝非法连接
  通常 2~4 字节（如 0xCAFE、0x12345678）

③ 版本号
  作用：支持协议演进，新旧版本兼容

④ 消息类型
  区分请求/响应/心跳/错误等不同消息

⑤ 序列号（Request ID）
  异步场景：对应请求与响应（同一连接可能有多个未完成请求）
```

## 4.3 RPC 协议设计示例

```java
// CustomProtocolDemo.java → RpcProtocol

/**
 * 自定义 RPC 协议格式：
 * ┌──────┬──────┬──────┬───────────┬────────┬──────────┐
 * │ 魔数 │ 版本 │ 类型 │  序列号   │  长度  │   数据   │
 * │ 2字节│ 1字节│ 1字节│  8字节   │  4字节 │  N字节   │
 * └──────┴──────┴──────┴───────────┴────────┴──────────┘
 * 总固定头：16 字节
 *
 * 魔数：0xCAFE（标识 RPC 协议，非此魔数直接断连）
 * 版本：0x01（当前版本）
 * 类型：0x01=请求, 0x02=响应, 0x03=心跳PING, 0x04=心跳PONG, 0x05=错误
 * 序列号：唯一标识一次请求-响应对（异步匹配用）
 * 长度：数据部分的字节数
 */
public class RpcProtocol {
    public static final short MAGIC  = (short) 0xCAFE;
    public static final byte  V1     = 0x01;
    public static final int   HEADER = 16;

    // 消息类型
    public static final byte REQUEST   = 0x01;
    public static final byte RESPONSE  = 0x02;
    public static final byte HEARTBEAT = 0x03;
}

// 编码器
public class RpcEncoder extends MessageToByteEncoder<RpcMessage> {
    @Override
    protected void encode(ChannelHandlerContext ctx, RpcMessage msg, ByteBuf out) {
        out.writeShort(RpcProtocol.MAGIC);   // 魔数 2字节
        out.writeByte(RpcProtocol.V1);       // 版本 1字节
        out.writeByte(msg.getType());         // 类型 1字节
        out.writeLong(msg.getRequestId());   // 序列号 8字节
        byte[] body = msg.getBody();
        out.writeInt(body == null ? 0 : body.length);  // 长度 4字节
        if (body != null && body.length > 0) {
            out.writeBytes(body);             // 数据
        }
    }
}

// 解码器
public class RpcDecoder extends ByteToMessageDecoder {
    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < RpcProtocol.HEADER) return;  // 头部未到齐
        in.markReaderIndex();

        // 校验魔数
        short magic = in.readShort();
        if (magic != RpcProtocol.MAGIC) {
            ctx.close();  // 非法连接，断开
            return;
        }

        byte version   = in.readByte();
        byte type      = in.readByte();
        long requestId = in.readLong();
        int  bodyLen   = in.readInt();

        if (bodyLen > 0 && in.readableBytes() < bodyLen) {
            in.resetReaderIndex();  // 数据未到齐，等待
            return;
        }

        byte[] body = null;
        if (bodyLen > 0) {
            body = new byte[bodyLen];
            in.readBytes(body);
        }

        out.add(new RpcMessage(version, type, requestId, body));
    }
}
```

## 4.4 异步请求-响应匹配

```java
// CustomProtocolDemo.java → AsyncRequestMatcher

// 同一个连接可以并发发送多个请求，用序列号匹配响应
public class RpcClient {
    private final AtomicLong reqId = new AtomicLong(0);
    // 等待中的请求：序列号 → Promise
    private final ConcurrentHashMap<Long, CompletableFuture<RpcMessage>> pending =
        new ConcurrentHashMap<>();

    public CompletableFuture<RpcMessage> sendRequest(RpcMessage request) {
        long id = reqId.incrementAndGet();
        request.setRequestId(id);

        CompletableFuture<RpcMessage> future = new CompletableFuture<>();
        pending.put(id, future);

        channel.writeAndFlush(request).addListener(f -> {
            if (!f.isSuccess()) {
                pending.remove(id);
                future.completeExceptionally(f.cause());
            }
        });

        // 超时处理：5秒内无响应则取消
        channel.eventLoop().schedule(() -> {
            CompletableFuture<RpcMessage> f = pending.remove(id);
            if (f != null) f.completeExceptionally(new TimeoutException("RPC 超时"));
        }, 5, TimeUnit.SECONDS);

        return future;
    }

    // 响应 Handler：收到响应时触发
    public void onResponse(RpcMessage response) {
        CompletableFuture<RpcMessage> future = pending.remove(response.getRequestId());
        if (future != null) {
            future.complete(response);
        }
    }
}
```

## 4.5 序列化格式选型

| 格式 | 序列化速度 | 数据大小 | 可读性 | 跨语言 | 推荐场景 |
|------|---------|---------|-------|-------|---------|
| JSON | 中等 | 大 | ✅ 好 | ✅ | 对外 API、调试 |
| Protobuf | 快 | 小（3~10x 于 JSON）| ❌ 二进制 | ✅ | 高性能 RPC |
| Thrift | 快 | 小 | ❌ | ✅ | 多语言 RPC |
| Java 序列化 | 慢 | 大 | ❌ | ❌ | 不推荐（有安全漏洞）|
| Kryo | 最快 | 最小 | ❌ | ❌（仅 JVM）| JVM 内部高性能 |
| MessagePack | 快 | 小 | ❌ | ✅ | JSON 的二进制版 |

## 4.6 本章总结

- **协议五要素**：消息边界（长度字段）+ 魔数（合法性校验）+ 版本号 + 消息类型 + 序列号
- **编解码器**：继承 `MessageToByteEncoder` 编码，继承 `ByteToMessageDecoder` 解码；解码时用 `markReaderIndex/resetReaderIndex` 处理数据不足
- **异步匹配**：用 `ConcurrentHashMap<Long, CompletableFuture>` 存储 pending 请求，序列号匹配响应；务必设超时防止内存泄漏
- **序列化选型**：对外 API 用 JSON；高性能 RPC 用 Protobuf；纯 JVM 内部可用 Kryo
