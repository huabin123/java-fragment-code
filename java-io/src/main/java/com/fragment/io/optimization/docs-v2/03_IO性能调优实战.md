# 第三章：I/O 性能调优实战

## 3.1 性能问题的定位方法

```bash
# 第一步：确认瓶颈在 I/O 而非 CPU 或内存

# 查看 CPU 使用率和 I/O 等待
top
# %wa（iowait）高 → 磁盘 I/O 瓶颈
# %sy（system）高 → 大量系统调用（频繁 read/write）

# 查看磁盘 I/O 详情
iostat -x 1 5
# %util 接近 100% → 磁盘饱和
# await 高（> 10ms）→ I/O 延迟大
# r/s、w/s → 读写 IOPS

# 查看网络 I/O
sar -n DEV 1 5
# rxkB/s、txkB/s → 网络吞吐量

# 查看系统调用频率（是否有大量小 read/write）
strace -c -p <PID>          # 统计系统调用次数和时间
perf top -p <PID>           # 热点函数分析
```

## 3.2 减少系统调用：缓冲区的正确设置

```java
// PerformanceTuningDemo.java → demonstrateBufferSizing()

// ❌ 问题：缓冲区太小，大量系统调用
byte[] buf = new byte[64];  // 64字节的缓冲区，读 1MB 需要 16384 次 read()

// ✅ 优化：对齐操作系统 I/O 单元
// - 网络 Socket：通常 8KB~64KB（对齐 TCP MSS 和内核 Socket 缓冲区）
// - 文件 I/O：4KB 的整数倍（对齐操作系统 Page 大小）
byte[] buf = new byte[8192];  // 8KB，一次系统调用读 8KB

// 实测（PerformanceTuningDemo.java → benchmarkBufferSizes）：
// 缓冲区    读1GB文件耗时
// 64B      ~45000ms   （太多系统调用）
// 4KB      ~1200ms
// 8KB      ~800ms     （推荐）
// 64KB     ~750ms     （边际收益递减）
// 1MB      ~740ms     （没有明显提升）

// BIO 使用 BufferedInputStream 时指定合适缓冲区大小
new BufferedInputStream(inputStream, 8192);  // 默认 8192，通常够用

// NIO 直接分配足够大的 ByteBuffer
ByteBuffer buf = ByteBuffer.allocateDirect(64 * 1024);  // 64KB
```

## 3.3 Socket 调优参数

```java
// PerformanceTuningDemo.java → configureSocketForPerformance()

// 服务端 ServerSocket
serverSocket.setReceiveBufferSize(128 * 1024);  // 接收缓冲区 128KB
serverSocket.setReuseAddress(true);              // 快速重启

// 客户端 Socket / SocketChannel
socket.setSendBufferSize(128 * 1024);    // 发送缓冲区 128KB
socket.setReceiveBufferSize(128 * 1024); // 接收缓冲区 128KB
socket.setTcpNoDelay(true);              // 禁用 Nagle（降低延迟）
socket.setKeepAlive(true);               // TCP 保活

// SO_SNDBUF / SO_RCVBUF 的含义：
// 这是 Java 建议内核的值，内核可能不完全遵守
// 实际缓冲区大小通常是设定值的 2 倍（Linux 内部机制）
// 高吞吐场景（大文件传输）：建议 128KB~4MB
// 低延迟场景（实时通信）：建议 8KB~32KB，不要太大（防止缓冲膨胀/Bufferbloat）

// Linux 系统级调优（运维配置，非 Java 代码）：
// net.core.rmem_max = 134217728    # 最大接收缓冲区 128MB
// net.core.wmem_max = 134217728    # 最大发送缓冲区 128MB
// net.ipv4.tcp_rmem = 4096 87380 134217728
// net.ipv4.tcp_wmem = 4096 65536 134217728
// net.ipv4.tcp_window_scaling = 1  # 启用窗口缩放
```

## 3.4 批量写：write coalescing

```java
// PerformanceTuningDemo.java → demonstrateWriteCoalescing()

// ❌ 低效：每条消息单独 flush（每次都是一次系统调用）
for (Message msg : messages) {
    channel.write(encode(msg));  // 每次 write 可能触发系统调用
}

// ✅ 高效：批量写，减少系统调用
List<ByteBuffer> buffers = new ArrayList<>();
for (Message msg : messages) {
    buffers.add(encode(msg));
}
channel.write(buffers.toArray(new ByteBuffer[0]));  // Gather Write，一次系统调用

// 或者用 writev：先累积到大 ByteBuffer，再一次写出
ByteBuffer batch = ByteBuffer.allocate(totalSize);
for (Message msg : messages) {
    batch.put(encode(msg));
}
batch.flip();
channel.write(batch);  // 一次系统调用

// 实测提升：100 条消息
// 逐条写：~2ms（100 次系统调用）
// 批量写：~0.1ms（1 次系统调用）
```

## 3.5 线程模型优化

```java
// PerformanceTuningDemo.java → demonstrateThreadModel()

// I/O 密集型服务的线程数公式：
// 最优线程数 = CPU 核心数 × (1 + 等待时间 / 计算时间)

// 例：4 核 CPU，每个请求 70% 时间在 I/O，30% 在计算
// 最优线程数 = 4 × (1 + 0.7/0.3) ≈ 13 个线程

// 实际建议：
// 纯 I/O 线程（Selector 线程）：= CPU 核心数（甚至更少，1个也够）
// 业务线程池：= CPU 核心数 × 2~5（根据 I/O 等待比例）

// 避免的反模式：
// ❌ 业务线程无限扩展（无界线程池）
Executors.newCachedThreadPool();  // 线程数无上限，OOM 风险

// ✅ 有界线程池 + 合理队列
new ThreadPoolExecutor(
    cpuCores * 2,     // coreSize
    cpuCores * 4,     // maxSize
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),  // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 满了由调用者执行（背压）
);
```

## 3.6 性能调优清单

| 优化项 | 低效写法 | 高效写法 | 预期提升 |
|-------|---------|---------|---------|
| 缓冲区大小 | 64B~256B | 8KB~64KB | 10~100x |
| Direct Buffer | `allocate()` | `allocateDirect()` | 10~30% |
| 批量写 | 逐条 write | Gather Write | 5~20x |
| 文件传输 | read+write | `transferTo()` | 3~5x |
| 线程模型 | 一连接一线程 | Reactor + 线程池 | 10x+ |
| Socket 缓冲区 | 默认（8KB）| 128KB | 20~50%（高吞吐）|
| TCP_NODELAY | 默认（启用 Nagle）| `true` | 延迟降低 50%+（低延迟）|

## 3.7 本章总结

- **定位 I/O 瓶颈**：`top`（iowait）+ `iostat`（磁盘饱和）+ `strace`（系统调用频率）
- **缓冲区大小**：8KB 通常是最优点，对齐 Page 大小（4KB 的整数倍）
- **批量写**：Gather Write（`channel.write(ByteBuffer[])`）将多条消息合并为一次系统调用
- **文件传输**：`FileChannel.transferTo()` 是最快选择，不要用 `read()+write()`
- **线程数公式**：`CPU核 × (1 + I/O等待/计算时间)`，I/O 密集型通常配 CPU 核的 2~5 倍
