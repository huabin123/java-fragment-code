# 第一章：连接池与对象池

## 1.1 为什么需要池化？

创建 TCP 连接的代价：
```
建立一个 TCP 连接的完整流程：
  1. 三次握手：客户端 SYN → 服务端 SYN-ACK → 客户端 ACK
     网络延迟：通常 1~5ms（同机房）、10~100ms（跨地域）
  2. TLS 握手（HTTPS）：额外 1~2 个 RTT
  3. 数据库认证（JDBC）：额外的握手和权限验证

代价汇总：
  单次建连：5~200ms
  每秒 1000 次请求：全部新建连接 → 5~200 秒的连接时间开销（远超业务处理时间）
  OS 资源：每个 TIME_WAIT 连接占用资源，持续 2MSL（60~240 秒）
```

**池化思路**：预先建立连接，请求来了直接取，用完归还（不销毁）。

## 1.2 连接池核心参数

```java
// ConnectionPoolDemo.java → SimpleConnectionPool

// 以 HikariCP（最快的 JDBC 连接池）参数为例
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");

// ① 核心参数
config.setMinimumIdle(5);          // 最小空闲连接数（预热）
config.setMaximumPoolSize(20);     // 最大连接数（并发上限）

// ② 超时参数
config.setConnectionTimeout(3000); // 获取连接超时（ms），超时抛异常
config.setIdleTimeout(600_000);    // 连接空闲超时（ms），超时归还并关闭
config.setMaxLifetime(1800_000);   // 连接最大存活时间（ms），防止数据库主动断连

// ③ 健康检查
config.setConnectionTestQuery("SELECT 1");  // 验证连接可用（旧驱动用）
config.setKeepaliveTime(60_000);            // 每60秒发一次保活查询

// 最大连接数估算：
// = 核心业务线程数 × (平均查询时间 / 平均请求间隔)
// 例：20 个业务线程，每次查询 10ms，每线程每 50ms 来一个请求
// = 20 × (10/50) = 4，最大连接数设 4~8 足够（别过度配置）
```

## 1.3 手写简单连接池（理解原理）

```java
// ConnectionPoolDemo.java → SimpleConnectionPool

public class SimpleConnectionPool {
    private final BlockingDeque<Connection> pool;
    private final int maxSize;
    private final AtomicInteger current = new AtomicInteger(0);

    public SimpleConnectionPool(int maxSize) {
        this.maxSize = maxSize;
        this.pool = new LinkedBlockingDeque<>(maxSize);
        // 预热：创建 min 个连接
        for (int i = 0; i < maxSize / 2; i++) {
            pool.offer(createConnection());
            current.incrementAndGet();
        }
    }

    public Connection acquire(long timeout, TimeUnit unit) throws Exception {
        // 先尝试从池中取
        Connection conn = pool.poll();
        if (conn != null) return validateAndReturn(conn);

        // 池空且未达上限，创建新连接
        if (current.incrementAndGet() <= maxSize) {
            return createConnection();
        }
        current.decrementAndGet();

        // 达到上限，阻塞等待
        conn = pool.poll(timeout, unit);
        if (conn == null) throw new TimeoutException("获取连接超时");
        return validateAndReturn(conn);
    }

    public void release(Connection conn) {
        if (conn == null) return;
        if (!pool.offer(conn)) {
            // 池已满（不该发生），直接关闭
            closeQuietly(conn);
            current.decrementAndGet();
        }
    }

    private Connection validateAndReturn(Connection conn) {
        try {
            if (!conn.isValid(1)) {  // 1秒内验证
                closeQuietly(conn);
                current.decrementAndGet();
                return createConnection();  // 创建新连接替换
            }
        } catch (SQLException e) { /* 忽略 */ }
        return conn;
    }
}
```

## 1.4 对象池：减少 GC 压力

```java
// ConnectionPoolDemo.java → ByteBufferPool

// 场景：网络服务器需要大量临时 ByteBuffer，频繁 allocateDirect 代价高
// 解决：ByteBuffer 对象池（类似 Netty 的 PooledByteBufAllocator）

public class ByteBufferPool {
    // 按 Buffer 大小分档，避免碎片
    private static final int[] SIZES = {256, 512, 1024, 4096, 8192, 65536};
    private final Map<Integer, ConcurrentLinkedDeque<ByteBuffer>> pools = new HashMap<>();

    public ByteBufferPool() {
        for (int size : SIZES) {
            pools.put(size, new ConcurrentLinkedDeque<>());
        }
    }

    public ByteBuffer acquire(int minCapacity) {
        int size = roundUpToPoolSize(minCapacity);
        ConcurrentLinkedDeque<ByteBuffer> pool = pools.get(size);
        ByteBuffer buf = pool != null ? pool.pollFirst() : null;
        if (buf == null) {
            buf = ByteBuffer.allocateDirect(size);  // 池中没有，新建
        }
        buf.clear();
        return buf;
    }

    public void release(ByteBuffer buf) {
        int size = roundDownToPoolSize(buf.capacity());
        ConcurrentLinkedDeque<ByteBuffer> pool = pools.get(size);
        if (pool != null && pool.size() < 64) {  // 防止池无限增长
            buf.clear();
            pool.offerFirst(buf);
        }
        // 否则丢弃（让 GC 回收堆外内存）
    }
}
```

## 1.5 本章总结

- **连接池的价值**：消除每次建连的网络握手开销（5~200ms），将连接复用率提升到接近 100%
- **核心参数**：`maxPoolSize`（上限）+ `connectionTimeout`（获取超时）+ `maxLifetime`（防数据库踢连接）
- **最大连接数估算**：`业务线程数 × (查询时间 / 请求间隔)`，通常不需要设很大
- **对象池**：减少 `ByteBuffer.allocateDirect()` 的系统调用，降低 GC 压力；按大小分档避免内存碎片
