# 第二章：零拷贝技术详解

## 2.1 传统 I/O 的四次拷贝

以"读文件内容发送到网络"为例，传统方式的数据流转路径：

```
传统方式（read() + write()）：

硬盘 ──DMA──→ 内核缓冲区（PageCache）
                    │
                    │ ① CPU 拷贝（read 系统调用）
                    ↓
              用户空间缓冲区（byte[]）
                    │
                    │ ② CPU 拷贝（write 系统调用）
                    ↓
              Socket 发送缓冲区
                    │
                    │ ③ DMA 拷贝
                    ↓
                  网卡

总计：2 次 CPU 拷贝 + 2 次 DMA 拷贝 + 4 次用户态/内核态切换
代价：CPU 拷贝消耗大量 CPU 时间；传输 1GB 文件约需占 CPU 数秒
```

## 2.2 零拷贝的三种实现

### mmap + write（减少一次 CPU 拷贝）

```
硬盘 ──DMA──→ 内核缓冲区（通过 mmap 映射到用户空间）
                    │
                    │ ① CPU 拷贝（write 系统调用）
                    ↓
              Socket 发送缓冲区
                    │
                    │ ② DMA 拷贝
                    ↓
                  网卡

总计：1 次 CPU 拷贝 + 2 次 DMA + 2 次用户态/内核态切换
```

```java
// ZeroCopyDemo.java → demonstrateMmapZeroCopy()

FileChannel fc = FileChannel.open(Paths.get("large.bin"), StandardOpenOption.READ);
MappedByteBuffer mmap = fc.map(FileChannel.MapMode.READ_ONLY, 0, fc.size());

// mmap 后，读内存就等于读文件（缺页中断时自动加载），省去 read() 系统调用
SocketChannel socket = (SocketChannel) key.channel();
socket.write(mmap);  // 还有一次 CPU 拷贝：内核缓冲区 → Socket 缓冲区
```

### sendfile（Linux 2.1，零 CPU 拷贝）

```
硬盘 ──DMA──→ 内核缓冲区
                    │
                    │ ① CPU 拷贝（内核内，将 fd 描述符传给 Socket）
                    ↓
              Socket 发送缓冲区
                    │
                    │ ② DMA 拷贝
                    ↓
                  网卡

Linux 2.4 + 支持 SG-DMA（scatter-gather DMA）：
硬盘 ──DMA──→ 内核缓冲区
  仅传 fd 描述符给网卡 DMA
                    │ 直接 DMA 到网卡（0 次 CPU 拷贝！）
                    ↓
                  网卡
```

```java
// ZeroCopyDemo.java → demonstrateSendfile()

// Java 的 FileChannel.transferTo() 底层调用 sendfile()
try (FileChannel src = FileChannel.open(filePath, StandardOpenOption.READ);
     SocketChannel dest = (SocketChannel) key.channel()) {

    long transferred = 0;
    long size = src.size();
    while (transferred < size) {
        // transferTo 底层 = sendfile 系统调用
        long n = src.transferTo(transferred, size - transferred, dest);
        if (n > 0) transferred += n;
    }
}
// 数据路径：磁盘 → 内核 PageCache → 网卡（完全绕过用户空间）
```

### Direct Buffer（减少堆→内核的拷贝）

```java
// ZeroCopyDemo.java → compareDirectVsHeap()

// Heap Buffer 的数据路径：
// 数据 → 内核缓冲区 → JVM 堆内存（GC 管理）→ 临时 Direct 内存 → Socket
//                                                ↑ 因为 GC 可能移动堆内存，NIO 需要先固定到 Direct

// Direct Buffer 的数据路径：
// 数据 → 内核缓冲区 → Direct 内存（固定，不被 GC 移动）→ Socket
//                                ↑ 省去了一次堆→Direct 的拷贝

ByteBuffer direct = ByteBuffer.allocateDirect(64 * 1024);
channel.read(direct);  // 直接读到 Direct 内存，无额外拷贝
```

## 2.3 各种零拷贝技术对比

| 技术 | CPU 拷贝 | DMA 拷贝 | 系统调用 | 适用场景 |
|------|---------|---------|---------|---------|
| 传统 read/write | 2 次 | 2 次 | 2 次 | - |
| mmap + write | 1 次 | 2 次 | 2 次 | 频繁随机读大文件 |
| sendfile | 1 次（或 0 次） | 2 次 | 1 次 | 文件→Socket 传输 |
| Direct Buffer | 减少 1 次 | 2 次 | 2 次 | 网络读写缓冲区 |
| Netty CompositeBuf | 0 次（逻辑合并）| - | - | 多段数据拼接 |

## 2.5 零拷贝的适用场景

```
✅ 必用 transferTo() 的场景：
  HTTP 静态文件服务（大量下载请求）
  FTP 服务器
  日志归档传输

✅ 必用 Direct Buffer 的场景：
  高并发网络服务（Netty 默认使用 Direct Buffer）
  频繁读写的网络 I/O 缓冲区

✅ 考虑 mmap 的场景：
  数据库文件的频繁随机读（如 RocksDB、LevelDB）
  日志文件的追加写（内存映射追加写 O(1)）

❌ 不适合零拷贝的场景：
  需要在用户空间对数据做处理（加密、压缩、解析）
  → 必须经过用户空间，无法完全零拷贝
```

---

## 2.6 本章总结

- **传统 I/O**：2 次 CPU 拷贝 + 2 次 DMA + 4 次内核态切换，文件传输代价大
- **sendfile（`transferTo`）**：最佳选择，0~1 次 CPU 拷贝，文件→网络传输首选
- **mmap**：适合频繁随机读大文件（数据库、索引文件），减少 read() 系统调用
- **Direct Buffer**：减少堆→内核的额外拷贝，网络 I/O 缓冲区首选
- **Netty**：综合运用了 Direct Buffer + CompositeByteBuf（逻辑合并）+ FileRegion（sendfile）
